# 按情况路由 —— 任务分析 × 用户安排

> 上级:[[index]] · 模型:[[pairing-model]] · 落地:[[pairing-with-this-hub]]

分派 **不是** 固定流水线。它是两个输入**合并**出来的决定 —— 这正是本 hub
`src/routing.ts` 里 `planRoute(goal, policy)` 做的事。

## 输入①:分析任务(这活是什么)

读目标,归到几个面(本 hub 的 `analyzeTask(goal)` 用关键词替身,真实里是
路由模型的判断):

- **只审查**(reviewOnly):「review / 审查 / 解释,别改」→ 不需要实现者。
- **琐碎**(trivial):「typo / 重命名 / 小修」→ 不需要先设计。
- **需先设计**(needsDesign):「重构 / 新特性 / 跨文件」→ 先有方案再动手。

## 输入②:用户的安排(你现在的约束)

声明式 `RoutingPolicy`,把「现实情况」喂进来:

- `profiles` —— 名册 + 各自强项(谁善 review/design,谁善 implement)。
- `unavailable` —— 谁现在**不在岗**(登出 / 限流 / 你不想用它)。**绝不派不在岗的。**
- `singleCoder` —— 预算只允许**一个** coder。
- `preferLead` —— 优先主理是谁。

## 合并:同一目标,安排变了就派得不同

先按任务算「需要哪些角色」,再从**在岗**名册按强项填;理想人选不在岗就**优雅
降级**给在岗的顶上,永不硬失败:

| 目标 | 安排 | 派谁 |
|---|---|---|
| 新特性(需设计) | 默认名册 | `deepseek-tui`(起草)→ `codex`(实现) |
| 新特性(需设计) | Codex 不在岗 | `deepseek-tui` 一人包办(设计+实现) |
| 新特性(需设计) | 预算限 1 个 | `deepseek-tui` 一人包办 |
| 琐碎小修 | 默认名册 | `codex` 直接做 |
| 琐碎小修 | Codex 不在岗 | `deepseek-tui` 顶上做小修 |
| 只审查不改 | 默认名册 | `deepseek-tui`(报结论,不实现) |

## 给 mentor 的判断口诀

> 先看清这活要哪些**角色**(审查?实现?要不要先设计?),再看名册上**谁在岗**,
> 然后把 [[pairing-model]] 的三种配法套上去。理想人选不在岗 → 在岗的顶;预算
> 限一个 → 主理一人包办。**永远不派你说不可用的 coder。**
