# pro-firm-hub — 专业服务公司 hub 开箱包

面向**精品咨询 / 设计 / 代理 / 服务类小公司**(几个人到十几个人)的一键装模板,
也是 [`solo-company-hub`](../solo-company-hub) 的**向上兼容**形态:一个「一人公司」
做大了要招人、要接企业级数据源,这个包证明同一个内核从「一个人顶一个团队」平滑长成
「一个小团队协作交付」——多个成员分角色、多个专业分身接力干活、客户库与交付知识库
接进来,对外正式方案由负责人签发。

配套叙事一页纸见 [`docs/zh/PRO-FIRM-HUB.md`](../../docs/zh/PRO-FIRM-HUB.md)
(多人分权 + 多 agent 接力 + 企业级数据全走 MCP + 负责人签发 + 开箱验收标准)。

## 装什么

| 件 | 说明 |
|---|---|
| 3 个专业分身 | 接洽分析师(查客户背景 / 团队周报)+ 知识库助手(检索交付先例 / 新事项研究)+ 方案起草师(拟草稿 / 出正式方案),都跑 DeepSeek |
| `client-proposal` 工作流 | **旗舰多角色协作流**:顾问发起 → 分析师查背景 → 知识库助手检索先例 → 起草师拟草稿 → **停在负责人的签发步**(跨角色:发起是顾问、签发是负责人)→ 签发才出正式方案,不签发就到此为止 |
| `weekly-portfolio-brief` 工作流 | 团队周报(在办项目 / 本周待跟进 / 风险与阻塞 / 负责人一问),每周一早 8 点定时建议 |
| `new-matter-workup` 工作流 | 来了新事项/新机会,知识库助手对着交付先例出一页研究 |
| 3 个只读连接器槽位 | 客户/项目库 `db`、交付知识库 `kb`、团队任务 `tasks`,全部可选——不挂进诚实模式 |
| 2 条 golden-run 验收 | 装完面板「跑验收」零 LLM 判卷(审批流刻意不进:它要验的就是「负责人真签发一次」) |

三个分身都跑 DeepSeek(`openai-compatible`,`deepseek-chat`)——一个小团队养几个分身,
调用量的账扛得住。

## 「多人」怎么落地(模板不带人)

gotong 模板**不声明成员账号**——它只声明 agents + workflows + 角色门 + 审批闸这套「结构」,
真实成员在 hub 运行时另建、分角色:

- **owner / admin = 负责人 / 合伙人**:签发对外方案、管理 hub、看得见全局与用量。
- **member = 顾问**:干客户的活、有自己的 `/me` 和收件箱、看不到金库里的凭证。

本模板的「多人」主要靠 `client-proposal` 的**跨角色签发**:发起的顾问在表单里填「签发
负责人 (userId)」,签发卡就落到那个 userId 的收件箱。这跟 solo-company-hub 的「自己签发」
不同:专业服务公司里,对外方案由负责人把关才有意义。

诚实边界(跟 [`cafe-ops`](../cafe-ops) 指派店长同理):**签发步是把卡路由到填进去的 userId,
不是一道角色门**——框架不校验那是不是负责人、是不是发起人自己。`human:` 步只认「一个
userId」;真要锁死「必须负责人签、不能自批」需要权威角色校验,纯模板做不到,靠发起人填对
+ 部署纪律。真正服务端每请求校验的**角色门**是管理面(`admin` 门后)和 `/me` 的
`allowed_roles`(决定谁能看到哪个工作流入口)。本模板三条流的 `/me` 都对 owner/admin/member
开——团队周报被定时自动跑,而定时 sweeper 以最低权限 member 判可运行性,所以它的
`allowed_roles` 必须含 member(定时流不能拿 `allowed_roles` 当角色门用)。

## 怎么装

- **面板**:admin →「工作流 / 模板画廊」→「专业服务公司 hub」→ 安装(导入时填一次 DeepSeek key)。
- **一条命令开荒**(含补人启用定时):

```bash
gotong provision examples/pro-firm-hub/template/pro-firm-hub.template.yaml \
  --url http://127.0.0.1:8745 --token <admin-token> --user <你的成员id>
```

装完在 hub 里加几个成员、分好 owner/admin/member 角色,就是一个小团队了。

## 装完点什么

1. 顾问(member)`/me` →「起草客户方案」→ 填客户与需求 + 填「签发负责人 (userId)」→ 提交。
2. 三个分身接力(查背景 → 检索先例 → 拟草稿),然后**填的那个 userId** 的 `/me` 收件箱出现
   「客户方案签发」卡,卡上**内联着草稿全文**(**卡先出现,正式方案还没出**)。
3. 负责人签发 → 工作流续跑出正式方案;换一次点拒绝 → `finalize` 步被跳过,正式方案为空 = 什么也没出。
4. 顾问 `/me` →「出团队周报」/「研究一个新事项」一键跑通,即 golden-run 同款路径(开箱
   不挂连接器也进诚实模式跑绿)。
5. 「定时」卡补人启用后,每周一早 8 点(默认 UTC+8)自动送团队周报(定时流对所有成员角色开,
   否则 sweeper 判不可运行)。

## 接企业级数据(全走 MCP,只读接入)

企业级数据库、知识库在 gotong 里永远是**外部**的——框架不存数据、不存知识(见
[`docs/zh/RAG-VIA-MCP.md`](../../docs/zh/RAG-VIA-MCP.md) / [`KB-CONNECTORS.md`](../../docs/zh/KB-CONNECTORS.md))。
本模板声明三个抽象槽位,由你用**只读**凭证把自己的源接进来,再在对应 agent 表单里勾上:

- `db` —— 客户/项目结构化数据库。接 Postgres 或结构化 DB 类 MCP server(pgvector 路线见 RAG-VIA-MCP)。
- `kb` —— 交付物/SOP/报价先例知识库。接 `chroma-rag`(rag 组)或企业级 `elasticsearch-kb`。
- `tasks` —— 团队任务/交付清单(如 `todoist-tasks`)。

不挂也能跑:分身进诚实模式,不编造客户名/项目/金额。

## 诚实边界

- `client-proposal` 签发后的产出是**一页正式方案文案 + 一条记录**,不是真的发给客户,也
  **不写回任何客户库**——模板不携带出站/写入凭证,槽位建议只读接入,写动作仍逐次过审批闸
  (**接入 ≠ 授权行动**)。
- 分身检索到的客户背景/交付先例只用于起草,**不做承接决定**;报价一律标「待确认」,不是
  市场实时行情。
- 没挂连接器时进诚实模式:没有的数据就不编,如实说明「未接数据源」。
- 生成所需上下文(客户名/需求/检索到的资料)会发送到所配置的模型(DeepSeek)与只读连接器后端。

## 再往上(小团队 → 更大 / 跨组织)

- 更重的数据源:`db` 换成企业级 Postgres 集群、`kb` 换成 elasticsearch-kb / 自建向量库,槽位 id 不变。
- 跨组织协作:多个 hub 之间走联邦,per-link 知识库白名单(`packages/host/src/peer-kb-gate.ts`)
  管住「哪条边能查哪个 kb」——见 [`docs/zh/FEDERATION-RUNBOOK.md`](../../docs/zh/FEDERATION-RUNBOOK.md)。
- 更多角色协作形态参考画廊里的 [`cafe-ops`](../cafe-ops)(门店运营多角色)。
