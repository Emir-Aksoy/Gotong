## 一、为什么 Gotong 适合做门店,又不该做成什么

Gotong 的定位是**店长大脑 + 流程编排 + 审批闸 + 金库**——它**不是** POS / ERP /
排班软件 / 算薪软件。想清楚这条,五件事就各归其位:

- **编排逻辑在 Gotong**:谁提报、谁审批、哪步该停下来等人、哪步的产出喂给哪步——
  这是声明式工作流 + `human:` 审批步,Gotong 的强项。
- **经营数据的「主副本」在你接的外部系统**:排班表、库存数、工时、工资记录都是
  **有状态业务数据**,它们的**权威账本**一律走 MCP 连接器接你自己的库(数据库 /
  表格 / POS / 考勤),框架不把它当主数据源来存。
- **但要诚实**:Gotong 走 file-first,**会把每次工作流运行的快照落到 `.gotong/`**
  ——触发时你在 `/me` 表单填的输入(姓名、工时、时薪、请假事由)、每步产出、审批
  记录、transcript 都写进磁盘的 RunState。所以「框架不存经营数据」指的是**主副本
  不在这里、精确计算不靠它**,不是「一个字节都不落盘」。搬走 `.gotong/` 带不走你
  外部库里的权威账本,但带得走这些运行快照——把它当**审计留痕**看,不是数据主权
  的免死金牌。

这跟 [`pro-firm-hub`](PRO-FIRM-HUB.md) 的 `db`/`kb` 只读槽位、`cafe-ops` 的
`store_ops_manual` 槽位是同一套设计,不是新发明。

## 二、五件事逐条怎么落

| 你要的 | 现有先例 | 缺口 | 怎么落(全走既有机制) |
|---|---|---|---|
| **每个岗位的知识** | ✅ `cafe-ops` 岗位培训师 + `store_ops_manual` KB 槽位 + 上手流 | 基本齐 | 每岗位 SOP 放你的知识库(只读 MCP 槽位),培训师 agent 按岗位调出带教;`surface.me` 成员自助无审批 |
| **排班** | ✅ `cafe-shift-availability`(提报→整理→店长确认) | 齐 | 成员提报可排时段 → 助手整理草案 → 店长 `human:` 确认 |
| **请假** | ⚠️ 无 | 补一条 | 与排班同构:成员提报请假 → 助手核对班表冲突 → 店长 `human:` 批;审批过了才算数 |
| **工资(含临时工)** | ⚠️ 加班费有(`cafe-overtime-claim`:建议金额+人定钱+倍率随日别) | **临时工工时结算无** | 见 §三红线——LLM 建议数字 + 人定钱 + record 步 when 门控;临时工按周期汇总工时结算 |
| **库存管理** | ✅ `tea-supply-link` 跨组织补货(独立样板,不在本模板内) | 店内**盘点 / 预警**无 | 本模板补 `cafe-inventory-count`:成员 `/me` 发起盘点 → 助手出清单、接了库存 MCP 就对低于安全线的项标红(无人值守)。补货申请是另一条路(`tea-supply-link`),本模板不含 |
| **合规要求** | ⚠️ `cafe-ops` SOP 里有「食品安全/现金/会员信息」红线提示 | **独立合规流无** | 本模板补 `cafe-compliance-check`:一份**通用合规自查清单**流(食品安全 / 保质期 / 现金 / 会员信息逐项过),`schedules` 每周一定时跑一遍。**注意**:它是通用清单提醒,不是「盯着某张证照的到期日报警」——真按证照到期日提醒需要你把证照台账接成数据源,本模板没有那个槽位 |

**关键**:请假、临时工薪、库存预警、盘点、合规检查、证照提醒——**没有一条需要新机制**。
它们分别是既有的 `surface.me`(自助发起)+ `human:`(审批停车)+ `requires.connectors`
(只读数据槽位)+ `schedules`(定时)+ `acceptance`(装完机器验收)的重新组合。这正是
「零内核改动」的底气。

## 三、一条必须钉死的红线:工资的钱不由 LLM 拍板

这是整个 track 最重要的设计点,也是「框架不跑 LLM 做**决策**」北极星在**钱**上的直接
体现。红线不是「LLM 不许碰工资数字」——而是**LLM 永远不拍板**:谁产生建议数字是一回事,
谁把它定成要发的钱是另一回事,后者永远是人。

**模板结构性强制的(不管谁算,这几条一定成立):**

- **④ 每一笔金额都门控在人审批后**:加班 / 临时工薪的金额都要店长 / 老板在 `human:` 步
  approve 才算数;临时工薪的 `record`(结算单)步 `when` 门控在审批上——**拒绝就不出
  结算单**(这是真的:拒批返回 `{approved:false}`,`when` 判假,record 步 skip)。
- **不自动发钱、不写回工资库、不做劳动法裁定**:模板不带任何支付 / 写工资库连接器,
  也不让 LLM 下「合法 / 违法」结论(看到异常只提示「需人工核对当地法规」)。
- **一律标「以你的正式工资系统为准」**:模板产出的永远是**建议 + 记录**,不是权威工资单。

**谁产生「建议数字」——诚实说清:**

```
 ① 工时/时薪数据  →  ② 算「建议金额」          →  ③ 整理/标异常      →  ④ 人审批(钱在此定)
   走 MCP 读或           模板默认:LLM 按你店政策算     列算式、标「工时偏高      店长/老板 human:
   表单手录              (demo 里换成确定性 stand-in) /时薪偏低→人工核对」     approve 才算数
```

- **② 谁算建议金额,取决于 compute 步派给谁**:
  - **默认可载入模板**里,加班 / 临时工薪的 compute 步派给 DeepSeek `ops-assistant`
    ——它按你店面政策算出的是 **LLM 建议数字**,不是可审计的确定性算术。
  - **想要流内确定性算薪**(时薪 × 工时纯算术、可审计),关键是让 compute 步派给一个
    **非 LLM 的参与者**——像 `cafe-ops/src/standins.ts` 里 demo 用的那个
    `assessOvertime = Math.round(h*baseHourly*multiplier*100)/100`,那是 **demo 里
    TypeScript 的确定性算法**,证明这条路能走通,但它**只在 demo 跑、不在可载入模板运行**。
  - **注意别踩坑**:给 `ops-assistant` 挂一个算薪 MCP **并不等于**确定性——compute 步
    仍是「派给 LLM,LLM 去调工具」,数字由 LLM 中转、原则上仍能被它改写。真确定性要
    换的是 **compute 步的参与者本身**(从 LLM agent 换成非 LLM 组件),不是给 LLM 加工具。
  - compute 步就是这个「换参与者」的接缝:LLM 建议 ↔ 确定性组件,换谁都不动别的。
- **③ 语言活**:把数字整理成结算单文案、**标出可能的异常**(工时偏高 / 时薪偏低),
  提示需人工核对——不下法律结论,也不擅自改数。

**临时工工时结算流**(本 track 补的主菜,与加班流同构但按周期汇总):

```
临时工/店长录本周期各班次工时 (或从考勤 MCP 读)
   → [compute] 助手按政策算建议金额,列明算式 (¥X/h × N 班次工时 + 周末/节假日倍率)
   → [approve]  老板 human: 审批 (钱的事人定)
   → [record]   when 门控:批准才建档结算单 (拒绝跳过;只记录,不发钱、不写回工资库)
```

**诚实边界**:默认模板里那个「建议金额」是 **LLM 按你填的政策算的建议**(接了算薪 MCP
则是工具算的),**最终以你的正式工资系统为准**;模板**不替你发钱、不写工资库、不做劳动法
裁决**。要真发要真入账,接你的工资系统且仍逐次过审批闸(接入 ≠ 授权)。

## 四、奶茶店 vs 酒吧:共享机制,差在角色 / 合规 / 临时工

五件事的**工作流机制两店完全一样**(同一套请假/工资/库存/合规流结构),差异集中在三处:

| 维度 | 奶茶店(扩 `cafe-ops`) | 酒吧(新 `bar-ops`) |
|---|---|---|
| **岗位角色** | 制饮(吧台)/ 收银 / 清洁支持 | 调酒师 / 侍酒·传菜 / 安保 / 收银 |
| **合规内容** | 食品安全、原料保质期、加盟标准化、现金/会员信息 | **酒牌、售酒年龄核查、营业时间限制、醉酒应对**、食品卫生 |
| **临时工** | 有(周末兼职),比例中 | **比例高**(兼职调酒 / 传菜 / 周末安保),深夜 / 周末排班 |

所以你特别强调的**「临时工工资」和「合规」在酒吧场景最实**——酒吧是这两个诉求的最佳
证明场;奶茶店则是改动成本最低的路(`cafe-ops` 底子在)。马来西亚语境下酒吧还涉执照 /
宗教敏感,合规维度更硬,这些进 KB 内容与合规清单流,不进框架代码。

**酒吧独有的招牌:合规决定也走同一道 `when` 门控。** 加班/临时工薪的红线是「钱不由 LLM
拍板」(§三);`bar-ops` 把**同一个结构**用在**合规**上——`bar-age-incident` 流:员工上报
拒售给疑似未成年的事件 → 助手整理成复核单 → 值班经理 `human:` 复核 → `record` 步 `when`
门控在复核上。**经理确认才写一条正式拒售条目;判定误报(拒绝)则跳过,不写正式条目、不
上报官方**。它记的是「员工按流程实施了拒售」这个事件,**不裁定顾客真实年龄、不执法**——
年龄核查本身是现场操作规范(入岗位知识),这条流只做「事件复核 + 建档」。这正是边界 §五.3
的落地:合规是提醒/记录,不是自动裁决。

## 四之二、开箱见产出(验收标准 / STORE-M3)

两个样板都是**画廊一键装 bundle + 可跑 demo**。装完你能立刻见到的产出,以及验收它「真跑起来了」的确切标准:

**奶茶店(`cafe-ops`)** —— 装完面板「跑验收」(golden-run)判卷:
- 盘点流 / 合规自查流两条无人值守流是验收项(填好 DeepSeek key 后,**不挂任何可选连接器**
  ——库存 / 考勤 MCP——也必须过;可选连接器才 optional,LLM key 是必填凭证)。
- 带 `human:` 的**四条流**(排班 / 加班 / 请假 / 临时工薪)刻意**不进 golden-run**——它们要验的
  就是「人真审批一次」,机器替不了(见 §三红线)。

**酒吧(`bar-ops`)** —— `pnpm demo:bar-ops`(确定性 stand-in,零网络零 key)exit 0,逐条自断言
全过(含 5 条从磁盘读回的 `RunStore` 落盘证明),四个可见时刻:
1. **岗位上手**:新人选「调酒师」→ 培训师调出该岗位 SOP + 规范,并标出**年龄核查红线**。
2. **年龄核查事件复核(HITL)**:上报事件 → 挂起在值班经理审批闸 → 经理确认 → 工作流恢复、
   写出一条正式拒售条目。
3. **拒绝也留痕**:第二次上报被经理判误报(拒绝)→ 工作流照样恢复(run 不中断),但 `record`
   步被 `when` 门控跳过 → **不写正式拒售条目**。demo 接真 `RunStore` 从磁盘读回**证死**:两次
   运行都落盘、都到终态 `done`,批准运行的 `record` 步是 `done`、拒绝运行的是 `skipped`——
   「每次运行都留运行记录,只是拒绝的运行不写正式条目」不是散文声称,是磁盘上的事实。
4. **深夜薪结合班次**:同样 4 小时,深夜 ×1.5 / 周末 ×2 / 节假日 ×3 → 建议 ¥150 / ¥200 / ¥300;
   数字是**建议**,老板审批才定钱。

专属防腐门 `bar-ops-template.test.ts`(画廊门把 workflow 当不透明 blob 抓不到这些):按 step
精确钉两条 `when` 门控 record 步、年龄事件的 `human:` desugar 形状、定时合规流对 `member` 开
(定时 sweeper 以最低权限 member 判可运行性,剔掉 member 会让每周自查永不触发)。

## 五、四条不可破边界

1. **工资的钱不由 LLM 拍板**(§三):金额一律人审批 + record 步 when 门控 + 不发钱不写工资库不做劳动法裁定;「谁产生建议数字」(默认 LLM,可换算薪 MCP)与「谁定钱」(永远是人)分开。
2. **有状态数据走 MCP**:排班表 / 库存数 / 工时 / 工资记录一律外部只读槽位,框架不存。
3. **合规是提醒不是裁决**:合规流出的是「对照清单的提示 + 到期提醒」,**不做法律裁定**;
   责任在人。售酒年龄这类是**现场操作规范**(入岗位知识),不是能自动执行的工作流。
4. **内核零改动、零新旋钮**:五件事全是模板层(工作流 YAML + agent config + MCP 槽位),
   走既有 `surface.me`/`human:`/`requires.connectors`/`schedules`/`acceptance`。旋钮冻结在 120。

## 七、非目标(明确不做)

- **不做算薪引擎 / POS / ERP / 排班优化器**——那些是外部系统,Gotong 接不造。
- **不替你发工资、不写回工资 / 库存 / 排班库**——模板只编排 + 建档,写动作过审批闸。
- **不做劳动法 / 酒牌法的自动裁决**——合规流是提醒与清单,责任在人。
- **不携带人员 / 不携带 key**——模板带结构(agents + workflows + 角色门 + 审批闸),
  真实成员在 hub 运行时另建;凭证进金库走 `${NAME}` 占位。
