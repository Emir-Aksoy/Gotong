# 专业服务公司 hub — 一个小团队协作交付

> **这是「企业级向上兼容」案例的首屏叙事页(ENT track)**:给精品咨询 / 设计 /
> 代理 / 服务类小公司(几个人到十几个人)的负责人读的一页说明 + 开箱验收标准。
> 它回答一个问题:[`solo-company-hub`](SOLO-COMPANY-HUB.md) 证明了「一个人顶一个
> 团队」,那**招了人、要接企业级数据源之后**,同一个内核撑不撑得住?撑得住——而且
> 不用换框架、不用重搭。深入设计与上线步骤见文末指针。
>
> 场景出处:这不是编的需求。一个自由职业者 / 一人工作室做顺了,自然会走到「接单
> 接不过来 → 招第一个顾问 → 再招 → 客户资料多到要进数据库 → 交付物多到要进知识库」。
> 这一步最怕的是「工具得推倒重来」:个人版的 AI 助手扩不成团队协作。这个案例证明
> Gotong 的同一个内核是**往上兼容**的:一人公司用的管家、审批闸、金库、工作流,加人、
> 加数据源之后原样还在,只是多了「分角色」和「跨角色签发」。
>
> Last updated: 2026-07-19（ENT-M1）

## 一、三十秒版本

同一台机器上,一个**你公司的 hub**:几个成员分角色(负责人 / 顾问)、三个专业分身
(接洽分析师、知识库助手、方案起草师)、你的客户库和交付知识库接进来。来了个客户,
顾问在 `/me` 发起一单 →**三个分身接力**:分析师查这个客户的背景与往来、知识库助手
翻出相关的交付先例和报价参考、起草师综合成一页方案草稿 →**停在负责人的签发步** →
负责人过一眼、签发,才整理成正式方案;不签发就到此为止。客户资料和交付知识库全走
**只读 MCP 连接器**接进来,凭证锁在**加密金库**里,对外正式方案**负责人签发**。

一句话:**分身接力起草、负责人签发;多人分角色、数据在你自己的库里;一人公司的那套
东西,加了人和数据源之后原样还在。**

## 二、用户点名的四件事,一条流里全发生

用户问这个案例要证明四件事:**多人、多 agent、MCP 企业级数据库、知识库**。旗舰工作流
`client-proposal` 把这四件事编织在**同一条流**里——这不是四个割裂的 demo,是一次真实的
客户接洽,四个维度同时在场:

| 维度 | 在 `client-proposal` 里怎么体现 |
|---|---|
| **多人** | 发起的是**顾问**(member),签发的是**负责人**(owner/admin)——跨角色审批,签发卡按表单填的 `approver_id` 路由到那位负责人收件箱(框架路由到填的 userId、不校验角色,见 3.1) |
| **多 agent** | `research`(接洽分析师)→ `reference`(知识库助手)→ `draft`(方案起草师)三个分身**接力**,一个的产出喂给下一个,不是一个 agent 硬扛全程 |
| **企业级数据库** | `db` 只读槽位:接洽分析师从你的客户/项目库里读这个客户的往来记录、在办项目 |
| **知识库** | `kb` 只读槽位:知识库助手从你的交付知识库里翻相关交付物、报价先例、SOP |

另外两条工作流覆盖团队日常:`weekly-portfolio-brief`(团队周报:在办项目 / 本周待跟进 /
风险与阻塞 / 负责人一问,每周一早自动跑)和 `new-matter-workup`(新事项研究:对着交付
知识库出一页背景 / 先例 / 可行性 / 风险)。

## 三、为什么同一个内核撑得住「从一个人到一个团队」

### 3.1 多人分权:角色是服务端的门,不是设置里的开关

同一个 hub 加成员,角色四级 `owner` / `admin` / `member` / `viewer`,管理面在 `admin`
门后——**服务端每请求校验**,不是界面上藏个按钮。负责人 / 合伙人拿 `owner` / `admin`
(签发、管理、看全局与用量);顾问拿 `member`(有自己的 `/me` 和收件箱、干客户的活、
**看不到金库里的凭证**)。招第一个顾问和招第十个顾问,是同一套机制。

**跨角色签发**是「多人」最实的体现:`client-proposal` 的签发步 `assignee` 走
`$trigger.payload.approver_id`——发起的顾问在表单里指定由哪位负责人签发,签发卡就落到
**那个 userId** 的收件箱。这跟一人公司的「自己签发」正好是一对照:公司里对外方案由别人
把关才有意义,而框架用的是同一套 `/me` 收件箱 + 审批闸,只是 `assignee` 从「自己」变成
表单里填的那个人。

诚实说清一条边界(跟 [`cafe-ops`](../../examples/cafe-ops) 指派店长同理):**签发步是把卡
路由到填进去的 userId,不是一道角色门**——框架不校验那个 userId 是不是负责人、是不是发起人
自己。`human:` 步只认「一个 userId」;要真正锁死「必须负责人签、不能自批」需要权威的角色
校验,纯模板做不到。填谁签、会不会自批,是发起人的责任 + 部署纪律。这条流负责把签发**路由**
到指定的人,不负责证明那个人有权签。

本案例里真正**服务端每请求校验的角色门**是管理面(`admin` 门后)和 `/me` 的
`allowed_roles`(谁能看到哪个工作流入口);跨角色签发是**路由**、不是角色门,两者别混。
(团队周报被定时自动跑,而定时 sweeper 以最低权限 `member` 判可运行性,所以它的
`allowed_roles` 必须含 `member`——定时流不能拿 `allowed_roles` 当角色门用。)

### 3.2 多 agent 接力:能力路由,不是把活堆给一个大模型

三个分身各有一个 capability(`firm.intake` / `firm.knowledge` / `firm.proposal`),工作流
的每一步 `dispatch` 到一个 capability,框架按能力路由到有该能力的 agent。这让「一条流里
多个专家接力」成为结构——分析师的客户背景喂给起草师、知识库助手的先例喂给起草师,每个
分身只干自己那段。要加第四个分身(比如合规复核),就是加一个 agent + 在流里加一步,不动
其他分身。三个分身都跑 **DeepSeek**,一个小团队养几个分身的调用量扛得住(框架不跑 LLM,
每次调用都在分身身上——省钱这条对小公司和一个人一样成立)。

### 3.3 企业级数据库 / 知识库:全走 MCP,框架不存你的数据

这是 Gotong 钉死的设计立场:**框架不存数据、不存知识**(见
[`RAG-VIA-MCP.md`](RAG-VIA-MCP.md) / [`KB-CONNECTORS.md`](KB-CONNECTORS.md))。客户库、
交付知识库永远是**外部**的,通过 MCP 连接器接进来——好处对企业尤其重要:

- **数据主权**:客户资料留在你自己的 Postgres / 向量库里,`.gotong/` 目录里没有你客户
  数据的副本,框架搬家不会把客户库一起搬走。
- **只读接入 + 接入≠授权**:模板给的槽位标的是**只读意图**,真正的只读来自你接入时用的
  **只读**凭证 / server——用了只读凭证,分身就只能读不能写;真要写回或对外发,仍逐次过审批闸。
- **换后端不换槽位**:`db` 从单机 Postgres 换成企业级集群、`kb` 从 `chroma-rag` 换成
  `elasticsearch-kb`,槽位 id(`db` / `kb`)不变,换的只是背后接哪个 server。

诚实说清:框架**不内置**关系型数据库连接器——这不是缺功能,是「不存数据」立场的延伸。
模板声明抽象槽位、指向 `RAG-VIA-MCP.md` 里已验证的 pgvector 路线,由你把自己的库接上。

### 3.4 凭证进金库、状态是磁盘目录 → 团队也完全掌握

所有 API key、连接器 token 进加密金库,界面和日志里只出现 `${名字}` 占位符,明文不落
配置文件(有防腐测试钉死)。整个 hub 的状态就是磁盘上一个 `.gotong/` 目录。团队版跟
一人版是同一套金库、同一套 file-first——不因为「人多了」就把钥匙交给别人的云。

## 四、「开箱见产出」验收标准

对照仓库的 FUN 承重门(`QUICKSTART.md` 五分钟漏斗 + `scripts/first-result-smoke.mjs`)
设计:**每个承诺是一条独立可见的时刻**,任何一条不亮就算不通过。

**前置**:一台能跑 Docker 的电脑(或 Node ≥ 20),一个 DeepSeek key,hub 里建好至少两个
成员(一个 member 当顾问、一个 owner/admin 当负责人)。**全程不写代码、不手改配置文件。**

| 时段 | 做什么 | 必须看见什么(可见时刻) |
|---|---|---|
| **0 → 5 分钟** | `docker compose up` → 向导建 owner + 粘 DeepSeek key → 再加一个 member 成员 | 面板首页亮起;两个成员各有自己的 `/me`(负责人和顾问看到的入口不一样) |
| **5 → 8 分钟** | **顾问(member)**`/me` →「出团队周报」一键跑 | **时刻 A(分身能干活)**:四小节周报当场生成、无连接器也进诚实模式不报错 |
| **8 → 12 分钟** | 顾问 `/me` →「研究一个新事项」贴需求 → 出一页研究 | **时刻 B(知识库助手成锚)**:背景 / 相关先例 / 可行性 / 风险四节齐全 |
| **12 → 18 分钟** | 顾问 `/me` →「起草客户方案」填客户 + 填负责人 userId → **那个 userId 的**收件箱弹签发卡(内联草稿全文)→ 负责人签发 | **时刻 C(多人 + 跨角色签发)**:三个分身接力后卡落到**填的那个人**手里、签发才出正式方案;换一次点拒绝,`finalize` 步被跳过、确实没出方案 |

时刻 A / B(周报 / 新事项研究)就是模板 `acceptance` 里的两条 golden-run——装完面板点
「跑验收」零 LLM 机器判卷,开箱不挂任何连接器也必须绿。时刻 C 刻意不进机器验收:带
`human:` 步的流无人值守必然停在签发步,「负责人真签发一次」恰恰是它要人来验的——而且
这一次是**跨角色**的(顾问发起、负责人签发),这正是「多人」的现场证明。(诚实说清:框架把卡
路由到表单里填的 userId,不校验那个人是不是负责人——见 3.1。)

## 五、诚实盘点

| 维度 | 真实状态 |
|---|---|
| 功能成熟度 | ✅ 骨架承重墙齐备(多成员 / 角色门 / 跨角色审批 / 金库 / 定时 / MCP 连接器),ENT 案例复用既有机制零内核改动 |
| 企业级数据库 | ⚠️ 框架**不内置**关系库连接器(「不存数据」立场):模板给只读 `db` 抽象槽位 + 指向 `RAG-VIA-MCP.md` 的 pgvector 路线,由你把自己的 Postgres 接上 |
| 对外动作 | ⚠️ `client-proposal` 签发后产出是**正式方案文案 + 记录**,不真发给客户、不写回客户库;要真发送 / 真写库需另挂连接器且**仍逐次过审批闸**(接入 ≠ 授权行动) |
| 报价参考 | ⚠️ 基于你知识库里的历史,**不是市场实时行情**,一律标「待确认」 |
| 多成员 | ⚠️ 模板**不携带人**:真实成员账号、角色分配在 hub 运行时另建(模板只给角色门 + 审批闸这套结构) |
| 真人实测 | ❌ 尚无外部专业服务公司真机实测——推广即是找首批小团队照第四节验收标准跑、逐条清摩擦 |

这个案例的定位是**证明向上兼容性**:同一个内核从「一个人的公司」平滑长成「一个小团队」。
推广一人公司(OPC)时**不必等它**——[`solo-company-hub`](SOLO-COMPANY-HUB.md) 本身开箱即用;
这个案例是给「已经做起来、要招人接企业数据」的下一步准备的答案。
