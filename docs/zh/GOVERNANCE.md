# AipeHub 治理

<!-- doc-version: 1.0 -->
> **文档版本 1.0** · 中文译本 · 最后更新 2026-06-27 · 权威源：[English](../../GOVERNANCE.md)。如译文与英文版冲突，以英文版为准。

本文件描述 AipeHub 里**决策如何形成**：谁维护项目、一个改动如何落地、一个社区模板如何进入官方画廊，以及人们意见不一致时会发生什么。它刻意保持精简——项目还年轻，在一个小项目上压一套沉重的治理结构只是排场。我们会随社区成长而扩充本文件，而不是提前。

本文件位于项目宪章 [`CHARTER.md`](CHARTER.md) 之下：宪章说 AipeHub *是什么*、拒绝变成什么；本文件说我们*如何*决策。两者相遇之处——例如「框架不跑 LLM」——宪章是源头，本文件是执行。

如果你只读一句话：**设计底线没有商量余地，但几乎其他一切都有。** 见[唯一不可妥协的底线](#唯一不可妥协的底线)。

## 角色

我们保留三个角色。没有秘密的第四层。

| 角色 | 含义 | 如何获得 |
|---|---|---|
| **贡献者（Contributor）** | 任何开 issue、发 PR、提交模板，或在 Discussions 里帮忙的人。 | 出现即可。无需申请。 |
| **维护者（Maintainer）** | 可审阅并合并 PR、分诊 issue、发布版本。对某个子系统或整个项目负责。 | 一段良好且与设计对齐的贡献记录，然后被公开提名——见[成为维护者](#成为维护者)。 |
| **监护人（Steward）** | 有争议决策的最终仲裁者，设计底线的守护者。今天这是创始维护者。 | 由创始人持有，直到项目大到足以选举监护人为止（见[通往组件委员会之路](#通往组件委员会之路)）。 |

当前维护者列在 [`MAINTAINERS.md`](MAINTAINERS.md) 里——今天那只有创始维护者，他同时是监护人、审阅者和发布经理。本文件存在的意义正是让这种安排**临时且写下来**，而非变成习惯，下一节即第二位维护者所走的路径。

### 成为维护者

晋升阶梯刻意保持轻——这是个年轻项目，目标是培养一批能守住设计底线的人，而不是把门。这是一条粗略的指导，不是一张可被钻空子的清单：

- **看记录，不看计数。** 大约 ~5 个非平凡的已合并 PR——或等价物：一个你持续维护的旗舰模板、一个有分量的适配器，或持续的审阅 / 分诊帮助——历时数月。这个数字是「我们看够了你的工作，足以信任你的判断」的地板，绝不是一个用 drive-by PR 去刷的目标。
- **对设计底线的手感。** 你的 PR 和审阅显示：当逻辑需要安家时，你伸手去找的是一个*参与者*，而不是 Hub（见[唯一不可妥协的底线](#唯一不可妥协的底线)）。
- **公开提名。** 一位现任维护者在公开 issue 上提名你——自我提名也可以，说明理由即可。批准是维护者间的 lazy consensus，监护人确认；你的名字在同一个 PR 里落入 [`MAINTAINERS.md`](MAINTAINERS.md)。

你将承担的：审阅你负责领域里别人的 PR、守护设计底线、回答你所维护内容的 issue。这是一份你也可以放下的责任——随时退一步，我们会在 `MAINTAINERS.md` 里把你移到荣誉退任，而不是假装你还在待命。

今天恰好只有一位维护者——创始监护人——所以这条阶梯**写下来了但休眠中**：目前还没有人可提名。它在这里，是为了让*第二位*维护者通过一条已知路径加入，而不是临时拍肩膀招呼。

## 一个改动如何落地

多数改动都很平淡，平淡是好事：

1. **先开一个 issue**——对任何非平凡的东西：一个新依赖、一次协议形状变更、一个新包、对调度或联邦的行为变更。Drive-by 错别字 PR 和小的文档修复可以跳过这一步。
2. **发一个小 PR。** 一个改动，一个 PR。见 [`CONTRIBUTING.md`](CONTRIBUTING.md)。
3. **一位维护者审阅它。** 审阅按顺序检查三件事：正确性、[设计底线](#唯一不可妥协的底线)、简单性。
4. **合并。** Lazy consensus：如果在合理窗口内没有维护者反对、且 CI / 本地检查通过，就合并。反对通过讨论解决；真正的僵局交给监护人。

我们不要求 CLA。贡献即表示你以本项目的 [MIT license](../../LICENSE) 提供你的工作。

### 需要不止一个 PR 的决策

少数几类会得到额外照顾，维护者会刻意把它们放慢：

- **Wire 协议变更**——任何改变 [`docs/PROTOCOL.md`](../PROTOCOL.md) 里形状的东西。这些会得到一次版本号提升和一份明确的迁移说明。
- **不可逆的 schema 变更**（drop 一列 / 一张表）。即便项目在 1.0 之前不承诺向前兼容，我们也会在丢弃已持久化的数据前讨论其爆炸半径。
- **新的运行时依赖**，尤其是原生依赖。请开一个 issue。
- **移除一个公开 API 表面。** 先描述影响，即使你相信没人在用。

## 模板如何进入官方画廊

AipeHub 附带**模板**（`aipehub.template/v1`——一份自包含的 YAML，携带一支 agent 团队 + 工作流 + 知识库*引用*，但永不含密钥、知识内容或人员）。被**随框架发布**——出现在 admin UI 的一键画廊和公开站点上——的门槛，高于被**接受为社区模板**的门槛。

有两个层级，它们是不同的承诺：

### 社区模板——「我们核对了 license，而且它能解析」

住在 [`templates/community/`](../../templates/community/) 下。要被合并，一个社区模板必须：

1. **能解析。** 它通过真实的 `parseTemplate`（且每个内嵌工作流通过真实的 `parseWorkflow`）。这由一个自动化校验测试强制，而非人眼——见 [`templates/community/templates/README.md`](../../templates/community/templates/README.md)。
2. **携带诚实的溯源。** 如果它派生自另一个模板或一个上游 prompt 库，它在 `provenance` 块里声明（`derivedFrom`、`author`、`notes`）。溯源是引用署名回流到上游的方式——别把它剥掉。
3. **不携带密钥。** 每个凭证都是一个 `${ENV}` 占位符。一个里面带字面密钥的模板会被拒绝，没有例外。
4. **为任何改写过的上游材料有一个清晰、商用友好的 license**（CC0 / MIT / Apache-2.0 / BSD）。非商用或无 license 的来源不被接受。见 [`templates/community/LICENSE-NOTICES.md`](../../templates/community/LICENSE-NOTICES.md)。

就这些。一个达到门槛的社区模板就会被合并。在这个层级我们不策展品味——我们策展的是*安全和诚实*。

### 旗舰模板——「我们为它背书」

一个小的策展集（见 [`docs/zh/FLAGSHIP-TEMPLATES.md`](FLAGSHIP-TEMPLATES.md)），项目主动向非技术用户推荐。在社区门槛之上，一个旗舰模板必须：

1. **附带一个确定性 demo**，无需 API key 即可运行并自我断言自身行为（`examples/*` 约定）。审阅者一条命令就能证明它能跑。
2. **平实地声明其治理立场**——它能触碰什么、不能触碰什么、人在环的位置在哪。一个能锁门、花钱，或把孩子的数据经联邦链路发出去的模板，必须把人工确认闸展示出来，而不是埋起来。
3. **被维护。** 一个旗舰模板有一位维护者回答关于它的 issue。如果它腐烂了而没人愿意修，它会跌回社区层。

从社区 → 旗舰的晋升是一个维护者决策，在 issue 上公开做出。降级同理。

## 当人们意见不一致时

意见不一致是正常且受欢迎的——这是一个设计被压力测试的方式。流程：

1. **在 issue / PR 上谈清楚。** 陈述取舍，不只是结论。「我更喜欢 X」很弱；「X，因为 Y，代价是 Z」才有用。
2. **如果讨论卡住，一位维护者拍板。** 维护者应在记录上解释*为什么*。
3. **监护人是仲裁者**，用于真正有争议的判断，也是一个改动是否越过设计底线的最终权威。这是一道兜底，不是常规步骤——一个常常不得不仲裁的监护人，是一个未能壮大维护者梯队的监护人。

行为准则争议另行处理——见 [`CODE_OF_CONDUCT.md`](CODE_OF_CONDUCT.md)。

## 唯一不可妥协的底线

AipeHub 恰好有一条架构承诺，是一个 PR 投票投不掉的，因为改变它意味着项目不再是 AipeHub：

> **框架不跑 LLM。** Hub 路由消息、派发任务、写 transcript、发事件。每一个决策都留在参与者手里——agent、人、外部服务。状态是磁盘上的文件；凭证留在本地；联邦是点对点的，带显式的 per-link 边界。

把 LLM 调用、agent 循环或业务规则塞进 *Hub* 的补丁会被引导改向——不是因为这个想法不好，而是因为它属于一个参与者，而非底座。其他一切——调度器、provider、适配器、传输、UI、模板——都对改变开放。

## 通往组件委员会之路

本项目的长期形态是一个**受治理、可复用组件的市集**——模板、适配器、知识库连接器——人们信任到敢把它指向自己的家、家人或钱。策展那个市集，不是一个人能做的，也不是一个人*应该*做的。

当贡献者基数大到画廊策展成为一份真实、周期性的工作时，我们会成立一个**组件委员会**：一个由维护者组成的小型选举团体，负责什么被晋升为旗舰、引用署名如何呈现、模板作者之间的争议如何解决。届时本文件会被修订，描述委员会成员如何被提名、选举和轮换。

我们现在、趁项目还小，就写下这一段，是为了让这个委员会成为一个**有书面授权的计划里程碑**，而不是日后临时的权力攫取。触发条件是持续的贡献量，而非一个日期。

## 区域分会

终态是一张主权 hub 的自由图，而非一个中心平台——而这张图的人的一侧是**区域分会**：本地的、以语言或社区为根的群体，运营自己的 hub、为自己的人策展模板、用自己的母语帮助新人。宪章（[`CHARTER.md`](CHARTER.md) §11）欢迎它们；本节说一个分会在实践中如何运作。

一个分会是**主权的，不是加盟店。** 它拥有自己的房间，不向任何中央房东负责。*开办*一个分会不需要任何人的许可——那会与整个「没有任何一方的许可是你保持运行所必需的」前提相矛盾，而项目正建立在这个前提之上。你今天就可以立起一个 hub、聚集一个本地社区、为他们策展模板，并称它为一个 AipeHub 分会。

一个分会**不是**什么：

- **不是项目的官方声音。** 一个分会为它自己的社区发声，不为 AipeHub 发声。它不设定设计底线、不批准宪章、不决定什么被晋升为旗舰——那些留在权威仓库（[`MAINTAINERS.md`](MAINTAINERS.md)）上的维护者和监护人手里。
- **不是守住底线的 fork。** 一个分会可以为它自己的人运行一个改过的构建，但权威的宪章、协议和旗舰集住在权威仓库里。一个希望它的改动成为*那个* AipeHub 的分会，会像任何人一样把它们作为 pull request 送上游。

### 认可是可选且轻量的

运营一个分会不需要任何祝福。被**列出**为一个受认可的分会——从项目链接过去，让新人能找到你——是一个小的、opt-in 的步骤，它的运作很像模板晋升：

1. **在 Discussions 里宣布它**——你是谁、你服务的区域 / 语言 / 社区，以及你的 hub 在哪。
2. **一位维护者在公开 issue 上为它背书。** 门槛是诚实，不是规模：你真实地代表项目、你遵循 [`CODE_OF_CONDUCT.md`](CODE_OF_CONDUCT.md)、你不宣称你并不拥有的官方身份。
3. **认可可以被撤回**，方式与授予时相同——在公开 issue 上、附理由——如果一个分会歪曲了项目或破坏了行为准则。项目无法（也不会试图）关停一个分会的 hub——那是它的主权房间——但它可以停止列出它，并请它停止暗示一个它已不再拥有的背书。

当**组件委员会**成立时（见上文），策展分会列表、解决分会间争议会自然成为它授权的一部分；在那之前，这是一个轻量的维护者判断。

## 使用 AipeHub 名称

代码是 [MIT](../../LICENSE) 的——你可以嵌入它、修改它，并在商业或闭源产品中发布它，只要保留 license 和版权行。**license 覆盖代码；它不交出项目的名称和身份。** 这里没有注册商标，我们也不会假装有——下面是一条**社区规范**，本着善意请求，不是法律威胁：

- **描述性使用受欢迎。**「基于 AipeHub 构建」「一个 AipeHub hub」「AipeHub 马来西亚分会」——这些尽管说。它们是真的，我们也很高兴你在说。
- **不要暗示一个你并不拥有的背书。** 不要以某种方式命名一个产品、fork 或服务，使它呈现得*就是* AipeHub 项目本身，或像被它官方祝福——没有「AipeHub Official」，没有冒充权威下载的山寨货。
- **不要给权威项目改头换面。** 一个改过的发行版归你发布，但权威的「AipeHub」——宪章、设计底线、旗舰集——是权威仓库里的那一个。如果你的 fork 在精神上分道扬镳，给它起你自己的名字；MIT license 保证你可以，而一个诚实的名字比一个借来的名字更好地服务你的用户。

这是能起作用的最轻量的名称保护版本：足以让「AipeHub」持续意味着宪章所描述的那个受治理、file-first、人在环的东西，仅此而已。

## 修订本文件

治理的改变方式与代码相同：开一个 issue、发一个 PR、获得维护者审阅。对[唯一不可妥协的底线](#唯一不可妥协的底线)的改动需要监护人签字，并清楚陈述为什么设计底线应当移动。我们预期那一节永不改变。其余部分则意在生长。
