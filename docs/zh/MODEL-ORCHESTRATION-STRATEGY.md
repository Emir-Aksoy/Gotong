# 模型编排策略研究:什么时候多模型齐发 / 轻量 / 重量级

> 缘起:用户 2026-07-14「什么时候多模型一起发力,什么时候轻量,什么时候用重量级
> 模型,这些研究一下」。本文是研究结论 + 落到 Gotong 的决策策略,喂给 LSA-M4 显式
> 推迟的 ensemble 配置面设计。**不含代码**,是策略骨架(同 MR-M0 / LSA-M0 的定位)。

---

## 零、一句话结论(先给答案)

**「什么时候用哪种」在 Gotong 里不是现场智能选路,而是一张策略表**——按 agent、按
任务类型、按**确定性信号**(消息长短 / 要不要调工具 / 前台交互还是后台维护 / 用户
显式旗标 / provider 健康 / 额度余量)决定,不是每条消息现场用一个 LLM 判难度再选。
这既是约束也是**特性**:可预测、可审计、零隐藏成本。

| 场景信号(确定性) | 用哪种 | Gotong 现状 |
|---|---|---|
| 后台 / 维护活(6h 记忆蒸馏、摘要)、高频、低风险、机械任务(分类 / 抽取 / 格式化) | **轻量**:便宜小快模型,单发 | ✅ 已落(NA-M5 `maintenanceModel`) |
| 前台交互默认、难推理 / 代码 / 长上下文、正确性要紧且量不大 | **重量级**:强模型,单发 | ✅ 已落(agent 主 provider) |
| 一次性 / 高价值 / 无 ground truth 复核 / 用户明确「要最好的答案」/ 扛得住 N× 成本+延迟 | **多模型齐发**:并行综合(ensemble) | ⚙️ 纯核已落(LSA-M4),配置面推迟 |
| 某厂商挂了 / 限流 / 超时 | **顺序降级**:换下一候选(routing failover) | ✅ 已落(MR track) |

---

## 一、一条不可破的约束先钉死:热路径零 LLM 选路

Gotong 北极星边界①**热路径零 LLM 决策**。这直接框死了「怎么决定用哪种模型」:

- **不能**在收到消息时先用一个 LLM 判「这条难不难 / 该用哪个模型」——那就是热路径
  跑 LLM 做决策。这正是 MR track 里**否掉 Hermes「Pareto Router」的原因**:它按任务
  类型分 8 槽选模型,本质要先用大模型读消息分类,撞边界①。
- **推论**:选路只能靠**策略(人预设 / 配置)+ 确定性信号**。可用的零 LLM 信号——
  - agent 身份 / 工作流步 id(这个 agent / 这一步天生用哪种);
  - 任务来源:后台 sweep vs 前台交互(`task.from`,A4 已在读);
  - 消息长度 / 是否带附件 / 是否要调工具(结构性可见,无需 LLM);
  - 用户**显式旗标**(「这个问题请多模型综合一下」——是用户点的,不是模型猜的);
  - provider 健康(MR-M3 RoutingHealthTracker)/ 额度余量(未来额度管家)。
- **绝不做**:一个 LLM 现场读消息判难度再选模型(= Hermes Pareto Router = 撞边界①)。

> 「智能」永远在**候选的排序 / 策略的设定**(便宜的打头、强的兜底、这个 agent 走
> ensemble),不在现场用大模型选路。这跟 MR track「确定性路由」一脉相承。

## 二、三种编排模式 + 真实数据(field survey,2026-07-14 核)

### A. 单模型 routing(顺序 failover)—— ✅ 已落(MR track)

有序候选链,首选挂了才换下一个。同一刻只有一个模型在跑。省钱 + 容错,不提质量。

### B. 多模型 ensemble(并行综合)—— ⚙️ 纯核已落(LSA-M4)

同一请求并行发给 N 个成员,收齐 N 份草稿综合成一份。**证据是硬的**:
- **Mixture-of-Agents**(Together AI,arXiv 2406.04692):**只用开源模型**集成,
  AlpacaEval 2.0 拿到 **65.1%**(更新到 65.8%),**超过 GPT-4 Omni 的 57.5%**;
  MT-Bench / FLASK 同样超。机制=「**协作性**」:一个模型看到别的模型(哪怕更弱)的
  答案后,自己会答得更好。
- **代价诚实**:①成本 ×N;②**延迟**——综合必须等所有草稿到齐,Time-To-First-Token
  被抬高(这正是 LSA-M4 缓冲式而非流式的原因);③并发压力(N 个端点同时打)。
  缓解旋钮=**MoA-Lite**(少放几个成员);未来方向=分块综合降 TTFT。

### C. 级联 cascade(便宜先 → 不够再升)—— ❌ 未落(下面论证为什么大部分不能落)

最便宜的模型先跑,用一个「够不够好」的信号判断,不够才升级到强模型。**证据也硬**:
- **FrugalGPT**:三级级联,匹配前沿模型质量的同时省最多 **98%** 成本(HellaSwag /
  MMLU / CommonsenseQA)。
- **RouteLLM**:**2× 省钱**同时保住 GPT-4 **95%** 的质量。
- 原理=**难度高度偏斜**:绝大多数请求是routine的、少数才真难,却被一股脑全发给
  贵模型。级联/路由吃这个偏斜。
- **但有个红色警告**:「一个只调成本、没有质量护栏的路由器,会**静默塌掉没人批准的
  质量**」——escalation 阈值必须来自评测数据、每条路由的质量要被监控。

| | A routing | B ensemble | C cascade |
|---|---|---|---|
| 并发 | 顺序(1 个) | 并行(N 个) | 顺序(1→升级) |
| 成本 | 1× | **N×**(+综合) | **<1×**(吃偏斜,可省 90%+) |
| 延迟 | 1×(失败才叠加) | **高**(等齐+综合) | 便宜活低,难活叠加 |
| 提质量? | 否(只容错) | **是**(协作性,超单模型) | 否(目标是省钱不掉质量) |
| 升级触发 | 硬失败(抛错) | —— | **「够不够好」信号** ← 命门 |

## 三、决策策略:什么时候用哪种(把上面落成规则)

全部由**确定性信号 / 配置**驱动,零现场 LLM:

- **轻量(便宜小快,单发)**——后台 / 维护、高频、低风险、延迟敏感、机械任务。
  信号:后台 sweep(6h 蒸馏已用 `maintenanceModel`)/ 短消息 / 无工具 / 无人等。
- **重量级(强模型,单发)**——难推理 / 代码 / 长上下文 / 正确性要紧且量不大。
  信号:交互式 + 高价值 agent(**由 agent 配置定,不是现场判**)/ 长上下文。
- **多模型齐发(ensemble)**——一次性、高价值、无 ground truth 复核、用户明确要
  最好答案、扛得住 N× 成本+延迟。信号:**用户显式旗标** / 特定「重要决策」agent /
  一次性研究报告 workflow 步。**默认关**(N× 成本 + 延迟不是白来的)。

### 关键结论:Gotong 版 cascade 只能做「结构性失败即升」,做不了「质量不够即升」

Cascade 的命门是 escalation 信号。「答得**够不够好**」这个判断:
- 若用一个 LLM 判 → **热路径跑 LLM**,撞边界①;
- 若不用 LLM 判、只调成本 → field 明确警告**静默塌质量**。

两条路我们都不走。**能用的零 LLM 信号只有「结构性失败」**:便宜模型报错 / 拒答 /
空产出 / 超时 / stopReason 异常 → 升级。而这恰好**就是 MR 的 failover**(便宜打头、
失败换强的)。所以:

> **cascade =「便宜打头的 routing failover」**。它「按质量升级」的那半在我们的约束下
> 做不了(要么热路径 LLM、要么静默塌质量);能做的那半 MR track 已经做了。结论=
> **不单列 cascade 模式,把「便宜候选打头 + 强模型兜底」当 routing 的一种候选排序**即可。

## 四、落到 Gotong:配置面怎么设计(喂给 LSA-M4 推迟的那块)

- **per-agent 编排策略字段**(additive,Hub 不解释):
  `single`(默认=现状)/ `routing`(fallbacks,已落)/ `ensemble`(members+strategy,
  M4 纯核已落待配置面)。cascade 不单列(=routing 的候选排序)。
- **信号驱动的默认**:后台 sweep → `maintenanceModel`(已落);前台交互 → single;
  高价值 agent → 人配 ensemble。全是配置 / 确定性信号,无现场 LLM 选路。
- **成本护栏**:ensemble 面板印 **N× 成本徽章**(M4 已列为推迟项);我们**不做**自动
  质量路由,所以天然免疫 field 警告的「静默塌质量」。
- **用户显式旗标**:/me 或管家里「这个问题请多模型综合一下」→ 那一轮走 ensemble
  (agent 层**显式**,不是现场 LLM 猜)。这是 ensemble 最干净的触发方式。

---

## 五、显式不做(边界)

- **现场 LLM 判难度选路**(= Hermes Pareto Router,撞边界①热路径零 LLM)。
- **按「答得好不好」自动升级的质量路由**(要么热路径 LLM 判、要么静默塌质量,field
  两头都警告)——只做「结构性失败即升」,那部分 MR failover 已覆盖。
- **每条消息级的成本 / 难度自适应**——难度自适应的收益(FrugalGPT 式省钱)以「LLM
  判难度」为前提,与边界①冲突;我们用**per-agent / per-task 策略**换取可预测 + 可审计。

## 六、下一步(如果继续)

1. **ensemble 配置面**(LSA-M4 推迟项):`ManagedAgentSpec.ensemble` 字段 + pool 装配缝
   (镜像 `buildRoutedProvider`)+ admin 面板 + 成本徽章。

   (「自己管理」的合法版,阿同监控+提醒、授权动作人点)。
3. **用户显式「综合一下」旗标**:管家 / /me 里一句话触发那一轮 ensemble。
