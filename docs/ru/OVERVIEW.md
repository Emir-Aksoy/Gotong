## В одном предложении

**AipeHub** — это **самохостируемое рабочее пространство для совместной работы на TypeScript и
Python**: люди и агенты ИИ делят одну «комнату», а намеренно простой Hub
распределяет задачи, собирает результаты и записывает весь прогон.

Это **не фреймворк для агентов** (он не запускает LLM) — это **субстрат
для многоучастниковой совместной работы**, где организации могут федерироваться
**не передавая своих ключей, данных или биллинга**.

---

## Что это такое — и что лежит *под* этим

Большинство «агентных» проектов — это агент или фреймворк для написания цикла одного агента
(LangGraph, CrewAI, AutoGen). AipeHub — **ни то, ни другое** — это слой, в который они
*подключаются*. Граф LangGraph, команда CrewAI, агент кодирования CLI (Claude Code,
Codex), внешний A2A-агент и человек — все присоединяются к одной комнате как один
`Participant`. Hub маршрутизирует их сообщения, распределяет задачи, записывает
transcript и обеспечивает соблюдение границ — он **никогда не запускает LLM**, поэтому каждое
решение остаётся за участником.

Три вещи делают его чем-то большим, чем шина сообщений:

- **Равные участники** — человек является `Participant`, точно так же, как и агент.
  Нет «инструмента request-human-input»; люди и агенты сотрудничают через
  одни и те же задачи + transcript и одни и те же асинхронные / долговременные примитивы.
- **Управление** — чувствительные и межорганизационные действия не выполняются просто так.
  Они могут требовать от человека одобрения из его входящих (предложение → рецензия →
  подтверждение) с полным журналом аудита.
- **Суверенитет** — каждое рабочее пространство является директорией на диске, принадлежащей вам. Когда
  две организации федерируются, учётные данные, данные и биллинг каждой остаются дома;
  то, что пересекает черту, ограничено **контрактом доверия для каждой связи**.

Это сочетание — а не какой-то один умный протокол — и есть AipeHub. Это первый субстрат,
объединяющий равенство людей и агентов, управляемую межорганизационную федерацию и
самохостируемый суверенитет в одном готовом к запуску, файл-первичном пакете.

## Четыре грани — как AipeHub соединяется с миром

AipeHub достигает остальной экосистемы через четыре грани. Он **говорит на открытых
протоколах там, где они существуют** — он не изобретает их заново:

| Грань | Протокол | Направление | Что несёт |
|---|---|---|---|
| Инструменты и данные | **MCP** | оба | Агенты вызывают внешние MCP-инструменты; внешние клиенты (Claude Desktop, Cursor) управляют Hub. |
| Агент ↔ агент | **A2A** | оба | Входящий `message/send` становится dispatch; исходящий вызов управляет удалённым A2A-агентом. |
| Агенты кодирования | **ACP** | исходящий | Hub порождает и поддерживает сессию с Claude Code / Codex и управляет ею пошагово. |
| Hub ↔ hub | **HubLink** | оба | Собственная федеративная связь AipeHub между двумя hub'ами — место, где живут контракты доверия для каждой связи, пересылка задач с сохранением родословной и ворота одобрения. |

Первые три — это стандарты экосистемы, которые реализует AipeHub. HubLink — единственная
часть, которой он владеет — **не** как умный проводной формат (под ним находятся WebSocket + bearer token +
JSON-RPC), но как **контракт о том, что обменивают два управляемых hub'а**:
манифест возможностей, пересылка задач с сохранением родословной и контракт доверия для каждой связи ниже.

→ Подробнее: [`MCP.md`](./MCP.md) · [`FEDERATION.md`](./FEDERATION.md) · [`PROTOCOL.md`](./PROTOCOL.md)

## Как участники подключаются

Основной путь — **два способа добавить LLM-агента**:

| Путь A · Управляемый хостом | Путь B · Внешний SDK |
|---|---|
| Заполните форму / импортируйте YAML / вставьте шаблон в admin UI → хост порождает `LlmAgent` внутри своего процесса | Напишите код (Node / Python), реализующий `AgentParticipant.handleTask`, затем `connect(url, agents)` к WebSocket-порту Hub |
| **0 строк кода** | Вы пишете код |
| Только LLM-агенты (обёрнутые Anthropic / OpenAI / Mock) | **Любого типа** (LLM, скрейперы, локальные инструменты, приватная логика, Python ML-модели) |
| Ключ провайдера зашифрован на диске в `secrets.enc.json` (на агента или по умолчанию для рабочего пространства), или читается из env | Вы управляете API-ключом; агент работает на вашей машине |
| Автоматически перезапускается при перезапуске хоста | Вы управляете его жизненным циклом; SDK имеет встроенное автоматическое переподключение |
| Лучше всего для: обычных пользователей / стандартных LLM-ролей / запуска за 60 секунд | Лучше всего для: разработчиков / приватных данных / не раскрывая свой код |

→ Путь A: [`HUMAN.md §1 Agents`](./HUMAN.md#1-智能体v21) + [`TEMPLATES.md`](./TEMPLATES.md)
→ Путь B: [`AGENT.md`](./AGENT.md)

…и поскольку всё является одним `Participant`, та же комната также принимает:

- **Агенты кодирования CLI / ACP** — Hub управляет Claude Code / Codex через удерживаемую
  ACP-сессию (верифицировано на реальных машинах), с воротами опасных действий, которые могут задержать
  деструктивные команды для одобрения человеком.
- **Внешние A2A-агенты** — зарегистрируйте удалённый агент под возможностью; шаг
  workflow маршрутизирует к нему, как к любому другому.
- **Адаптеры фреймворков** — оберните граф LangGraph или команду CrewAI как
  `Participant` через Python SDK; сам фреймворк никогда не импортируется
  Hub.

Все они **смешиваются свободно** — одна комната может содержать управляемый хостом `writer-zh`, ваш
подключённый по SDK `rag-agent` и сессию кодирования Codex, полностью прозрачно для
планировщика.

## Несколько людей в одной комнате

AipeHub моделирует «команду» как **одну комнату** = одну директорию `.aipehub/`. Три уровня ролей:

| Роль | URL | Что можно делать в этой комнате |
|---|---|---|
| **admin** | `/admin` | Настраивать комнату, одобрять/отклонять запросы агентов, распределять задачи, оценивать работу, приглашать других администраторов |
| **worker** | `/` (рабочее место `/me`) | Выбрать никнейм + работу, которую вы можете делать, запускать для себя workflow, обработать свои входящие, завершать или отклонять задачи |
| **agent** | WS-порт | Автоматически получать распределённые задачи, возвращать результаты |

### Типичный workflow небольшой команды (сценарий)

```
0  Alice (admin) запускает hub → при запуске браузер показывает одноразовый
   URL администратора; она сохраняет его в 1Password.
1  Alice настраивает ключ провайдера в admin UI → ключ рабочего пространства по
   умолчанию шифруется на диске.
2  Alice устанавливает шаблон (или импортирует storyteller.yaml) → хост
   немедленно порождает LLM-агента, показанного как онлайн.
3  Alice отправляет пригласительные URL Bob'у и Carol. Они выбирают никнеймы, отмечают
   возможности, которые могут делать (draft / review) → они в комнате.
4  Alice распределяет задачу: «написать детскую историю о настойчивости»,
   стратегия = capability:[story] → управляемый хостом storyteller принимает её
   → через 30 с возвращается история из 600 слов.
5  Шаг workflow требует подписи → он паркуется во входящих Bob'а; Bob одобряет его
   со своего рабочего места /me, и прогон возобновляется — человек в контуре, а не
   вызов инструмента.
6  Alice оценивает работу; таблица лидеров вкладов обновляется; каждое
   событие находится в transcript.jsonl, поэтому после сбоя + перезапуска восстановление происходит полностью.
```

**Ключевые концепции** (подробности в HUMAN.md):

- **Три стратегии dispatch**: `direct` (по имени), `capability` (по навыку), `broadcast` (первый, кто принимает, выигрывает)
- **Human-in-the-loop (человек в контуре)**: шаг workflow может направить задачу во входящие человека и ждать одобрения / выбора / правки перед продолжением
- **Рабочее место `/me`**: члены запускают свои собственные workflow, видят свои последние прогоны, управляют своими агентами (BYO-ключ), всё ограничено собой
- **API-ключ, три уровня**: приватный на агента → рабочее пространство по умолчанию → переменная окружения

→ Полное описание: [`HUMAN.md`](./HUMAN.md)

## Между организациями — управляемая федерация

**Два разных значения «многокомандного» — не путайте их:**

### Одна комната, много ролей (= раздел выше)

Все находятся в одной директории `.aipehub/`, одном hub-процессе. Это по умолчанию.

### Много комнат, федерированных (= настоящий межорганизационный)

Каждая организация запускает свой независимый hub (свой `.aipehub/`, своих людей и
агентов, **собственные API-ключи и собственный биллинг**). Два hub'а соединяются через
**HubLink**, и то, что один может просить другого, фиксируется **контрактом доверия для каждой связи**:

- **allowlist возможностей** — именно какие возможности peer может вызывать
- **ворота класса данных** — какие классы данных разрешены пересекать связь (fail-closed)
- **квота** — потолок частоты / бюджета на связь, сохраняемый между переподключениями
- **отзыв** — разрыв связи в любое время
- **allowlist базы знаний** — какие общие KB может достичь peer

Простейший паттерн — `TeamBridgeAgent`: целый под-hub появляется апстриму как
**один агент**, его внутренние члены / ключи / под-задачи невидимы
родителю.

```
   Company Hub (Bob is admin)
       │
       ├── agent · alice-team   ←─┐
       │                          │  TeamBridgeAgent  (over HubLink)
       │                  ┌───────┴────────┐
       │                  │ Alice's Hub    │ (Alice is admin)
       │                  │  · writer-bot  │   keys / people / billing
       │                  │  · reviewer-bot│   all stay on Alice's hub
       │                  └────────────────┘
       └── agent · david-team   ←── another team, same idea
```

Помимо бриджинга, **workflow на одном hub'е может выполнить шаг с возможностью другого hub'а**.
Если этот peer требует одобрения, шаг паркуется во входящих человека до тех пор, пока кто-то
его не одобрит — межорганизационный вызов управляется, двухступенчато и полностью
проверяем, и YAML workflow никогда даже не называет peer'а (только возможность;
связь — это конфигурация рантайма).

**Почему это важно — суверенитет сохраняется:**

- Апстрим видит *агрегированные результаты* («alice-team выполнил N задач»), никогда ключи peer'а или сырые данные
- Каждый hub сохраняет **собственное хранилище учётных данных** и **собственный учёт использования / стоимости** — биллинг на hub
- Хотите приватный внутренний PoC? Запустите локальный hub — нулевые затраты на онбординг
- Хотите, чтобы вся компания сотрудничала? Повесьте управляемую связь поверх — **не касаясь существующей командной структуры**
