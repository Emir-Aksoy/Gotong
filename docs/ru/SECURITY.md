# Политика безопасности

<!-- doc-version: 1.0 -->
> **Версия документа 1.0** · Перевод на русский язык · Обновлено: 2026-06-27 · Авторитетный источник: [English](../../SECURITY.md). При расхождении перевода с английской версией приоритет имеет английская версия.

## Как сообщить об уязвимости

**Пожалуйста, не открывайте публичный GitHub issue, обсуждение или PR для проблем
безопасности.** Используйте частный канал:

### Предпочтительно — частный отчёт об уязвимостях через GitHub

Откройте частный advisory по адресу:

> **<https://github.com/Emir-Aksoy/AipeHub/security/advisories/new>**

Встроенная форма GitHub предоставляет:

- сквозной закрытый тред с мейнтейнерами (без утечки через электронную почту)
- вложения + шаги воспроизведения в одном месте
- отслеживаемую временну́ю шкалу от отчёта → исправления → присвоения CVE

Это канал, который мы читаем первым и отвечаем на него быстрее всего. Вам понадобится
бесплатная учётная запись GitHub; это единственное требование.

### Нет канала электронной почты (до версии 1.0)

В период v0.x намеренно **нет электронной почты безопасности**.
`security@aipehub.dev` появлялся в более ранних версиях этого репозитория как
*желательный* адрес — домен не зарегистрирован и почтовый ящик
не активирован, поэтому письма на него никуда не приходят. Мы перестали рекламировать
его в качестве запасного варианта, чтобы не оставлять мёртвый контакт, которому кто-то может доверить
реальный отчёт.

GitHub Private Vulnerability Reporting (выше) является **единственным** каналом
сегодня: бесплатным, закрытым и тем, который мы читаем первым. Вопрос о том, стоит ли
создавать настоящий почтовый ящик, является решением [контрольного списка выпуска](.github/RELEASE-CHECKLIST.md#security-contact),
отложенным до подготовки к версии 1.0; до тех пор, пожалуйста, используйте форму advisory.

Если вы действительно не можете использовать GitHub, откройте **не связанное с безопасностью** GitHub
Discussion с просьбой к мейнтейнеру выйти на связь — без каких-либо деталей уязвимости — и
мы организуем частный канал для этого одного отчёта.

Включите в свой advisory:

- описание проблемы
- точные шаги воспроизведения
- хеш commit'а, против которого вы тестировали (`git rev-parse HEAD`)
- (опционально) предложенное исправление или патч
- (опционально) имя / псевдоним, который вы хотите указать в advisory

### Как насчёт PGP?

Мы **не** публикуем PGP-ключ сегодня. Причины:

- Частный канал advisory GitHub уже зашифрован TLS сквозным образом
  между вами и уведомлением мейнтейнера, поэтому PGP добавляет немного.
- Поддержание PGP-ключа для раннего проекта создаёт больше режимов отказа
  (потерянные ключи, просроченные ключи, церемонии подписи), чем пользы.

Если политика вашей организации требует раскрытия, зашифрованного PGP, пожалуйста,
свяжитесь с нами через GitHub-канал сначала, и мы организуем внеполосный
PGP-обмен для этого одного отчёта.

## Сроки реагирования

| Фаза | Цель |
|---|---|
| Подтверждение получения | в течение **72 часов** с момента отчёта |
| Первичная сортировка + оценка серьёзности | в течение **7 дней** |
| Исправление или смягчение в `main` | высокая серьёзность: **7 дней**, средняя: **30 дней**, низкая: по возможности |
| Публичное раскрытие | **7–14 дней** после применения исправления (или по взаимному соглашению) |

Вы будете получать обновления на каждом этапе. Если вы не получаете ответа от нас
в течение 72-часового окна подтверждения, это само по себе является ошибкой —
пожалуйста, обострите ситуацию через GitHub Discussion (общий, без содержимого безопасности),
пометив мейнтейнера.

## Поддерживаемые версии

AipeHub находится на стадии до версии 1.0 внутри (метки v2.0 / v2.1, которые вы видите в
`CHANGELOG.md`, относятся к поколению перезаписи с приоритетом файлов, а не к порогу
SemVer 1.0). Мы исправляем проблемы безопасности только в текущей ветке `main`. Нет
**ни одной ветки LTS**.

Если вам нужна долгосрочная стабильность, зафиксируйтесь на commit'е, который вы проверили,
и предусмотрите бюджет для патчей на месте; мы не можем портировать обратно бесконечно.

## Модель угроз

AipeHub предназначен для **небольших, доверенных, однопользовательских** развёртываний —
исследовательская лаборатория, проектная команда, небольшая группа публичного предпросмотра. Настройки по
умолчанию предполагают, что пространством управляют люди, которые доверяют друг другу.

В области применения (мы принимаем отчёты о):

- ✅ Неаутентифицированный доступ к конечным точкам администратора
- ✅ Раскрытие токенов / cookie (между пользователями, между комнатами, между процессами)
- ✅ Ошибки шифрования / дешифрования в `secrets.enc.json` и файле мастер-ключа
- ✅ Обход авторизации — например, worker, достигающий маршрутов только для администратора
- ✅ CSRF / кликджекинг / XSS в поставляемом admin UI
- ✅ Исчерпание ресурсов, не требующее аутентификации (анонимный DoS)
- ✅ Ошибки парсинга проводного протокола, приводящие к сбою хоста или повреждению transcript
- ✅ Эскалация привилегий в `TeamBridgeAgent` (например, локальная команда получает незапланированную видимость апстрима)
- ✅ Проблемы confused deputy в LocalAgentPool / пути spawn управляемого агента

Вне области применения (низкий приоритет — патчи приветствуются, но не обрабатываются как безопасность):

- ❌ **Недоверенные администраторы.** После того как учётная запись получает роль администратора, она может делать
  всё, что открывает роль администратора. Если вам нужен внутренний брандмауэр для администраторов,
  откройте запрос на функцию.
- ❌ **DoS на уровне приложения** *аутентифицированным* пользователем. Ограничение частоты
  осуществляется по IP и сбрасывается при перезапуске; это не защита от преднамеренного
  внутреннего злоупотребления.
- ❌ **Огромные полезные нагрузки задач**, вызывающие давление памяти. Квот пока нет.
- ❌ **Атаки по боковым каналам** вне сравнения токенов (само сравнение токенов
  выполняется за постоянное время).
- ❌ Проблемы, требующие физического / shell-доступа к хост-машине.
- ❌ Находки против исходников `templates/community/` — это сторонние репозитории
  промптов под своими лицензиями и управлением; сообщайте им напрямую.

Если ваша находка находится на границе, отправьте её через канал GitHub advisory,
и мы проведём сортировку.

## Существующие меры защиты (чтобы вы знали, какие защиты уже существуют)

При оценке проблемы проверьте, не покрывает ли её уже одна из этих мер:

- **Хранение токенов**: токены администратора / worker'а хешируются с SHA-256
  перед записью на диск. Открытый текст показывается ровно один раз при создании.
  Верификация использует сравнение с постоянным временем.
- **Хранение cookie**: HttpOnly всегда; `SameSite=Strict` + `Secure` когда
  `AIPE_COOKIE_SECURE=1` (требуется за HTTPS).
- **CSRF**: `AIPE_ALLOWED_HOSTS` применяет проверки как `Host:`, так и `Origin:`
  для каждого изменяющего состояние метода. **Устанавливайте это при каждом производственном
  развёртывании.** Неустановленное означает «безопасен только loopback».
- **Ограничение частоты**: `AIPE_ADMIN_RATE_MAX` / `_SEC` ограничивает попытки
  верификации токена администратора на IP на скользящее окно. По умолчанию 10 / 60 с.
- **Заголовки безопасности**: `X-Frame-Options: DENY`, строгий CSP,
  `Referrer-Policy: no-referrer`, `X-Content-Type-Options: nosniff`
  на каждом ответе.
- **Контроль допуска**: `AIPE_GATING=admin-approval` (по умолчанию) требует,
  чтобы каждый удалённый агент был одобрен человеком перед присоединением. `gating=open`
  предназначен **только для разработки** и отклоняется в production с предупреждением при запуске.
- **Шифрование API-ключей**: API-ключи рабочего пространства и отдельных агентов живут в
  `<space>/secrets.enc.json`, AES-256-GCM, мастер-ключ в
  `<space>/runtime/secret.key` (0600) или переменная окружения `AIPE_SECRET_KEY`. Только
  зашифрованного файла недостаточно для восстановления ключей.
- **Привязка идентичности к агенту (v0.4)**: `authenticate()` может возвращать
  `{ ok: true, allowedAgents: [...] }`, поэтому утёкший API-ключ не может
  выдать себя за произвольный id агента — только за те, к которым он привязан.
- **Transcript только для добавления**: нет API для удаления или перезаписи
  записей transcript из runtime. Подделка требует доступа к файловой системе
  (что находится вне области применения; см. «вне области применения» выше).

## Скоординированное раскрытие

Мы следуем стандартному скоординированному раскрытию:

1. Вы отправляете детали конфиденциально (предпочтительно канал GitHub advisory).
2. Мейнтейнеры подтверждают, определяют область, разрабатывают и тестируют исправление.
3. Исправление попадает в `main` (и ветку backport, если есть какое-либо обязательство LTS).
4. Публичное раскрытие через 7–14 дней с:
   - id CVE (мы запросим один, если это уместно)
   - благодарностью вам в advisory, если вы не просите остаться анонимным
   - кратким изложением воздействия + смягчения в `CHANGELOG.md`

Если вы раскрываете публично до того, как мы выпустили исправление, мы всё равно выпустим
исправление, но поле благодарности advisory отметит «нескоординированное».

## Контрольный список безопасности для операторов

Если вы **запускаете** hub, а не сообщаете об ошибках против него, контрольный список
усиления безопасности со стороны развёртывания находится в
[`docs/DEPLOY.md` § "Production checklist"](../DEPLOY.md#production-checklist).

Вкратце:

- [ ] `AIPE_COOKIE_SECURE=1` при наличии HTTPS перед ним
- [ ] `AIPE_ALLOWED_HOSTS` установлен на ваши реальные имена хостов
- [ ] `AIPE_GATING=admin-approval` (никогда не `open` в публичном интернете)
- [ ] Caddy / nginx завершает TLS; backend привязан к `127.0.0.1`
- [ ] `runtime/secret.key` (chmod 600) или переменная окружения `AIPE_SECRET_KEY` установлена
- [ ] Резервные копии существуют для директории `<space>/`
- [ ] Минимум 2 учётные записи администратора, чтобы можно было восстановиться при блокировке
- [ ] `/healthz` мониторится
