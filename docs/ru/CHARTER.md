## §1 Предназначение

Gotong — это **основа для связи между людьми, агентами и организациями** в эпоху ИИ.

```
   Слой 1  Человек ↔ собственный ИИ / агент
            «Мой ИИ-рабочий стол»: личный hub, приватные workflow,
            учётные данные хранятся только на локальной машине.
            Цель: запустить за 5 минут без написания кода.

   Слой 2  Человек / агент ↔ другой человек / агент / организация
            «Межорганизационное сотрудничество»: несколько пользователей,
            роли, приглашения, кросс-hub федерация.
            Цель: workflow, пересекающие границы организаций,
            при этом учётные данные/данные/биллинг остаются у каждой стороны.

   Слой 3  Сам фреймворк
            «Понятный + стабильный + адаптируемый»: Hub намеренно прост,
            файлы — первичны, `Participant` — единая абстракция,
            протокол / учётные данные / квоты имеют явные границы.
            Цель: workflow реально работают, успевают за быстрым развитием ИИ.
```

Три принципа, которые **никогда не нарушаются**:

1. **Фреймворк не запускает LLM.** Hub только маршрутизирует сообщения, распределяет
   задачи, пишет transcript, отправляет события. Принятие решений всегда остаётся
   за участниками (агентами / людьми / внешними сервисами).

2. **Человек и агент — это один и тот же `Participant`.** Человека не нужно
   представлять как инструмент `request_human_input`. Всё взаимодействие между
   людьми и агентами проходит через одни и те же сообщения + задачи + transcript.

3. **Всё состояние хранится в файлах на диске.** Директория `.gotong/` содержит
   transcript, agents, sessions, secrets, vault. Скопировать директорию = перенести
   рабочее пространство. Перезапуск прозрачен.

## §2 Сфера применения

Этот устав распространяется на:

- Монорепозиторий по адресу [`github.com/Emir-Aksoy/Gotong`](https://github.com/Emir-Aksoy/Gotong)
- Все пакеты под `packages/`, примеры под `examples/`, шаблоны под `templates/`
- Любое производное программное обеспечение, которое называет себя «Gotong» без
  квалификатора (типа «based on Gotong» или «MyHub powered by Gotong» — это допустимо;
  «Gotong» наедине без квалификатора — нет)

## §3 Три уровня использования

| Уровень | Пользователь | Что получает |
|---|---|---|
| **Личный** | один человек | Запустить hub, подключить API-ключи, использовать готовые шаблоны, автоматизировать workflow без написания кода |
| **Команда** | небольшая группа (семья, команда) | Приглашать других, разделять роли, создавать workflow с человеческим подтверждением (HITL), совместно использовать подписку |
| **Федерация** | несколько организаций | Создавать peer-связи между hub'ами, запускать межорганизационные workflow, при этом учётные данные/данные остаются в своей организации |

Один двоичный файл охватывает все три уровня. Архитектура не разная — только конфигурация.

## §4 Северная звезда (что никогда не меняется)

Эти инварианты образуют «линию дизайна», упомянутую в `GOVERNANCE.md`:

1. **Hub — это тупой маршрутизатор.** Он не принимает решений за участников.
   Если PR или feature требуют, чтобы hub вызвал LLM и использовал ответ для выбора
   поведения (за пределами инструментального цикла самого агента), это пересекает черту.

2. **Люди — участники, а не команды «запросить ввод».** Протокол не разделяет
   людей и агентов на уровне wire. Если feature вынуждает hub обращаться с человеком
   иначе, чем с агентом на уровне маршрутизации задач, это пересекает черту.

3. **Состояние наблюдаемо и переносимо.** Каждое действие, которое hub совершает
   постоянно, должно быть видно в файлах на диске в `.gotong/`. Если feature
   вводит скрытое состояние только в памяти, которое невозможно перенести через
   резервное копирование, это пересекает черту.

## §5 Три столпа доверия

Почему кто-то доверяет Gotong настолько, чтобы направить его к своему дому,
семье или деньгам:

| Столп | Что это значит |
|---|---|
| **Управляемый** | Необратимые действия требуют явного подтверждения человеком (`human:` шаги, isходящие approval gates, флаги governance); ни один workflow не может обойти ворота |
| **Локальный** | Учётные данные зашифрованы в локальном vault; ключи не передаются в hub и не отправляются в облако как часть нашей инфраструктуры |
| **Проверяемый** | Каждая отправка и результат записываются в доступный для чтения transcript; нет скрытых суждений; код открытый |

## §6 Видение федерации

Долгосрочная форма Gotong — **свободный граф суверенных peer'ов**, а не иерархическое дерево с центральным узлом:

- Ни один hub не является «главным» по отношению к другому.
- Workflow могут пересекать границы организаций; учётные данные/данные/квоты остаются дома.
- Связи открываются и закрываются по взаимному согласию, с явными контрактами доверия.
- Управление федерацией является децентрализованным: нет привратника, который
  может «выгнать» hub из сети.

Этот образ («свободный граф», а не «звезда с нами в центре») является директивой
против «притяжения к платформе» — тенденции сетевых продуктов незаметно становиться
центрами, которым принадлежат связи.

## §7 Что Gotong не является

| Вещь | Почему не Gotong |
|---|---|
| SaaS-платформа или cloud-сервис, которым мы управляем | Мы публикуем программное обеспечение; вы запускаете его сами |
| Фреймворк для оркестрации агентов (LangGraph / CrewAI) | Он работает внутри процесса; мы — маршрутизатор между процессами |
| Потребительское приложение | Целевые пользователи: операторы / разработчики / power-users, которые готовы настраивать |
| Монопольный агент-агрегатор | Мы не стремимся быть точкой трения между пользователями и их агентами |

## Приложение A: Краткое описание архитектуры

> Предназначено для читателей-агентов, которым нужна быстрая ориентация.

```
┌─────────────────────────────────────────────────────────┐
│                    Gotong hub                          │
│                                                         │
│  Scheduler  ──dispatch──▶  Participant.onTask()         │
│      ▲                           │                      │
│      │                      (агент решает)              │
│      │                           │                      │
│  EventBus  ◀──result/event──────┘                      │
│      │                                                   │
│  TranscriptStore (файлы на диске)                       │
│  SessionStore    (файлы на диске)                       │
│  VaultStore      (зашифрованные файлы на диске)         │
└─────────────────────────────────────────────────────────┘
        │                          │
   HubLink (WS)              MCP Client/Server
   к другим hub'ам           к внешним инструментам
```

Важные факты:
- Hub **получает** задачи от участников и **маршрутизирует** их обратно к участникам.
- Hub **никогда** не вызывает LLM напрямую.
- Каждый вызов LLM происходит **внутри** реализации `Participant` (например, `LlmAgent`).

## Приложение B: Глоссарий

| Термин | Значение |
|---|---|
| **Hub** | Экземпляр Gotong; маршрутизирует сообщения и задачи, пишет transcript |
| **Participant** | Всё, что получает задачи: агент, человек, внешний сервис |
| **Space** | Рабочая директория, которой принадлежит hub (содержит `.gotong/`) |
| **HubLink** | WebSocket-соединение между двумя hub'ами для федерации |
| **Transcript** | Неизменяемый лог каждого события в задаче (файлы на диске) |
| **Vault** | Зашифрованное хранилище учётных данных на локальном диске |
| **Capability** | Строка, декларирующая, что умеет participant (`gotong.human/v1` и т.д.) |
| **Workflow** | Декларативная YAML-программа, описывающая последовательность шагов dispatch |
| **HITL** | Human-in-the-loop (человек в контуре): шаг, который требует подтверждения человека |
| **Federation** | Peer-to-peer соединение между двумя hub'ами с явными контрактами доверия |
