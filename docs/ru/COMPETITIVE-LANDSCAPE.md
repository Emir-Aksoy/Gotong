## 2. Позиционирование

> Другие — это либо «**фреймворк — это мозг**» (①), либо «**LLM встроен в холст, человек — узел одобрения**» (③), либо «**просто бэкенд-движок / просто мост сообщений**» (④). AipeHub — это «**тупой hub + человек как участник + файлы как состояние + суверенная федерация организаций**» — **субстрат для совместной работы**, а не ещё один оркестратор внутри процесса.

---

## 3. Ров (архитектурные преимущества)

1. **Тупой hub / решения у участников** — ни один из ① не является пассивным маршрутизатором; все запускают LLM внутри процесса и удерживают решение. Лишь дух «you own the loop» LlamaIndex Workflows приближается, но это всё равно внутрипроцессный движок событий. Не привязан ни к какому отдельному SDK поставщика — последовательная текучка Swarm→Agents SDK и AutoGen→MAF — именно то, что доказывает риск «runtime coupling».
2. **Человек и агент — один и тот же `Participant`** — каждый конкурент моделирует человека как особый случай: UserProxyAgent (AutoGen) / interrupt (LangGraph) / deferred-tool (Pydantic) / узел графа (ADK) / узел «Human Input» (Dify) / форма одобрения в Outlook (Copilot). **Ни один не делает человека и агента равными пирами на одной шине сообщений+задач+transcript.**
3. **Файлы как состояние, переносимое и проверяемое** — состояние конкурентов находится в памяти / SQLite / Postgres / Redis / Mongo / облаке поставщика. Ближайшие — лишь единственный файл SQLite (Flowise/Open WebUI), строки Postgres с возможностью запросов (DBOS) или определение YAML-графа (Rivet). **Никто не хранит transcript+agents+sessions+secrets+vault все как простые файлы, которые можно grep/diff/rsync/редактировать вручную.** «Скопировать директорию = переместить комнату» — сильнейший дифференциатор.
4. **Зашифрованное хранилище на организацию + квота API на организацию как первоклассные граждане** — Windmill (шифрование workspace-ключей) и Copilot (Key Vault) ближе всего, но никто не моделирует «изолированное хранилище учётных данных на организацию + квота LLM на организацию» как федерационно-осознанную границу. Слой протоколов (A2A/MCP) доходит только до «объявить схему аутентификации», ничего о хранении секретов или квотах.
5. **Межорганизационная федерация + учётные данные/данные/биллинг у каждого дома** — наиболее ясное белое пятно. ③ — всё однотенантные или единственно-поставщиковые SaaS, где команда/рабочее пространство только разделяют внутри одного развёртывания; движки ④ — просто бэкенды. **Никто не предлагает открытую P2P-федерацию, позволяющую workflow пересечь организационную границу, сохраняя при этом у каждой организации собственные учётные данные/данные/квоту.** И **«cross-hub HITL» (человек в организации B выполняет задачу, инициированную организацией A) вообще не охвачен A2A (стандарт 150+ организаций)** — A2A имеет только состояние задачи `input-required`, без модели человека-участника между организациями.

## 4. Слабые стороны (честный список)

1. **Широта интеграции/коннекторов** — крупнейший реальный ров на другой стороне: Zapier 8000+, Make 3000+, Lindy 4000+, n8n 1200+. У нас сейчас практически ноль.
2. **Полировка UX + NL-оркестрация** — Make's Reasoning Panel, «Gummie» NL→workflow от Gumloop, опыт HITL от Relay — всё это значительно более зрелое, чем YAML-first (даже с NL→YAML-помощником).
3. **Зрелость устойчивости** — Temporal (сигнал + бесконечные ожидания с нулевым потреблением ресурсов + воспроизведение событий) / DBOS (устойчивый sleep на недели) / Inngest / Restate на **годы впереди** по suspend/resume. Наш `SuspendTaskError`+SQLite sweep концептуально то же самое, но молодой, однонодовый, со слабыми гарантиями.
4. **Корпоративное управление** — истории SSO/аудит/соответствия Copilot (Entra ID+Key Vault+детальный RBAC), Windmill (5 ролей+ACL папок) и Lindy/Sema4 (SOC2/HIPAA) — то, чего мы не построили.
5. **UX многоагентной оркестрации** — Flowise Agentflow (supervisor/worker, разрешение конфликтов, динамические роли), Lindy Agent Swarms, вызов агент-к-агенту в Zapier — всё это готовые UI продуктов; у нас только примитивы dispatch.
6. **Широта IM-мостов не уникальна** — LangBot уже связывает больше платформ (+DingTalk/LINE/KOOK/WeChat Official Accounts) и не привязан к бэкенду. «6 мостов» — не ров в сырой широте — ров в «hub с файловым состоянием и моделью участников, где hub — просто маршрутизатор».
7. **Экосистема / mind share** — у другой стороны 50k–110k звёзд (CrewAI 52k, MetaGPT 68k, Dify 110k+); мы на раннем этапе.

## 5. Слой протоколов взаимодействия (наиболее активная цель согласования)

Во 2-м полугодии 2025 года протоколы взаимодействия были коллективно поглощены Linux Foundation и разделены на два слоя, AipeHub охватывает оба:

- **Слой инструментов (агент↔инструмент): MCP побеждает однозначно.** В декабре 2025 Anthropic передал его в **Agentic AI Foundation (AAIF)**, размещённую LF (совместно создана с OpenAI/Block), ~97M ежемесячных загрузок, ~10k серверов.
- **Слой агентов (агент↔агент межорганизационный): A2A побеждает однозначно.** Присоединился к LF в 2025-06; **поглотил IBM ACP** в 2025-08; на годовой отметке — **150+ организаций** в производстве.
- Остальные размещены выше и ниже: **AGNTCY/SLIM** = инфраструктурный/транспортный уровень; **NANDA** = исследовательское доверие идентичности (DID+AgentFacts); **Matrix** = наш философский кузен (федерация, суверенитет, состояние на вашем собственном сервере).

| Протокол | Слой | Управление | Межорганизационная идентичность | Транспорт/семантика | Принятие |
|---|---|---|---|---|---|
| **MCP** | вызовы инструментов | Anthropic→AAIF/LF | OAuth2.1+PKCE+RFC8707 (клиент↔сервер) | оба (JSON-RPC/stdio/Streamable HTTP) | доминирующий |
| **A2A** | агент↔агент | Google→LF | Agent Card объявляет OAuth2/OIDC/API-key/mTLS | оба (JSON-RPC/HTTPS+SSE) | 150+ организаций |
| ACP (IBM) | агент↔агент | →вошёл в A2A (2025-08) | (вошёл) | — | устарел |
| AGNTCY+SLIM | обнаружение+идентичность+**транспорт** | Cisco→LF | децентрализованный агентский сервис идентичности | SLIM=транспорт (gRPC/H2/H3), несёт A2A/MCP | 75+ компаний |
| NANDA | обнаружение+идентичность+экономика | MIT Media Lab | DID+верифицируемые учётные данные+AgentFacts | семантический (реестр) | исследования/не в продакшне |
| Matrix | федеративный транспорт сообщений | Matrix.org | homeserver-федеративный MXID | транспорт | 60M+ пользователей |

**Примитивы федерации AipeHub → сопоставление со стандартами:**

| Наш примитив | Согласованный стандарт | Вывод |
|---|---|---|
| `peerToken` | схема аутентификации A2A (Bearer/OAuth2/OIDC/mTLS) | **Согласовать** — переформулировать как объявленную схему A2A |
| `Task.origin` | метаданные задачи A2A / цепочка делегирования OIDC-A | **Впереди** — сохранить, отобразить на метаданные задачи A2A |
| входящий ACL | «непрозрачные агенты» A2A + избирательное раскрытие | сохранить, семантически согласовано |
| хранилище на организацию | (стандарт не охватывает) | **уникально, сохранить** |
| квота на организацию (OrgApiPool) | (нет стандарта; приближается к экономическому слою NANDA, в исследовании) | **уникально, сохранить** |
| реестр peer + репутация | реестр A2A / NANDA Index / ANS | долгосрочное согласование, следить за верифицируемым направлением NANDA |
| cross-hub HITL | **ни один протокол не охватывает** | **уникально + попадает в северную звезду** |

## 6. Направления улучшений (отсортированы по «рычагу / вкладу в северную звезду»)

**🔴 Высокий рычаг**
1. **Согласование с A2A (единственное наиболее ценное действие)** — открыть `/.well-known/agent-card.json`, переформулировать `peerToken` как объявленную A2A Bearer/OAuth2/mTLS схему, чтобы hub AipeHub мог федерироваться с 150+ организациями экосистемы A2A, а не только AipeHub↔AipeHub. Сквозное происхождение `Task.origin` фактически опережает текущую спецификацию A2A.
2. **Заполнить широту интеграции через экосистему MCP**, а не строить собственные коннекторы — MCP уже размещён в LF с ~10k серверами. Сделать «интеграционная возможность = установить MCP-сервер» первоклассным онбордингом, превращая «8000 коннекторов» другой стороны в «принятие открытого стандарта».
3. **Обновить примитивы dispatch в переиспользуемые шаблоны оркестрации** — создать supervisor/worker, дебаты, параллельный рой в `templates/`, соответствуя готовому опыту Flowise Agentflow / Lindy Swarms (architect-team уже закладывает основу).

**🟡 Средний рычаг**
4. **Устойчивость: честная калибровка + опциональный сильный бэкенд** — документировать правдивое сравнение наших гарантий с Temporal/DBOS; рассмотреть опциональный **режим на базе DBOS/Temporal** для поддержки suspend/resume (DBOS — это библиотека с состоянием в вашем собственном Postgres, лучше всего соответствующая этосу «состояние видно вам»).
5. **Полировка UX передачи HITL** — концептуально превосходит Slack/Rocket.Chat, но не хватает готовых запасных выходов: создать «передать человеку с полным контекстом / одобрение несколькими людьми / эскалация по тайм-ауту» как готовые шаблоны.
6. **Заполнение корпоративного управления** — SSO (OIDC/SAML), журналы аудита, детальный RBAC, чтобы пройти планку для организационных сценариев.

**🟢 Наблюдать / долгосрочно**
7. **Следить за слоем доверия идентичности** — NANDA (DID+AgentFacts) / ANS / цепочка делегирования OIDC-A — это верифицируемая будущая версия «реестра peer + репутации», ни одна из которых ещё не утверждена как стандарт, поэтому **не принимать сейчас**, следить за ней.
8. **Нарратив позиционирования** — внешне чётко обозначить «**нативный для edge-A2A/MCP, но несущий примитивы организационной границы, которые проводные протоколы намеренно игнорируют (хранилище / квота / cross-org HITL / происхождение origin)**».

**Итоговый вывод**: не конкурировать с Temporal/DBOS в устойчивости или с Dify/n8n в широте интеграции. Защитный клин — **та комбинация**: файл-первичная переносимость + человек как участник + несколько IM-нативных мостов + достаточно хороший suspend/resume, всё упакованное в один самохостируемый OSS-бинарный файл. Два наиболее стоящих заполнения: **согласование A2A** (для охвата экосистемы) + **интеграция через MCP-маршрут**.
