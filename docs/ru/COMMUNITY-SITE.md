## 1. Почему «нулевые вычисления»

Вся позиция дизайна Gotong — **хаб не запускает LLM сам / состояние — это всё файлы на диске / учётные данные остаются на вашей машине / федерация — это peer-to-peer**. Следуя этой позиции, **инфраструктуре сообщества тоже не нужен сервер**:

- **GitHub уже хранит суть** — шаблон — это файл, отправка — это PR.
- **Недостаёт только витрины** — а витрина для проекта, ориентированного на файлы, сама по себе является набором статических файлов.

Таким образом, эта витрина = один генератор + производимые им статические файлы. Генератор — [`packages/web/scripts/build-site.mjs`](../packages/web/scripts/build-site.mjs), создающий `site/` (корень репозитория, в gitignore):

- `index.html` — автономный единый файл (без фреймворка, без runtime, встроенный CSS): герой нарратива доверия + сетка карточек галереи шаблонов + таблица лидеров цитирования.
- `templates.json` — машиночитаемый фид `gotong.site/v1` (витрина — это тоже данные, file-first).

Загрузите `site/` на любой бесплатный уровень GitHub Pages / Cloudflare Pages / Netlify, и витрина будет доступна по цене **$0**. Сервер Tencent Cloud 2c2G продолжает простаивать как резерв.

## 2. Как построить

```bash
pnpm build:site          # корневой скрипт, делегирует в packages/web
# или
pnpm -C packages/web build:site
```

Вывод:

```
build-site: 11 templates → site/ (index.html + templates.json), 2 on the leaderboard
```

`site/` — это производный артефакт, который **строится по требованию и не фиксируется в репозитории** (та же позиция, что и у `dist-portable/`, см. `.gitignore`). Единственный источник истины остаётся в `examples/` и `templates/community/` (разделение шаблонов/фреймворка); витрина — это их проекция только для чтения — измените шаблон и перезапустите генератор.

**Детерминизм**: генератор не записывает временны́е метки и сортирует стабильно → одни и те же входные данные производят **побайтово идентичный** `site/`, так что пересборки не создают бессмысленных диффов.

---

## 3. Корпус = тот же набор, который проходит валидацию

Генератор сканирует **ровно** те же два корня, которые проверяет шлюз валидации уровня репозитория (`pnpm check:templates` / [`tests/all-templates-parse.test.ts`](../packages/web/tests/all-templates-parse.test.ts)):

| Происхождение | Путь | Примечание |
|---|---|---|
| `flagship` | `examples/*/template/*.template.ya?ml` | флагманские шаблоны, поставляемые с фреймворком |
| `community` | `templates/community/templates/**/*.ya?ml` | куда попадают отправки от сообщества |

Таким образом, «каждый шаблон, прошедший CI, появляется на витрине» выполняется **по построению** — манифест, который не парсится, никогда не попадёт на карточку (он не проходит `check:templates` и никогда не попадает внутрь).

## 4. Таблица лидеров цитирования = входящая степень `provenance.derivedFrom`

Таблица лидеров читает аддитивное поле происхождения `template.provenance.derivedFrom` (пункт 6 контрольного списка перед запуском):

- Одна запись `derivedFrom` — это одно **рёбро цитирования**: оно объявляет «этот шаблон адаптирован от кого».
- Рейтинг = **входящая степень** = «сколько шаблонов произведено от меня».
- Ребро ссылается на **slug** целевого шаблона (его публичный дескриптор, см. ниже), поэтому при форке шаблона написание **slug апстрима** в вашем `provenance.derivedFrom` завершает цепочку атрибуции.

Два реальных рёбра цитирования, поставляемых с фреймворком:

```yaml
# examples/codex-deepseek-hub/template/codex-deepseek-hub.template.yaml
provenance:
  derivedFrom: [personal-coding-hub]   # дочерний пример, тот же скелет диспетчеризации

# examples/tea-chain-hq/template/chain-hq.template.yaml
provenance:
  derivedFrom: [tea-supply-link]       # MIRROR, обратная межорганизационная оркестрация
```

→ В таблице лидеров `personal-coding-hub` и `tea-supply-link` получают по 1 голосу.

**Опечатки не поглощаются молча**: когда `derivedFrom` указывает на несуществующий slug, генератор выводит `WARNING … no template with that slug` в stderr (`buildModel` собирает это в `unresolved`), никогда не пропуская молча как 0 голосов.

## 5. Схема slug (публичный дескриптор)

Slug — это **стабильная публичная идентичность** шаблона — галерея (`builtin-templates.ts`), `FLAGSHIP-TEMPLATES.md` и эта витрина используют один и тот же дескриптор, чтобы `derivedFrom` форка мог ссылаться на апстрим по «имени, которое все знают». Правила `assignSlugs`:

| Источник | slug |
|---|---|
| flagship, с **ровно одним** файлом шаблона в `examples/<dir>` | базовое имя `<dir>` (например, `examples/tea-supply-link` содержит `tea-shop.template.yaml` → slug `tea-supply-link`, **а не** имя файла) |
| flagship, с **несколькими** файлами шаблонов в одном dir | разграничение по стему имени файла (например, `examples/family-learning-hub` содержит `family-tutor` + `child-desk`) |
| community | стем имени файла |

**Конфликт — это сбой сборки**: два шаблона, вычисляющих один и тот же slug → `assignSlugs` выбрасывает исключение. Неоднозначный публичный дескриптор должен давать громкую ошибку во время сборки, а не тихо перезаписанную карточку / ребро, указывающее на неправильный шаблон. (Эта защита уникальности — реальная яма, в которую попали: `family-tutor` и `child-desk` находятся в одном dir и раньше оба брали имя dir `family-learning-hub` и сталкивались.)

## 6. Развёртывание (бесплатный статический хостинг)

`site/` — это чистый статический артефакт; подойдёт любой бесплатный уровень. В качестве примера возьмём **GitHub Pages** (квота Actions не нужна — собирайте локально, вручную отправляйте в ветку `gh-pages` или используйте Pages-конвенцию `/docs`):

```bash
pnpm build:site
# затем опубликуйте содержимое site/ на выбранном вами статическом хостинге:
#   · Cloudflare Pages / Netlify: перетащите site/ внутрь или настройте хук
#     "build: pnpm build:site, output: site" (у них есть своя квота сборки,
#     не связанная с квотой Actions этого репозитория);
#   · GitHub Pages: собирайте локально, затем отправьте site/ в ветку gh-pages.
```

> ⚠️ **Квота GitHub Actions этого репозитория исчерпана**, поэтому сборка витрины **не** зависит от CI этого репозитория. Генератор запускается локально (бесплатно); квота сборки статического хостинга — отдельный вопрос. `site/` не фиксируется в репозитории, поэтому не увеличивает его размер.

## 7. Тест защиты от деградации

[`tests/build-site.test.ts`](../packages/web/tests/build-site.test.ts) фиксирует чистую логику генератора (его IO-оболочка защищена, поэтому `import` не вызывает сканирование файлов и не записывает файлы):

- `assignSlugs` — три правила slug + защита уникальности (забор регрессии для той реальной ямы);
- `extractTemplate` — читает поверхность отображения + `provenance.derivedFrom` (фильтруя пустые записи) из необработанного манифеста, громко выбрасывает исключение при плохой схеме;
- `buildModel` — подсчёт входящей степени цитирования + сортировка таблицы лидеров + вынос опечатки в `unresolved`;
- `escapeHtml` / `render*` — имена/описания, предоставленные сообществом, **не являются доверенными**; XSS-случаи фиксируют, что `<script>` никогда не может вырваться из разметки.

## 8. Ограничения (честно)

- Витрина — **не** редактор шаблонов и ничего не устанавливает — это окно отображения только для чтения. Установка происходит через кнопку «Одним кликом» галереи шаблонов в консоли администратора / `POST /api/admin/templates/import` (см. [`TEMPLATE-GALLERY.md`](zh/TEMPLATE-GALLERY.md)).
- **Разделение шаблонов/фреймворка** не нарушается: витрина читает только **структуру + ссылки** манифеста, никогда не отображая и не перенося контент знаний или персонал (решения #4/#5).
- `site/` — это снимок во время сборки: после изменения `examples/*/template/` или добавления шаблона сообщества необходимо **перезапустить** `pnpm build:site`; тест защиты от деградации является сторожем.
