# AipeHub 개요 · 5분 안내 지도

<!-- doc-version: 1.0 -->
> **문서 버전 1.0** · 한국어 번역 · 최종 업데이트 2026-06-27 · 권위 있는 원본: [English](../OVERVIEW.md). 번역본이 영어 버전과 충돌하는 경우 영어 버전이 우선합니다.

> 이것은 프로젝트의 **단일 페이지 지도**입니다. 이 문서를 마치면 AipeHub가 무엇인지,
> 무엇이 어디에 속하는지, 참여자가 어떻게 연결하는지, 템플릿이 어디서 오는지, 몇
> 사람이 함께 작업하는 방식, 그리고 조직이 키를 포기하지 않고 어떻게 연합하는지
> 알 수 있습니다. 각 섹션의 끝에는 더 깊이 읽고 싶을 때를 위한 → 링크가 있습니다.

---

## 한 문장으로

**AipeHub**는 **TypeScript와 Python을 위한 자체 호스팅 협업 워크스페이스**입니다:
사람과 AI 에이전트가 하나의 "방"을 공유하며, 의도적으로 단순하게 설계된 Hub가
태스크를 분배하고, 결과를 수집하고, 전체 실행을 기록합니다.

이것은 **에이전트 프레임워크가 아닙니다** (LLM을 실행하지 않습니다) — **다중 참여자
협업을 위한 기반**이며, 조직이 **키, 데이터, 또는 청구를 넘기지 않고** 연합할 수 있습니다.

---

## 그것이 무엇이며 — 무엇의 *아래*에 있는가

대부분의 "에이전트" 프로젝트는 에이전트이거나, 하나의 에이전트 루프를 작성하기 위한
프레임워크입니다 (LangGraph, CrewAI, AutoGen). AipeHub는 **어느 쪽도 아닙니다** —
그것들이 *플러그인하는* 레이어입니다. LangGraph 그래프, CrewAI 크루, CLI 코딩 에이전트
(Claude Code, Codex), 외부 A2A 에이전트, 그리고 사람이 모두 같은 `Participant`로서
같은 방에 합류합니다. Hub는 메시지를 라우팅하고, 태스크를 분배하고, 트랜스크립트를
기록하고, 경계를 강제합니다 — **LLM을 절대 실행하지 않으므로**, 모든 결정은 참여자에게
남습니다.

메시지 버스 이상을 만드는 세 가지:

- **동등한 참여자** — 사람은 에이전트와 똑같이 `Participant`입니다. '인간 입력 요청
  도구'가 없습니다; 사람과 에이전트는 같은 태스크 + 트랜스크립트, 그리고 같은 비동기
  / 장기 실행 프리미티브를 통해 협업합니다.
- **거버넌스** — 민감하고 조직 간 작업은 그냥 실행되지 않습니다. 완전한 감사 추적과
  함께 인간이 수신함에서 승인하도록 요구할 수 있습니다 (제안 → 검토 → 확인).
- **주권** — 모든 워크스페이스는 귀하가 소유한 디스크의 디렉토리입니다. 두 조직이
  연합할 때, 자격 증명, 데이터, 청구는 각자 집에 머뭅니다; 링크를 건너는 것은
  **링크별 신뢰 계약**에 의해 제한됩니다.

그 조합 — 하나의 영리한 프로토콜이 아닌 — 이 AipeHub입니다. 사람-에이전트 동등성,
거버넌스된 조직 간 연합, 자체 호스팅 주권을 하나의 실행 가능한 파일 우선 패키지에
담은 첫 번째 기반입니다.

어댑터**도 들어갑니다 — 모두 스케줄러에게 투명합니다.

## 네 가지 엣지 — AipeHub가 세계와 연결하는 방식

AipeHub는 네 가지 엣지를 통해 나머지 에코시스템에 도달합니다. **표준 프로토콜이 있는
곳에서는 그것을 사용합니다** — 재발명하지 않습니다:

| 엣지 | 프로토콜 | 방향 | 무엇을 전달하는가 |
|---|---|---|---|
| 도구 & 데이터 | **MCP** | 양방향 | 에이전트가 외부 MCP 도구를 호출하고; 외부 클라이언트 (Claude Desktop, Cursor)가 Hub를 구동합니다. |
| 에이전트 ↔ 에이전트 | **A2A** | 양방향 | 인바운드 `message/send`가 dispatch가 되고; 아웃바운드 호출이 원격 A2A 에이전트를 구동합니다. |
| 코딩 에이전트 | **ACP** | 아웃바운드 | Hub가 Claude Code / Codex를 스폰하고 세션을 유지하며 턴마다 구동합니다. |
| Hub ↔ Hub | **HubLink** | 양방향 | 두 Hub 간의 AipeHub 자체 연합 링크 — 링크별 신뢰 계약, 조직 간 태스크 전달, 승인 게이트가 있는 곳입니다. |

처음 세 개는 AipeHub가 구현하는 에코시스템 표준입니다. HubLink는 AipeHub가 소유하는
한 가지 조각입니다 — 영리한 와이어 형식으로서가 아니라 (내부는 WebSocket + bearer token
+ JSON-RPC), **두 거버넌스된 Hub가 교환하는 것에 대한 계약**으로서: 기능 매니페스트,
조상 보존 태스크 전달, 그리고 아래의 링크별 신뢰 계약.

→ 더 깊이: [`MCP.md`](../MCP.md) · [`FEDERATION.md`](../FEDERATION.md) · [`PROTOCOL.md`](../PROTOCOL.md)

## 참여자가 연결하는 방법

헤드라인 경로는 **LLM 에이전트를 추가하는 두 가지 방법**입니다:

| 경로 A · 호스트 관리 | 경로 B · 외부 SDK |
|---|---|
| 관리자 UI에서 양식 작성 / YAML 가져오기 / 템플릿 붙여넣기 → 호스트가 자체 프로세스 내에서 `LlmAgent`를 스폰 | Node / Python으로 `AgentParticipant.handleTask`를 구현하는 코드 작성, 그런 다음 Hub의 WebSocket 포트에 `connect(url, agents)` |
| **코드 0줄** | 코드를 작성합니다 |
| LLM 에이전트만 (래핑된 Anthropic / OpenAI / Mock) | **모든 종류** (LLM, 스크래퍼, 로컬 도구, 비공개 로직, Python ML 모델) |
| 제공자 키는 `secrets.enc.json`에 디스크에 암호화됨 (에이전트별 또는 워크스페이스 기본값), 또는 환경에서 읽음 | API 키를 직접 관리; 에이전트는 본인 기기에서 실행 |
| 호스트 재시작 시 자동 재스폰 | 생명주기를 직접 관리; SDK에 내장 자동 재연결 있음 |
| 최적: 일반 사용자 / 표준 LLM 역할 / 60초 내 라이브 | 최적: 개발자 / 비공개 데이터 / 코드 미노출 |

→ 경로 A: [`HUMAN.md §1 에이전트`](../HUMAN.md#1-智能体v21) + [`TEMPLATES.md`](../TEMPLATES.md)
→ 경로 B: [`AGENT.md`](../AGENT.md)

…그리고 모든 것이 같은 `Participant`이므로, 같은 방은 다음도 받습니다:

- **CLI / ACP 코딩 에이전트** — Hub가 유지된 ACP 세션을 통해 Claude Code / Codex를
  구동하며 (실제 기기 검증됨), 위험 작업 게이트가 파괴적인 명령을 인간 승인을 위해
  파킹할 수 있습니다.
- **외부 A2A 에이전트** — 기능 아래에 원격 에이전트를 등록; 워크플로우 단계가 다른
  모든 것처럼 라우팅합니다.
- **프레임워크 어댑터** — Python SDK를 통해 LangGraph 그래프나 CrewAI 크루를
  `Participant`로 래핑; 프레임워크 자체는 Hub에 의해 임포트되지 않습니다.

모두 **자유롭게 혼합** — 하나의 방이 호스트 관리 `writer-zh`, 귀하 자신의 SDK 연결
`rag-agent`, 그리고 Codex 코딩 세션을 보유할 수 있으며, 스케줄러에게 완전히 투명합니다.

## 템플릿이 어디서 오는가

```
                  templates/
                  ├── agents/           원본 공식 템플릿
                  ├── teams/            원본 공식 팀
                  └── community/        제3자에서 적응됨 (CC0 + MIT)
```

세 가지 방법으로 얻을 수 있습니다, 취향대로 선택:

1. **템플릿 갤러리, 한 클릭** — 관리자 UI에는 미리 만들어진 Hub 갤러리가 있습니다
   (개인 / 조직 / 조직 간); 하나 선택 → 설치 → 에이전트 + 워크플로우 + KB 슬롯이
   귀하의 Space에 배치됩니다.
2. **복사-붙여넣기** — GitHub에서 `.yaml`의 **Raw**를 클릭 → 복사 → 관리자 UI
   "에이전트 → 가져오기", 붙여넣기.
3. **파일 다운로드** — `.yaml`을 로컬에 저장 → 관리자 UI "파일 업로드".

모든 파일에는 `# Source` / `# Upstream` / `# License` / `# Adapted` 헤더 주석이
있으므로, **업스트림 출처가 절대 손실되지 않습니다**. 제3자 라이선스 전문은
[`../templates/community/LICENSE-NOTICES.md`](../templates/community/LICENSE-NOTICES.md)에 있습니다.

> **템플릿과 프레임워크는 설계상 분리됩니다.** 템플릿은 에이전트, 워크플로우, KB 슬롯
> 등 *구조와 참조*를 전달합니다 — 지식 *콘텐츠* 자체는 절대 포함하지 않으며, 귀하의
> 사람이나 비밀도 절대 포함하지 않습니다. 하나를 설치하면 연결이 구성됩니다; 다른
> 조직의 데이터를 복원하지 않습니다.

→ 전체 흐름: [`TEMPLATES.md`](../TEMPLATES.md)
→ 설치할 수 있는 미리 만들어진 Hub: [`zh/HANDS-ON-HUBS.md`](../zh/HANDS-ON-HUBS.md) (zh)

## 하나의 방에 몇 명의 사람들

AipeHub는 "팀"을 **하나의 방** = 하나의 `.aipehub/` 디렉토리로 모델링합니다. 세 가지
역할 티어:

| 역할 | URL | 이 방에서 할 수 있는 것 |
|---|---|---|
| **admin** | `/admin` | 방 구성, 에이전트 요청 승인/거부, 태스크 분배, 작업 평가, 다른 admin 초대 |
| **worker** | `/` (`/me` 워크벤치) | 닉네임 + 할 수 있는 작업 선택, 멤버 전용 워크플로우 실행, 수신함 처리, 태스크 완료 또는 거부 |
| **agent** | WS 포트 | 분배된 태스크를 자동으로 받고 결과 반환 |

### 일반적인 소규모 팀 워크플로우 (스크립트)

```
0  Alice (admin)가 Hub를 시작합니다 → 시작 시 브라우저에 일회용
   admin URL이 표시됩니다; 그녀는 1Password에 저장합니다.
1  Alice가 관리자 UI에서 제공자 키를 구성합니다 → 워크스페이스 기본
   키가 디스크에 암호화됩니다.
2  Alice가 템플릿을 설치합니다 (또는 storyteller.yaml을 가져옵니다) → 호스트가
   즉시 LLM 에이전트를 스폰하고, 온라인으로 표시됩니다.
3  Alice가 Bob과 Carol에게 초대 URL을 보냅니다. 그들은 닉네임을 선택하고,
   할 수 있는 기능(초안 / 검토)을 확인합니다 → 방에 입장합니다.
4  Alice가 태스크를 분배합니다: "인내에 관한 어린이 이야기 작성",
   전략 = capability:[story] → 호스트 관리 스토리텔러가 태스크를 수락합니다
   → 30초 후 600단어 이야기가 돌아옵니다.
5  워크플로우 단계에 서명이 필요합니다 → Bob의 수신함에 파킹됩니다; Bob이
   /me 워크벤치에서 승인하면 실행이 재개됩니다 — 도구 호출이 아닌 루프에 있는
   사람입니다.
6  Alice가 작업을 평가합니다; 기여 리더보드가 새로 고침됩니다; 모든 이벤트가
   transcript.jsonl에 있으므로, 충돌 + 재시작 시 완전히 복구됩니다.
```

**핵심 개념** (세부 사항은 HUMAN.md에):

- **세 가지 분배 전략**: `direct` (이름으로), `capability` (기술로), `broadcast` (먼저 수락하면 됩니다)
- **Human-in-the-loop**: 워크플로우 단계가 사람의 수신함에 분배하고 계속하기 전에 승인 / 선택 / 편집을 기다릴 수 있습니다
- **`/me` 워크벤치**: 멤버가 자신의 멤버 전용 워크플로우를 실행하고, 최근 실행을 확인하고, 자신의 에이전트 (BYO 키)를 관리하며, 모두 자신으로 범위 지정됩니다
- **API 키, 세 가지 티어**: 에이전트별 비공개 → 워크스페이스 기본값 → 환경 변수

→ 전체 설명: [`HUMAN.md`](../HUMAN.md)

## 조직 간 — 거버넌스된 연합

**두 가지 다른 "다중 팀" 의미 — 혼동하지 마세요:**

### 하나의 방, 많은 역할 (= 위 섹션)

모든 사람이 같은 `.aipehub/` 디렉토리, 같은 Hub 프로세스에 있습니다. 이것이 기본값입니다.

### 많은 방, 연합됨 (= 진정한 조직 간)

각 조직은 자체 독립적인 Hub를 실행합니다 (자체 `.aipehub/`, 자체 사람과 에이전트,
**자체 API 키와 자체 청구**). 두 Hub가 **HubLink**를 통해 연결되며, 하나가 다른
하나에 요청할 수 있는 것은 **링크별 신뢰 계약**에 의해 고정됩니다:

- **기능 허용 목록** — 피어가 호출할 수 있는 정확한 기능
- **데이터 클래스 게이트** — 링크를 건널 수 있는 데이터 클래스 (실패 폐쇄)
- **할당량** — 재연결에 걸쳐 유지되는 링크당 속도 / 예산 상한
- **해지** — 언제든지 링크 차단
- **지식 베이스 허용 목록** — 피어가 접근할 수 있는 공유 KB

가장 단순한 패턴은 `TeamBridgeAgent`: 전체 서브 허브가 업스트림에 **단일 에이전트**로
나타나며, 내부 멤버 / 키 / 서브 태스크는 부모에게 보이지 않습니다.

```
   Company Hub (Bob is admin)
       │
       ├── agent · alice-team   ←─┐
       │                          │  TeamBridgeAgent  (over HubLink)
       │                  ┌───────┴────────┐
       │                  │ Alice's Hub    │ (Alice is admin)
       │                  │  · writer-bot  │   keys / people / billing
       │                  │  · reviewer-bot│   all stay on Alice's hub
       │                  └────────────────┘
       └── agent · david-team   ←── another team, same idea
```

브리징을 넘어서, **한 Hub의 워크플로우가 다른 Hub의 기능에서 단계를 취할 수
있습니다**. 그 피어가 승인을 요구하면, 누군가 승인할 때까지 단계는 사람의 수신함에
파킹됩니다 — 조직 간 호출은 거버넌스되고, 두 단계이며, 완전히 감사 가능하며, 워크플로우
YAML은 피어를 이름으로 지정하지도 않습니다 (기능만 이름 지정; 링크는 런타임 구성입니다).

**왜 중요한가 — 주권이 온전히 유지됩니다:**

- 업스트림은 *집계된 결과*를 봅니다 ("alice-team이 N개 태스크 완료"), 피어의 키나 원시
  데이터는 절대 보지 않습니다
- 각 Hub는 **자체 자격 증명 볼트**와 **자체 사용량 / 비용 원장**을 유지합니다 — 청구는
  허브별입니다
- 개인 내부 PoC가 필요하신가요? 로컬 Hub를 실행하세요 — 온보딩 비용 없음
- 전체 회사가 협업하길 원하신가요? 위에 거버넌스된 링크를 걸어두세요 — **기존 팀
  구조를 건드리지 않고**
