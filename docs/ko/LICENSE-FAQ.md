## 1. 비공개 소스 제품 / SaaS / 내부 도구에 AipeHub를 임베드할 수 있나요?

**네.** MIT는 가장 관대한 OSS 라이선스 중 하나입니다. 다음을 허용합니다:

- ✅ 상업적 사용, AipeHub 전체를 재패키징하여 판매하는 것 포함
- ✅ 소스 수정, 이름 변경 (단, 이름을 바꾸는 경우 "AipeHub 기반"이라고 명시해 주세요)
- ✅ 비공개 소스 파생물 — 변경 사항을 **오픈 소스화할 필요 없습니다**
- ✅ npm 의존성으로 `@aipehub/core`를 비공개 소스 SaaS에 포함

**유일한 강제 요구 사항**: LICENSE 파일 + 저작권 고지를 유지하세요 (제품의
NOTICE / 제3자 라이선스 페이지에 AipeHub를 나열하면 충분합니다).

## 2. 소스를 수정했는데 변경 사항을 반드시 기여해야 하나요?

**아니오.** MIT는 카피레프트가 아닙니다. 다음을 할 수 있습니다:

- 수정 사항을 비공개로 유지
- 상업 제품의 일부로 배포
- 업스트림에 PR을 보내지 않아도 됩니다 — 완전히 괜찮습니다

그렇다고 해도, PR을 환영합니다 — 프로젝트가 나아질수록 다음 업그레이드가
더 저렴해집니다. 절차는 [`CONTRIBUTING.md`](../../CONTRIBUTING.md)를 참조하세요.

## 3. `templates/community/`의 서드 파티 프롬프트 템플릿을 상업적으로 사용할 때 무엇을 주의해야 하나요?

`templates/community/`는 두 업스트림 소스를 수집합니다:

| 소스 | 라이선스 | 상업적 사용 | 비고 |
|---|---|---|---|
| [`awesome-chatgpt-prompts`](https://github.com/f/awesome-chatgpt-prompts) | **CC0 1.0** (퍼블릭 도메인) | ✅ 모든 사용 | 법적으로 귀속 **불필요**; 존중의 표시로 출처 줄을 유지합니다 |
| [`awesome-chatgpt-prompts-zh`](https://github.com/PlexPt/awesome-chatgpt-prompts-zh) | **MIT** | ✅ 모든 사용 | 저작권 + 라이선스 고지를 **반드시 유지**해야 합니다 |

고지는 어떻게 유지되나요? `templates/community/`에는 이미 세 겹으로 포함되어 있습니다:

1. 모든 yaml 파일의 **4줄 헤더 주석**: `# Source` /
   `# Upstream` / `# License` / `# Adapted`
2. 집계 파일
   [`templates/community/LICENSE-NOTICES.md`](../../templates/community/LICENSE-NOTICES.md)에는
   전체 MIT 텍스트 + CC0 요약 + 업스트림 저장소 URL이 있습니다
3. 디렉토리의 [`README.md`](../../templates/community/README.md)에서
   적응 규칙과 라이선스 매트릭스를 설명합니다

**이 세 겹이 온전히 유지된 상태로** `templates/community/`를 재배포하는 한
(git 포크 / 클라우드 원본 URL / 내부 CDN — 모두 괜찮습니다), 완전히 준수한 것입니다.

> "템플릿의 내용을 admin UI에 붙여넣었고 `secrets.enc.json` /
> `agents.json`에 저장되었는데 — 그게 배포인가요?" —
> **아닙니다.** 자신의 배포 내에서만 사용하는 것이지, 제3자에게 전달하는 것이
> 아닙니다. 귀속 조치가 필요하지 않습니다.

## 4. LICENSE를 변경하고 "우리의" 제품으로 재배포할 수 있나요?

**제품 이름을 변경하고 자체 라이선스 줄을 추가할 수 있지만**, 원본 MIT 텍스트를
**삭제할 수는 없습니다**:

- ✅ 파생물을 `BobHub`라고 명명하고, Apache-2.0 / 독점 / 직접 작성한 라이선스를
  적용할 수 있습니다
- ✅ 자체 LICENSE 파일에 자체 저작권을 넣을 수 있습니다
- ⚠️ 하지만 **어딘가에** (예: NOTICE.md 또는 THIRD-PARTY.md) AipeHub의 원본 MIT
  텍스트 + 업스트림 저작권 줄을 **반드시 유지**해야 합니다
- ❌ "AipeHub는 우리의 원작"이라고 주장할 수 **없습니다** — 이는 라이선스와
  관계없이 사기입니다

## 5. 동료가 GPT로 작성한 비공개 프롬프트를 에이전트로 가져왔는데 라이선스 위험이 있나요?

**AipeHub 측에서는 없습니다.** 귀사 또는 귀사의 직원이 작성한 프롬프트는
귀사의 자산입니다; AipeHub는 단순히 런타임 컨테이너입니다. 하지만 다음을
확인하세요:

- 동료의 GPT 출력이 OpenAI의 서비스 약관을 준수하는지 여부 (모델 출력의
  "소유권"에 대한 OpenAI의 정책은 시간이 지남에 따라 달라졌습니다 — 법무팀에
  문의하세요)
- 프롬프트가 다른 사람의 코드 / 기사 발췌를 **인용**하는 경우, 해당 인용의
  라이선스가 허용하는지 여부

이 두 가지는 AipeHub 프로젝트가 관리하는 범위가 아닙니다 — MIT는 소프트웨어
자체에 라이선스를 부여하는 것이지, 그것으로 생성하는 콘텐츠가 아닙니다.

## 6. 고객의 인트라넷에 AipeHub를 배포하는데 어떤 라이선스 파일을 전달해야 하나요?

최소한:

- AipeHub 저장소 루트의 `LICENSE` 파일
- `templates/community/`를 사용하는 경우: `LICENSE-NOTICES.md`도 함께
- `@aipehub/core` npm 패키지를 임베드하는 경우: 패키지는 설치 시 자체 라이선스를
  포함합니다; 다운스트림 재배포는 `node_modules/@aipehub/*/LICENSE`를
  삭제하지 않으면 됩니다

일반적인 방식은 제품의 "제3자 라이선스" 페이지에 모든 업스트림 OSS 라이선스
텍스트를 나열하는 것입니다. 거기에 AipeHub의 MIT를 추가하면 완료입니다.

## 7. AipeHub의 런타임 의존성에 GPL/AGPL 스타일의 카피레프트가 포함되어 있나요?

현재는 아닙니다. 주요 의존성:

| 의존성 | 라이선스 |
|---|---|
| `ws` (WebSocket) | MIT |
| `yaml` | ISC |
| `better-sqlite3` (선택 사항) | MIT |
| `@anthropic-ai/sdk` (선택적 피어 의존성) | MIT |
| `openai` (선택적 피어 의존성) | Apache-2.0 |
| `vitest` (개발 전용) | MIT |
| `tsx` (개발 전용) | MIT |

모두 허용적 라이선스입니다. GPL/AGPL 의존성이 제안될 경우 먼저 이슈를 열겠습니다;
다운스트림 유연성을 위해 카피레프트 의존성을 **피하는** 것이 우리의 방침입니다.

## 8. AipeHub의 와이어 프로토콜이 라이선스에 포함되나요?

아닙니다. `docs/PROTOCOL.md`에 설명된 JSON 프레임 형식은 **사실상의 사양**입니다 —
누구든 **어떤 허가 없이도** 자체 허브 서버 또는 SDK를 구현할 수 있습니다. 언어
생태계 포트(Go / Rust / 브라우저 SDK 등)를 장려합니다; 각각 자체 라이선스를 선택합니다.

## 9. 취약점을 어떻게 신고하나요?

프로젝트 저장소의 **GitHub 보안 권고** (비공개 제출)를 통해 — 유일한 보안
채널입니다; 의도적으로 보안 이메일이 없습니다 ([`SECURITY.md`](../../SECURITY.md) 참조).
공개 이슈에 취약점 세부 내용을 게시하는 것은 **적절하지 않습니다** — 라이선스는
허용할지라도.

## 10. 회사 내에서 AipeHub를 포크하면서 포크를 오픈 소스화하지 않아도 되나요?

**물론입니다.** MIT는 전파되지 않습니다. 다음을 할 수 있습니다:

- 비공개 Git으로 포크 → 자유롭게 수정 → 인트라넷에 배포
- 포크 이름을 변경하고 고객을 위해 비공개로 배포
- 포크의 빌드 아티팩트를 비공개 소스 바이너리로 판매

**최종 결과물 어딘가에 AipeHub의 원본 MIT 라이선스를 유지하는 한** (일반적으로
"오픈 소스 고지" 페이지), 완료입니다.
