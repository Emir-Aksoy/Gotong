# 보안 정책

<!-- doc-version: 1.0 -->
> **문서 버전 1.0** · 한국어 번역 · 최종 업데이트 2026-06-27 · 권위 있는 원본: [English](../../SECURITY.md). 번역본이 영어 버전과 충돌하는 경우 영어 버전이 우선합니다.

## 취약점 신고 방법

**보안 이슈에 대해 공개 GitHub 이슈, Discussion, 또는 PR을 열지 마세요.**
비공개 채널을 사용하세요:

### 권장 — GitHub 비공개 취약점 신고

다음 주소에서 비공개 보안 권고를 열어주세요:

> **<https://github.com/Emir-Aksoy/AipeHub/security/advisories/new>**

GitHub의 내장 양식은 다음을 제공합니다:

- 메인테이너와의 종단 간 비공개 스레드 (이메일 유출 없음)
- 첨부 파일 + 재현 단계가 한 곳에
- 신고 → 수정 → 공개 CVE 할당까지 추적된 타임라인

이것이 우리가 먼저 읽고 가장 빠르게 답변하는 채널입니다. 무료 GitHub 계정이
필요합니다; 그것이 유일한 사전 요구 사항입니다.

### 이메일 채널 없음 (1.0 이전)

v0.x 기간 동안에는 의도적으로 **보안 이메일이 없습니다**. 이 저장소의 이전
개정판에는 `security@aipehub.dev`가 *희망적인* 주소로 나타납니다 — 도메인이
등록되어 있지 않고 메일함이 활성화되지 않았으므로, 그쪽으로 보내는 메일은
어디에도 도달하지 않습니다. 실제 신고를 신뢰할 수 있는 누군가가 믿을 수 있는
죽은 연락처를 매달지 않도록 광고를 중단했습니다.

GitHub 비공개 취약점 신고(위)가 오늘의 **유일한** 채널입니다: 무료이고, 비공개이며,
먼저 읽는 채널입니다. 실제 메일함을 설치할 가치가 있는지는
[릴리즈 체크리스트](../../.github/RELEASE-CHECKLIST.md#security-contact) 결정으로
1.0 준비까지 연기되었습니다; 그때까지는 보안 권고 양식을 사용해 주세요.

GitHub를 정말로 사용할 수 없으면, **보안 상세 내용 없이** 메인테이너에게 연락해
달라는 GitHub Discussion(일반, 보안 내용 없음)을 열어 주세요 — 그 한 건의 신고를
위해 비공개 채널을 마련하겠습니다.

보안 권고에 포함할 내용:

- 이슈 설명
- 정확한 재현 단계
- 테스트한 커밋 해시 (`git rev-parse HEAD`)
- (선택 사항) 제안된 수정 또는 패치
- (선택 사항) 권고에서 크레딧을 원하는 이름 / 핸들

### PGP는 어떤가요?

오늘 PGP 키를 **공개하지 않습니다**. 이유:

- GitHub의 비공개 권고 채널은 이미 귀하와 메인테이너 알림 사이에서 TLS 암호화가
  종단 간으로 이루어지므로, PGP는 거의 추가하지 않습니다.
- 초기 프로젝트를 위한 PGP 키 유지 관리는 혜택보다 더 많은 실패 모드(잃어버린
  키, 만료된 키, 서명 의식)입니다.

귀하의 조직 정책이 PGP 암호화 공개를 요구한다면, GitHub 채널을 통해 먼저
연락해 주세요 — 그 한 건의 신고를 위해 대역 외 PGP 교환을 준비하겠습니다.

## 대응 타임라인

| 단계 | 목표 |
|---|---|
| 접수 확인 | 신고 **72시간** 이내 |
| 첫 번째 분류 + 심각도 평가 | **7일** 이내 |
| `main`에 수정 또는 완화 | 높은 심각도: **7일**, 중간: **30일**, 낮음: 최선 노력 |
| 공개 공개 | 수정 반영 **7–14일** 후 (또는 상호 합의에 의해) |

각 전환 시 업데이트를 받으실 것입니다. 72시간 확인 창 내에 연락이 없으면, 그것 자체가
버그입니다 — 메인테이너를 태그한 GitHub Discussion(일반, 보안 내용 아님)을 통해
에스컬레이션해 주세요.

## 지원 버전

AipeHub는 내부적으로 1.0 이전(`CHANGELOG.md`에서 보이는 v2.0 / v2.1 레이블은
파일 우선 리라이트 세대를 가리키며, SemVer 1.0 임계값이 아닙니다). 현재 `main`
브랜치에서만 보안 이슈를 패치합니다. **LTS 브랜치가 없습니다**.

장기적인 안정성이 필요하면, 감사한 커밋에 고정하고 인플레이스 패치를 위한 예산을
세우세요; 무한정 백포트할 수 없습니다.

## 위협 모델

AipeHub는 **소규모, 신뢰, 단일 테넌트** 배포를 위해 설계되었습니다 — 연구소, 프로젝트
팀, 소규모 공개 프리뷰 그룹. 기본값은 방이 서로 신뢰하는 사람들에 의해 운영된다고 가정합니다.

범위 내 (신고를 수락함):

- ✅ 관리자 엔드포인트에 대한 비인증 접근
- ✅ 토큰 / 쿠키 공개 (사용자 간, 방 간, 프로세스 간)
- ✅ `secrets.enc.json` 및 마스터 키 파일의 암호화 / 복호화 버그
- ✅ 인증 우회 — 예: 작업자가 관리자 전용 경로에 접근
- ✅ 번들된 관리자 UI의 CSRF / 클릭재킹 / XSS
- ✅ *인증이 전혀 필요 없는* 리소스 고갈 (익명 DOS)
- ✅ 호스트를 충돌시키거나 트랜스크립트를 손상시키는 와이어 프로토콜 파싱 버그
- ✅ `TeamBridgeAgent`의 권한 상승 (예: 로컬 팀이 업스트림에 의도하지 않은 가시성 획득)
- ✅ LocalAgentPool / 관리되는 에이전트 스폰 경로의 혼란스러운-대리인 이슈

범위 외 (낮은 우선순위 — 패치 환영, 하지만 보안으로 처리하지 않음):

- ❌ **신뢰할 수 없는 관리자.** 계정이 관리자 역할을 보유하면, 관리자 역할이 노출하는
  모든 것을 할 수 있습니다. 내부 관리자 방화벽이 필요하면, 기능 요청을 열어주세요.
- ❌ *인증된* 사용자에 의한 **애플리케이션 레이어 DDoS**. 속도 제한은 IP별로 재시작
  시 초기화됩니다; 의도적인 내부 남용에 대한 방어가 아닙니다.
- ❌ 메모리 압력을 일으키는 **거대한 태스크 페이로드**. 아직 할당량 없음.
- ❌ 토큰 비교 외부의 **사이드 채널 타이밍 공격** (토큰 비교 자체는 상수 시간).
- ❌ 호스트 기기에 대한 물리적 / 셸 접근이 필요한 이슈.
- ❌ `templates/community/` 업스트림 소스에 대한 발견 — 그것들은 자체 라이선스와
  거버넌스 아래의 제3자 프롬프트 저장소입니다; 그들에게 직접 신고하세요.

발견이 경계에 있으면, GitHub 권고 채널을 통해 보내주시면 분류하겠습니다.

## 기존 완화 조치 (이미 어떤 방어가 있는지 알 수 있도록)

이슈를 평가할 때, 다음 중 하나가 이미 다루는지 확인하세요:

- **토큰 저장**: 관리자 / 작업자 토큰은 디스크에 쓰기 전에 SHA-256으로 해싱됩니다.
  평문은 발행 시 정확히 한 번 표시됩니다. 검증은 상수 시간 비교를 사용합니다.
- **쿠키 저장**: 항상 HttpOnly; `AIPE_COOKIE_SECURE=1`이면 `SameSite=Strict` + `Secure`
  (HTTPS 뒤에서 필요).
- **CSRF**: `AIPE_ALLOWED_HOSTS`는 모든 상태 변경 메서드에서 `Host:`와 `Origin:` 검사를
  모두 강제합니다. **모든 프로덕션 배포에서 설정하세요.** 설정되지 않으면 "루프백만 안전"을
  의미합니다.
- **속도 제한**: `AIPE_ADMIN_RATE_MAX` / `_SEC`은 슬라이딩 창당 IP별 관리자 토큰 검증
  시도를 제한합니다. 기본값 10 / 60초.
- **보안 헤더**: 모든 응답에 `X-Frame-Options: DENY`, 엄격한 CSP, `Referrer-Policy: no-referrer`,
  `X-Content-Type-Options: nosniff`.
- **입장 게이팅**: `AIPE_GATING=admin-approval` (기본값)은 모든 원격 에이전트가 합류 전에
  인간 승인을 받도록 요구합니다. `gating=open`은 **개발 전용**이며 시작 경고와 함께 프로덕션에서
  거부됩니다.
- **API 키 암호화**: 워크스페이스 및 에이전트별 API 키는 `<space>/secrets.enc.json`에 AES-256-GCM으로,
  마스터 키는 `<space>/runtime/secret.key` (0600) 또는 `AIPE_SECRET_KEY` 환경 변수에 있습니다.
  암호화된 파일만으로는 키를 복구하기에 충분하지 않습니다.
- **에이전트별 신원 바인딩 (v0.4)**: `authenticate()`는 `{ ok: true, allowedAgents: [...] }`를
  반환할 수 있으므로, 유출된 API 키는 임의의 에이전트 id를 사칭할 수 없습니다 — 바인딩된 것들만 가능합니다.
- **트랜스크립트는 추가 전용입니다**: 런타임에서 트랜스크립트 항목을 삭제하거나 재작성하는 API가
  없습니다. 변조에는 파일시스템 접근이 필요합니다 (범위 외입니다; 위의 "범위 외"를 참조하세요).

## 협력 공개

표준 협력 공개를 따릅니다:

1. 세부 내용을 비공개로 보냅니다 (GitHub 권고 채널 권장).
2. 메인테이너가 확인, 범위 설정, 수정 개발 및 테스트합니다.
3. 수정이 `main`에 반영됩니다 (LTS 약속이 있으면 백포트 브랜치도).
4. 7–14일 후 공개 공개:
   - CVE id (적절하면 요청할 것임)
   - 익명으로 남기를 원하지 않으면 권고에 귀하에게 크레딧
   - `CHANGELOG.md`의 영향 + 완화 요약

수정 출시 전에 공개적으로 공개하면, 여전히 수정을 출시하겠지만, 권고의 크레딧
필드에 "비협력적"으로 표시됩니다.

## 운영자를 위한 보안 체크리스트

허브를 **운영 중**이지 버그를 신고하는 것이 아니라면, 배포 측 강화 체크리스트는
[`docs/DEPLOY.md` § "프로덕션 체크리스트"](../../docs/DEPLOY.md#production-checklist)에 있습니다.

요약:

- [ ] HTTPS 뒤에 있을 때 `AIPE_COOKIE_SECURE=1`
- [ ] `AIPE_ALLOWED_HOSTS`를 실제 호스트명으로 설정
- [ ] `AIPE_GATING=admin-approval` (공용 인터넷에서 절대 `open`이 아님)
- [ ] Caddy / nginx가 TLS를 종료; 백엔드는 `127.0.0.1`에 바인딩
- [ ] `runtime/secret.key` (chmod 600) 또는 `AIPE_SECRET_KEY` 환경 변수 설정
- [ ] `<space>/` 디렉토리에 대한 백업 존재
- [ ] 잠금을 복구할 수 있도록 최소 2개의 관리자 계정
- [ ] `/healthz` 모니터링됨
