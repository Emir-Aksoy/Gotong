## 脆弱性の報告方法

**セキュリティの問題については、公開の GitHub イシュー、ディスカッション、PR を開かないでください。** プライベートチャネルを使用してください:

### 推奨 — GitHub プライベート脆弱性報告

次の場所でプライベートアドバイザリを開いてください:

> **<https://github.com/Emir-Aksoy/Gotong/security/advisories/new>**

GitHub の組み込みフォームにより以下が得られます:

- メンテナーとのエンドツーエンドのプライベートスレッド（メールの漏洩なし）
- 添付ファイルと再現手順が一箇所に
- 報告 → 修正 → 公開 CVE 割り当てまでの追跡されたタイムライン

これは私たちが最初に読み、最も早く回答するチャネルです。無料の GitHub アカウントが必要です。それが唯一の前提条件です。

### メールチャネルなし（1.0 以前）

v0.x 期間中は意図的に**セキュリティメールがありません**。`security@gotong.dev` はこのリポジトリの古いリビジョンに*希望的な*アドレスとして表示されますが — ドメインは登録されておらず、メールボックスはアクティブ化されていないため、そこへのメールはどこにも届きません。本物の報告を信頼するかもしれない誰かに空の連絡先をぶら下げるのではなく、フォールバックとしての広告を停止しました。

GitHub プライベート脆弱性報告（上記）が今日の**唯一**のチャネルです: 無料、プライベート、そして最初に読むチャネルです。本物のメールボックスを設立する価値があるかどうかは、1.0 のリリース直前まで延期された [リリースチェックリスト](.github/RELEASE-CHECKLIST.md#security-contact) の決定です。それまでの間、アドバイザリフォームを使用してください。

本当に GitHub を使用できない場合は、GitHub ディスカッション（セキュリティコンテンツなし）で**非セキュリティ**のディスカッションを開き、メンテナーに連絡を求めてください — 脆弱性の詳細なしで — その一つの報告のためにプライベートチャネルを手配します。

アドバイザリに含めるもの:

- 問題の説明
- 正確な再現手順
- テストしたコミットハッシュ（`git rev-parse HEAD`）
- （オプション）提案された修正またはパッチ
- （オプション）アドバイザリでクレジットされたい名前 / ハンドル

### PGP については?

現在、PGP キーを公開していません。理由:

- GitHub のプライベートアドバイザリチャネルは、あなたとメンテナーの通知の間でエンドツーエンドで TLS 暗号化されているため、PGP はほとんど追加されません。
- 初期プロジェクトの PGP キーを維持することは、メリットよりも多くの失敗モード（失われたキー、期限切れのキー、署名式）があります。

組織のポリシーが PGP 暗号化された開示を必要とする場合は、まず GitHub チャネルを通じて連絡してください。その単一の報告のためにアウトオブバンドの PGP 交換を手配します。

## 対応タイムライン

| フェーズ | 目標 |
|---|---|
| 受信確認 | 報告から **72 時間**以内 |
| 最初のトリアージ + 重大度評価 | **7 日**以内 |
| `main` への修正または緩和 | 高重大度: **7 日**、中程度: **30 日**、低: ベストエフォート |
| 公開開示 | 修正が着地してから **7〜14 日**後（または合意により） |

すべてのトランジションで更新を受け取ります。72 時間の確認ウィンドウ内に連絡がない場合、それ自体がバグです — GitHub ディスカッション（一般、セキュリティコンテンツなし）でエスカレーションし、メンテナーにタグ付けしてください。

## サポートされているバージョン

Gotong は内部的に 1.0 以前です（`CHANGELOG.md` に見られる v2.0 / v2.1 ラベルはファイルファーストの書き直し世代を指し、SemVer 1.0 の閾値ではありません）。現在の `main` ブランチのみにセキュリティの問題にパッチを当てます。**LTS ブランチはありません**。

長期的な安定性が必要な場合は、監査したコミットにピン留めし、インプレースパッチを予算化してください。バックポートを無期限に行うことはできません。

## 脅威モデル

Gotong は**小さな、信頼できる、シングルテナント**のデプロイメント向けに設計されています — 研究室、プロジェクトチーム、小さな公開プレビューグループ。デフォルトは、部屋がお互いを信頼する人々によって運営されることを前提としています。

スコープ内（報告を受け入れるもの）:

- ✅ 管理エンドポイントへの未認証アクセス
- ✅ トークン / クッキーの開示（ユーザー間、部屋間、プロセス間）
- ✅ `secrets.enc.json` とマスターキーファイルの暗号化 / 復号化バグ
- ✅ 認証バイパス — 例えば、ワーカーが管理者専用ルートに到達する
- ✅ バンドルされた管理 UI の CSRF / クリックジャッキング / XSS
- ✅ *認証なし*で必要なリソース枯渇（匿名 DOS）
- ✅ ホストをクラッシュさせるか、トランスクリプトを破損させるワイヤープロトコルの解析バグ
- ✅ `TeamBridgeAgent` における権限昇格（例: ローカルチームが上流への意図しない可視性を得る）
- ✅ LocalAgentPool / マネージドエージェントのスポーンパスにおける confused-deputy の問題

スコープ外（優先度低 — パッチは歓迎するが、セキュリティとして処理しない）:

- ❌ **信頼できない管理者。** アカウントが管理者ロールを保持すると、管理者ロールが公開するすべてのことができます。内部の管理者ファイアウォールが必要な場合は、機能リクエストを開いてください。
- ❌ *認証済み*ユーザーによる**アプリケーション層 DDoS**。レート制限は IP ごとで、再起動時にリセットされます。意図的な内部悪用に対する防御ではありません。
- ❌ メモリ圧迫を引き起こす**巨大なタスクペイロード**。まだクォータがありません。
- ❌ トークン比較以外の**サイドチャネルタイミング攻撃**（トークン比較自体は定数時間です）。
- ❌ ホストマシンへの物理 / シェルアクセスを必要とする問題。
- ❌ `templates/community/` のアップストリームソースに対する調査結果 — それらは独自のライセンスとガバナンスの下にあるサードパーティのプロンプトリポジトリです。直接報告してください。

あなたの調査結果が境界にある場合は、GitHub アドバイザリチャネルを通じて送ってください。トリアージします。

## インプレース緩和策（既存の防御を把握するために）

問題を評価するとき、これらのいずれかが既にカバーしているかどうかを確認してください:

- **トークンストレージ**: 管理者 / ワーカートークンはディスクに書き込まれる前に SHA-256 でハッシュ化されます。平文はミント時に一度だけ表示されます。検証は定数時間比較を使用します。
- **クッキーストレージ**: 常に HttpOnly。`GOTONG_COOKIE_SECURE=1` のとき `SameSite=Strict` + `Secure`（HTTPS 背後で必須）。
- **CSRF**: `GOTONG_ALLOWED_HOSTS` は、すべての状態変更メソッドで `Host:` と `Origin:` の両方のチェックを強制します。**すべての本番デプロイメントで設定してください。** 未設定は「ループバックのみが安全」を意味します。
- **レート制限**: `GOTONG_ADMIN_RATE_MAX` / `_SEC` は IP ごとのスライディングウィンドウごとの管理トークン検証試行回数を制限します。デフォルト 10 / 60 秒。
- **セキュリティヘッダー**: すべてのレスポンスに `X-Frame-Options: DENY`、厳格な CSP、`Referrer-Policy: no-referrer`、`X-Content-Type-Options: nosniff`。
- **アドミッションゲーティング**: `GOTONG_GATING=admin-approval`（デフォルト）は、すべてのリモートエージェントが参加する前に人間によって承認されることを要求します。`gating=open` は**開発専用**であり、起動警告とともに本番環境では拒否されます。
- **API キー暗号化**: ワークスペースとエージェントごとの API キーは `<space>/secrets.enc.json`、AES-256-GCM、マスターキーは `<space>/runtime/secret.key`（0600）または `GOTONG_SECRET_KEY` env に存在します。暗号化ファイルだけではキーを回復するのに十分ではありません。
- **エージェントごとのアイデンティティバインディング（v0.4）**: `authenticate()` は `{ ok: true, allowedAgents: [...] }` を返せるため、漏洩した API キーが任意のエージェント ID を偽装できません — バインドされたもののみです。
- **トランスクリプトは追記のみ**: ランタイムからトランスクリプトのエントリを削除または書き換える API はありません。改ざんにはファイルシステムアクセスが必要です（スコープ外。上記の「スコープ外」を参照）。

## 調整済み開示

私たちは標準的な調整済み開示に従います:

1. 詳細をプライベートに送ってください（GitHub アドバイザリチャネルを推奨）。
2. メンテナーが確認、スコープ設定、修正の開発とテストを行います。
3. 修正が `main` に着地します（LTS のコミットメントがある場合はバックポートブランチにも）。
4. 7〜14 日後に公開開示、次とともに:
   - CVE ID（適切であればリクエストします）
   - アドバイザリであなたへのクレジット、匿名を希望する場合を除く
   - `CHANGELOG.md` への影響 + 緩和策の概要

修正を出荷する前に公開で開示した場合でも、修正を出荷しますが、アドバイザリのクレジットフィールドには「調整なし」と記載されます。

## オペレーター向けセキュリティチェックリスト

バグを報告するのではなく hub を**運営している**場合、デプロイメント側の強化チェックリストは [`docs/DEPLOY.md` § "Production checklist"](../../docs/DEPLOY.md#production-checklist) にあります。

要約:

- [ ] HTTPS の前にある場合は `GOTONG_COOKIE_SECURE=1`
- [ ] `GOTONG_ALLOWED_HOSTS` を実際のホスト名に設定
- [ ] `GOTONG_GATING=admin-approval`（公共インターネットで `open` は絶対に禁止）
- [ ] Caddy / nginx が TLS を終端。バックエンドは `127.0.0.1` にバインド
- [ ] `runtime/secret.key`（chmod 600）または `GOTONG_SECRET_KEY` env を設定
- [ ] `<space>/` ディレクトリのバックアップが存在する
- [ ] ロックアウトを回復できるように少なくとも 2 つの管理者アカウント
- [ ] `/healthz` を監視している
