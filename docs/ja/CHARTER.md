## 1. AipeHub とは何か

**AipeHub は、AI 時代における人・エージェント・組織間のつながりのための、セルフホスト型の基盤です。** AI + 人 + Hub。

それはエージェントでも、別のエージェントフレームワークでもありません。それらの*下*にある層——レジストリ、メッセージバス、タスクルーター、統治されたフェデレーションリンク、および追記専用トランスクリプトです。LangGraph グラフ、CrewAI クルー、CLI コーディングエージェント（Claude Code、Codex）、外部 A2A エージェント、そして人間が一つの抽象化：`Participant` を通じて同じ部屋に接続されます。Hub はシグナルの流れを維持し、境界を強制します。モデル自体を実行しないため、すべての決定は参加者に留まります。

メンタルモデルは片手で持てるほど小さいです：

- **意図的に単純な Hub。** メッセージをルーティングし、タスクをディスパッチし、トランスクリプトを永続化し、イベントを発行し、ガバナンスゲートを強制します。エージェントループを所有したり、判断を下したりしません。
- **一種類の参加者。** 人間はエージェントとまったく同じ `Participant` です。「request-human-input ツール」は存在しません。人とエージェントは同じタスク、トランスクリプト、および長時間実行プリミティブを共有します。
- **ただのファイルである状態。** ワークスペースはディスク上のディレクトリ（`.aipehub/`）です。削除すれば部屋は消え、コピーすれば部屋全体を手渡したことになり、再起動しても何も失われません。

その組み合わせ——巧みなプロトコル一つではなく——が AipeHub の*本質*です。

## 2. 北極星——3 つの譲れない原則

3 つの文がプロジェクトを定義します。これらは機能ではなく、アイデンティティです。これらのいずれかを変更しても、より良い AipeHub は生まれません——もはや AipeHub ではない何かが生まれるだけです。

1. **フレームワークは LLM を実行しない。** Hub はルーティング、ディスパッチ、記録、ゲートを行います。推論と意思決定は参加者に委ねられます。これが、システムが同時に単純で、監査可能で、信頼できる理由です。
2. **人間とエージェントは同じ `Participant` です。** Human-in-the-loop は後付けではありません。人間はファーストクラスの参加者であり、タスクをディスパッチされ、サスペンドし、再開できます——エージェントが使用するのと同じ仕組みを使って。
3. **状態はディスク上のファイルです。** 主権は設定ではありません。あなたの部屋、認証情報、トランスクリプト、履歴はあなたが所有しコピーできるディレクトリです。他に復元するものがないため、再起動は透明です。

これら 3 つはプロジェクトで最も変更が難しいものです（§10）。それ以外のすべては、これらのために交渉可能です。

## 3. なぜ存在するのか——3 つの層

AipeHub は、ユーザーがたどり着く順序で、3 つの層にまたがる作業基盤を構築します：

**第 1 層——人と自分自身の AI。**「私の AI デスクトップ。」一人の人の Hub、プライベートなワークフロー、マシンから決して出ない認証情報。目標は、5 分で動作するものを、コードなしで、AI があなたのために実際の作業をする状態にすることです。

**第 2 層——境界を越えた人々とエージェント。** クロスオーガニゼーションコラボレーション：複数のユーザー、ロール、招待状、ピアツーピアフェデレーション。ワークフローは組織の境界を越えることができますが、認証情報、データ、課金はそれぞれ自組織に留まります。境界を越えるものは明示的なリンクごとの信頼契約によって制約され、重要な越境には人間の確認を待ちます。

**第 3 層——フレームワーク自体。** 明確で、安定していて、適応可能。Hub は意図的に単純な状態を保ちます。`Participant` は唯一の抽象化として留まります。プロトコル、認証情報、クォータはすべて明示的で可視的なエッジを持ちます。フレームワーク自体の仕事は、第 1 層と第 2 層を裏切ることなく、AI の急速な発展に追いつき続けることです。

AI、エージェント、そして現在登場している多エージェントフレームワークは、驚くべき速さで日常生活に溶け込むでしょう。大手モデルベンダー、コミュニケーションプラットフォーム、その他の既存勢力は、それを独占として占領しようと同じ速さで動いています——依存するすべての人から搾取するためのレバレッジ確保が、その価値全体の源です。AipeHub は代替案をテーブルに残すために存在します：AI、エージェント、ワークフローを一つのスタックに統合し、オープンソースとしてリリースすることで、そのレバレッジを鈍らせ、誰でもそれを入手し、デプロイし、自分で実行できるようにします——ゲートキーパーなし、家賃なし、許可を求める必要なし。

AI に*重要なことを実行させたい*——家庭を管理し、家族を助け、お金を処理し、チームを調整したい——が、そのためのキーをコントロールできないクラウドに渡したくない、という人のために存在します。

## 4. 信頼の楔——なぜ重要なものを任せられるのか

ほとんどの AI ツールは 2 つの選択肢を提供します：コントロールできないクラウドにすべてを委ねるか、自分ですべてをつなぎ合わせるか。AipeHub は 3 つ目の選択肢です——境界が本物であり、あなたのものであるため、家庭、家族、またはお金に向けられる AI。3 つの特性がそれを真実にし、プロジェクトの堀となります：

- **ガバナンス。** 可逆的なアクションはそのまま実行されます。不可逆的なもの——ドアに鍵をかける、お金を使う、子どものデータをリンク越しに送る——は、人間がインボックスで確認するまで待ちます。ワークフローはゲートをスキップできません。危険な越境アクションは構造的に fail-closed です。
- **ローカル。** 認証情報はあなた自身の `.aipehub/` ディレクトリに暗号化されて保存されます。別の Hub とフェデレートすることで共有されるのは*機能*であり、ボルトではありません。各 Hub は独自の認証情報ストアと独自の使用量/コスト台帳を保持します。
- **オープン。** すべてのディスパッチとすべての結果は、読むことができる追記専用トランスクリプトです。フレームワークはモデルを実行しないため、信頼のうえに乗っかる隠れた判断は存在しません。

そして権限はあなたのものであり、私たちのものではありません。AipeHub はメカニズム——ゲート、ボルト、トランスクリプト——を提供しますが、ポリシーは Hub を運営する人物が設定し保持します。このプロジェクトではありません。どのアクションが重要とみなされるか、ホワイトリストが何を許可するか、誰がインボックスに答えるか：オペレーターが決定し、オペレーターが決定を保持します。家庭学習 Hub では、ゲートを設定し確認するのは AipeHub ではなく親です。私たちはあなたの決定にベンダーの席を設けません。フレームワークはコントロールを手渡し、邪魔をしません。

これが楔です：「他よりも有能」ではなく、*ブラックボックスに決して渡さないものに信頼できる*。

## 5. ビジョン——どこへ向かうのか

最終的な状態は**自由なグラフであり、ツリーではありません。** テナントが内部にスペースを借りる一つの中央プラットフォームではなく、ピアツーピアで相互接続する多くの主権 Hub——それぞれがそれを運営する人または組織によって所有され、どれも他者の信頼を所有しない。コントロールプレーンはオプトイン、プライバシー保護、カウントのみのサマリーで*観察*できますが、決して*乗っ取らない*。このようなグラフはコーナーに追い込まれることはありません：乗っ取る中心がなく、家賃を請求する家主がなく、運営を続けるために許可を必要とする単一の当事者がいません。

そのグラフ上の参加者は人々とソフトウェアエージェントだけではありません。具体化されたインテリジェンス——ロボット、センサー、物理世界で行動するマシン——は同じタイムラインで到来しており、AipeHub はそれを受け入れるよう構築されています：そのようなシステムが **MCP** を話す限り、同じガバナンス境界の後ろで一つの `Participant` として参加します——タスクをディスパッチされ、同じゲートに従い、他のすべてと同じトランスクリプトに書き込まれます。フレームワークは、人々、エージェント、具体化されたシステムを一つの屋根の下で管理するのに自然に適しています。なぜなら、すでにそれらを一種のものとして扱っているからです。

そのグラフの上に**再利用可能なコンポーネントのガバナンスされた市場**が成長します——テンプレート、アダプター、知識ベースコネクター——一つのファイルで完全に機能するアーキテクチャを誰かに手渡せるように構築され、ガバナンスの姿勢がそれとともに移動し、知識コンテンツは移動しないため、信頼できます。出所は誠実です：テンプレートは構造と参照を運びますが、別の組織のデータや人々は運びません。

そのエコシステムの通貨は**認識であり、家賃ではありません**（§7）。貢献者が貢献し続けるのは、誠実な帰属とプロジェクトへの発言権への道——支払いではありません。自由なグラフが分散化されたまま維持されるのは、台帳を清算するために中央当事者が必要とされないからです。

## 6. 使い方

AipeHub は、それぞれに合ったサーフェスで人々に応じ、同じ Hub がそれらすべての後ろにあります：

- **パーソナルモード** — 5 分、コードなし。フラッグシップテンプレートをインポートするか、フォームからエージェントを構築します。ホストがそれを生成します。認証情報はディスク上に留まります。
- **チームモード** — 一つの部屋、多くのロール（admin / worker / agent）、招待状、リソースレベルの RBAC、`/me` でのメンバーセルフサービス。
- **フェデレーション** — 各組織が独自の Hub を実行します。`HubLink` がリンクごとの信頼契約（機能アローリスト・データクラスゲート・クォータ・失効・知識ベースアローリスト）の下で 2 つをつなぎます。ワークフローはリンクを越えることができます。主権は両側で維持されます。

サーフェスは意図的に複数あります：ブラウザ管理 UI、`/me` メンバーデスクトップ、IM ブリッジ（現在 Telegram、Lark、Slack、QQ；Discord と Matrix は計画中）、インタラクティブ CLI/REPL、ツールと外部クライアントのための MCP、インストール可能な PWA。AipeHub はエコシステムのオープンプロトコルが存在する場所で話します——**MCP**（ツールとデータ、双方向）、**A2A**（エージェント間、双方向）、**ACP**（保持されたコーディングエージェントセッションの駆動）——そして独自のものを 1 つだけ所有します：**HubLink**、2 つの Hub 間のガバナンスされたフェデレーションリンク。

テンプレートは Hub 全体を運びます：1 つのファイルに N 個のエージェント、N 個のワークフロー、アドレス可能な知識ベーススロット、そしてワンプロンプトのキーセットアップが詰め込まれています——構造と配線であり、知識そのものでも、あなたの人々や秘密でもありません。

→ [`docs/OVERVIEW.md`](../../docs/OVERVIEW.md) から始める · [`docs/zh/HANDS-ON-HUBS.md`](../../docs/zh/HANDS-ON-HUBS.md) で既製の Hub を実行する · [`docs/AGENT.md`](../../docs/AGENT.md) でエージェントをつなぐ。

## 7. 意思決定方法——ガバナンスと認識

決定は公開の場で、設計ラインの感覚を身に付けた人々によって行われます。権限の階段、メンテナーが内面化しなければならない一つの設計ルール（「フレームワークは LLM を実行しない」）、そして貢献者からメンテナー、スチュワードへの道は [`GOVERNANCE.md`](../../GOVERNANCE.md) に記載されています。現在の名簿は [`MAINTAINERS.md`](../../MAINTAINERS.md) です。

貢献は**認識のみで報われます**——誠実な出所、可視的な帰属、プロジェクトへの実際の発言権への文書化された道。お金なし、トークンなし、バウンティ層なし。経済的な層は、プロジェクト全体が依拠するファイルファースト、分散型信頼モデルを曇らせるからです。そのシステムの 4 つの柱——引用リーダーボード、メンテナーラダー、摩擦のない共有、共有サンプル——は [`docs/RECOGNITION-SYSTEM.md`](../../docs/RECOGNITION-SYSTEM.md) に集約されています。

貢献方法と、コミュニティテンプレートがクリアしなければならないバー（ライセンスが明確、パース可能、プレーンテキストの秘密がゼロ、出所が宣言されている）は [`CONTRIBUTING.md`](../../CONTRIBUTING.md) にあります。行動規範は [`CODE_OF_CONDUCT.md`](../../CODE_OF_CONDUCT.md) にあります。セキュリティレポートは [`SECURITY.md`](../../SECURITY.md) を通じて行われます。

## 8. 非目標——何になることを拒否するか

憲章は約束するものだけでなく、除外するものによっても基準を定めます。AipeHub は意図的に**以下ではありません**：

- **モデルホスト。** Hub に LLM 推論ループを成長させません。中立な `LlmProvider` インターフェースの後ろに自分のモデルを持ち込んでください。
- **あなたの信頼を所有する中央 SaaS。** あなたのキー、データ、または課金を使用の対価として保持するプラットフォームにはなりません。コントロールプレーンは観察します。管理権を取りません。
- **階層的な組織ツリー。** フェデレーションは主権ピアの自由なグラフであり、家主の中に入れ子にされたテナントではありません。Hub は構造的に他より上位にはありません。
- **お金・トークン・バウンティ経済。** インセンティブ層は認識です（§7）。通貨を導入することは、プロジェクトが分散化しようとしている信頼を中央集権化することになります。
- **暗闇で行動する自律ループ。** 人間のゲートなしにエージェントが不可逆的または越境アクションを取ることを許可しません。「プロアクティブ」は決して「重要事項で監督なし」を意味しません。

提案された機能がこれらの一つを破ることを要求する場合、それは憲章レベルの会話です（§10）。プルリクエストではありません。

## 9. オープン性

フレームワークは全体として **MIT ライセンス**——クローズドソースと SaaS 製品に組み込み可能、変更可能、再配布可能。ライセンスファイルと著作権表示が保持されます（[`LICENSE`](../../LICENSE)、[`docs/LICENSE-FAQ.md`](../../docs/LICENSE-FAQ.md)）。コミュニティテンプレートは独自の CC0/MIT 出所を持ち、すべて商用利用互換です。

AipeHub が発明したフォーマットは、実行中のコードとしてだけでなく、仕様として公開されています。ワークフローとエージェント定義言語（`aipehub.workflow/v1` およびエージェント、チーム、テンプレートマニフェスト）と **HubLink**——2 つの Hub 間のフェデレーションプロトコル——は私たちの設計であり、誰ものものです：誰でも読み取り、実装し、その上に競合するランタイムを構築し、商業ソフトウェアでリリースできます。料金も許可も必要ありません。フォーマットは共有言語であり、財産ではありません。これらは単にプロジェクトの仕様であり、書き記したこと以上の主張はしません。その形状は**付録 A**（HubLink）と**付録 B**（定義フォーマット）に示されています。

ここでのオープン性はライセンスだけではありません。それはアーキテクチャです。あなたのワークスペースは読み取り、コピー、バックアップ、そして離れることができるディレクトリです。脱出すべきロックインがないのは、最初から管理権が存在しなかったからです。

## 10. この憲章の修正

これは生きているドキュメントですが、軽率に編集されることはありません。変更が及ぶ深さが深いほど、バーは高くなります：

- **北極星（§2）** は、メンテナーとスチュワードの明示的で意図的なコンセンサスによってのみ、公開の場で、理由を記録した上で修正可能です。ほとんど決して変わらないべきです。変わる場合、プロジェクトは別のプロジェクトになっており、それは声を大にして言われるべきです。
- **その他のすべて** は、リポジトリの残りの部分と同様に進化します：*なぜ*を論じるプルリクエスト、北極星に照らしてレビューされ、[`GOVERNANCE.md`](../../GOVERNANCE.md) に従ってレイジーコンセンサスでマージされます。

採択された修正はすべて、このドキュメントの先頭に記録されたバージョンを更新します（ソフトウェアのバージョンとは別です）：通常の変更にはマイナーバンプ、北極星への変更にはメジャーバンプが予約されています——プロジェクトが別のものになったことを意味する唯一の変更。バージョンは読者が保持している憲法を知る方法です。

構築したいものが合致するかどうか不明な場合、この憲章が議論すべきドキュメントです。それがこれの目的です。

## 11. ホームと開かれた招待

このプロジェクトの正規のホームはその GitHub リポジトリです：
**[github.com/Emir-Aksoy/AipeHub](https://github.com/Emir-Aksoy/AipeHub)**——オリジナルであり、真実の記録。フォークは歓迎され、奨励されますが、設計ラインが保持され、憲章、プロトコル、フラッグシップテンプレートが権威を持つのはここです。

ワークフローを持ち帰ってください。家庭、店舗、チームのために構築したワークフロー——どんなに小さくても、どんなに荒削りでも——共有する価値があります：ある人の午後のハックは別の人の 5 分のスタートになります。プルリクエストを開き、コミュニティテンプレートとして提出するか、単に Discussions で見せてください。基準は安全と誠実さであり、磨かれた完成度ではありません（[`CONTRIBUTING.md`](../../CONTRIBUTING.md)）。

**誰でも参加を歓迎します**——使用、貢献、メンテナンス、そしてこれがどこへ向かうかについての実際の発言権を得ることができます（[`GOVERNANCE.md`](../../GOVERNANCE.md)）。そして、最終状態が一つの中央プラットフォームではなく主権 Hub の自由なグラフであるため、**地域チャプターを歓迎します**：独自の Hub を運営し、自分たちの人々のためにテンプレートをキュレートし、自分の言語で newcomers を助けるローカル、言語、またはコミュニティルートのグループ。チャプターは独自の部屋を所有し、中央の家主には答えません。アーキテクチャが意図する通り、より広いグラフにピアとして接続します。

## 付録 A — HubLink、フェデレーションプロトコル

*これらの付録は、方向付けのために各フォーマットの現在の形状を要約しています。権威ある、バージョン管理された仕様は参照元に存在し、この憲章より速く進化します。それらが異なる場合、参照仕様がワイヤーを統治し、憲章が原則を統治します（§9）。*

**HubLink** は 2 つの Hub 間のガバナンスされたリンクです：対称的なピアツーピア WebSocket チャネルで、どちら側も相手にタスクをディスパッチしたり、メッセージを発行したり、リンクを閉じたりできます。それは `ws://` / `wss://` 上の JSON フレームであり、`MESH_PROTOCOL_VERSION` によってバージョン管理され、エージェント↔Hub ワイヤープロトコル（[`docs/PROTOCOL.md`](../../docs/PROTOCOL.md)）とは意図的に分離されています——ピアリンクはそのプロトコルの入学ゲーティングやエージェントレジストリをまったく必要としません。

フレーム、それぞれ `type` 識別子を持つ自己完結型の JSON オブジェクト：

| フレーム | 方向 | 意味 |
|---|---|---|
| `MESH_HELLO` / `MESH_HELLO_ACK` | ハンドシェイク | 相互ピア認証（リンクごとの認証情報、fail-closed） |
| `MESH_TASK` / `MESH_RESULT` | どちらの方向も | タスクをディスパッチする / その結果を返す（タスク id で照合） |
| `MESH_MESSAGE` | どちらの方向も | ファイアーアンドフォーゲットメッセージ |
| `MESH_PING` / `MESH_PONG` | どちらの方向も | キープアライブ |
| `MESH_GOODBYE` | どちらの方向も | 協調的なクローズ |

リンクを越えるものはリンクごとの**信頼契約**——機能アローリスト、データクラスゲート、クォータ、失効、知識ベースアローリスト——によって制限されており、ワークフローが組織の境界を越えながら認証情報、データ、課金が両側に留まることができます。

HubLink は AipeHub 独自の設計であり、誰もが所有するオープン仕様として提供されています：任意の言語や製品で、商用かどうかに関わらず、料金なし、許可不要で実装できます。リファレンス実装は `packages/transport-ws/` にあります。

## 付録 B — ワークフローとエージェント定義フォーマット

これらの YAML フォーマットは、Hub の構造が書き記され、ガバナンスされ、共有される方法です。すべて AipeHub 独自の設計であり、すべて HubLink と同じ条件で提供されます：誰もが所有するオープン仕様——どこでも実装でき、商用かどうかに関わらず、料金なし、許可不要。

- **`aipehub.workflow/v1`** — 宣言的ワークフロー：`trigger`、各ステップが*機能*（名前付きエージェントではなく）にディスパッチする `steps` のリスト、`$ref` でつながれたディスパッチグラフ、およびコントロールフローシュガー `when:`（条件付き）、`parallel:`（ファンアウト）、`human:`（Human-in-the-loop ステップ）。オプションの `surface.me` はメンバーデスクトップに公開し、`governance` はリスクの姿勢を宣言します。YAML はワークフローのガバナンスされた、バージョン管理されたルートです——各実行は不変の改版に固定されるため、ドリフトしません。
- **`aipehub.agent/v1`** と **`aipehub.team/v1`** — エージェントマニフェスト、1 つのファイルに 1 つのエージェントまたはチーム全体：プロバイダー、モデル、システムプロンプト、機能、MCP サーバー、サブエージェントディスパッチアローリスト、オプションのハートビート。
- **`aipehub.template/v1`** — テンプレートは N 個のエージェント、N 個のワークフロー、アドレス可能な知識ベース*スロット*、およびワンプロンプトのキーセットアップを 1 つの共有可能なファイルにパックします。構造と参照のみを運びます——知識コンテンツ、あなたの人々、あなたの秘密は決して運びません。

リファレンス実装：`packages/workflow/`（ワークフローランナーとスキーマ）と `packages/web/src/manifest.ts` / `template-manifest.ts`（エージェント、チーム、テンプレートマニフェスト）。
