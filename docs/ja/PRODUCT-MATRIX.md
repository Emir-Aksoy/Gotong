## 2. 弱みマトリックス: 正直な逆比較（エンタープライズバイヤーが気にする次元に入れ替えると、AipeHub が弱い方になる）

上の表はホームフィールドアドバンテージで勝ちます。エンタープライズバイヤーが実際に尋ねる次元に入れ替えると、差は直ちに逆転します:

| 次元 | 強い製品（具体的な製品） | AipeHub |
|---|---|---|
| 顧客の検証 / スケール | Salesforce Agentforce（8000 以上の顧客）、ServiceNow | ❌ 初期段階 / 自己使用 |
| エコシステム統合（CRM/ITSM/Office/SAP） | Salesforce、ServiceNow、Microsoft | ❌ MCP 経由で自分でワイヤー |
| コンプライアンス認定（SOC2/ISO/HIPAA） | すべての商業プラットフォーム | ❌ 認定なし |
| 即使い / ノーコードの成熟度 | n8n、Dify、Agentforce | ⚠️ 設定が必要 / example-first |
| バンドルされた強力なモデル + SLA + 商業サポート | すべての商業プラットフォーム | ❌（設計上モデルを実行しない; ワイヤーする MiMo/DeepSeek/Claude に依存） |

## 3. 一行の結論

**AipeHub は「より良い Agentforce」でも「より強力な n8n」でもなく; 誰も占有していない交差点のセルに座っています**: セルフホスト主権 + 組織間フェデレーション + 組織レベルのガバナンス / HITL + 個人から組織への継続性パス + フレームワークが LLM を実行しない。代償は「成熟度 / エコシステム / コンプライアンス / 顧客検証」において初期段階の製品であること — これはまさに商業プラットフォームが最も強い領域です。

> データソース: vdf.ai ベンダーパノラマ / Futurum プラットフォームウォーズ / Zylos プロトコル層 / Knowlee OSS セルフホスト / Strata HITL、プラスコードベースの測定（32 パッケージ / 85.7k LOC / テスト比率 > 1:1 / 41 デモ）。[`COMPETITIVE-LANDSCAPE.md`](../COMPETITIVE-LANDSCAPE.md) を参照。

## 4. 私たちに最適なユーザー — 「ニーズはあるが今日は満たされていない」

2 つのマトリックスを重ねると、**サービスを受けているセルはすでに混雑している; 未サービスのセルは 1 つだけ**です:

> **ガバナンス / 監護 / 承認 + データ主権 + 境界を越えた協働を必要とするが、①エンタープライズグレードのプラットフォームを買えず使えない、かつ② OSS フレームワークの「シングル組織、ガバナンスなし」の上限に閉め出されている小規模な組織。**

このセルは今日ほぼ真空です — 誰も構築したくないからではなく、**2 つの壁に同時に塞がれているから**です:

- **壁 A（価格 / 成熟度）**: エンタープライズプラットフォーム（Agentforce/ServiceNow/Microsoft/Google）はガバナンスと HITL を持つが、高 ACV、高タッチ SaaS GTM であり、家庭、タピオカミルクティーショップ、3 人の法律事務所には**単純に販売しない**。
- **壁 B（アーキテクチャ）**: OSS フレームワーク（LangGraph/Dify/n8n）と個人エージェント（OpenClaw/Goose）は安くてセルフホスト可能だが、**アーキテクチャ的に組織間フェデレーション、組織レベルのガバナンス、アウトバウンド承認ゲートがない** — そして安さがそれらの能力を育てることはありません。

AipeHub は 2 つの壁の隙間に立っています: エンタープライズプラットフォームの「ガバナンス + HITL + データ主権」と、OSS フレームワークの「セルフホスト + 安価 + クレデンシャルはあなたのマシンに」の両方を持ち、**さらに独自にあの 3 列（組織間フェデレーション / 個人↔組織の継続性 / フレームワークが LLM を実行しない）を所有しています**。

### 4.1 最もシャープな 2 つのビーチヘッド（例を通じて既に構築したもの）

| ビーチヘッド | 誰が | 今日満たされていない理由 | 構築した例 |
|---|---|---|---|
| **A. 家族 / 教育** | 子供のために AI を開く親、複数メンバーの家族主権 AI、保護者の監護 + 承認 | エンタープライズプラットフォームは家族に販売しない; 個人エージェントは 1 人 / 1 マシンで監護 / クロスメンバー主権 / 承認ゲートがない | `family-learning-hub`（2 つの主権ハブ + アウトバウンド承認ゲート + 子供データをロックするデータクラス）、`/teach` 導師、親へのトランスクリプトフォーク |
| **B. 中小企業の組織間** | サプライチェーン（ショップ↔サプライヤー）、フランチャイズチェーン（本部↔店舗）、メンタリング / クラブ / 会社間プロジェクト | エンタープライズプラットフォームは非常に小さい組織には重 / 高価すぎる; OSS フレームワークはシングル組織 | `tea-supply-link`、`tea-chain-hq`、`warband-club`、`cafe-ops` |

1 つ外側に押し出すと、同じセルにはまた**規制された小チームのフェデレーション**も: 法律事務所コンソーシアム、クリニック、会社間 RFP、研究協働 — すべて「組織間協働 + データは自分の手元に + 監査 / 承認が必要」であり、今日は同様に頭から満たされていません。

### 4.2 DeepSeek が「価格の壁」を押し倒した — ユーザーが指摘したまさにその変数

ユーザーの判断は完全に正しい: **以前は価格 / 性能の根拠がなかった製品が後でできる場合があります。** 仕組みを詳しく説明すると:

1. **このセルは歴史的に二重の制約でロックされていました**: ① LLM が高すぎる + ② 「主権 + ガバナンス + 組織間 + 消費者価格」を満たす製品がない。家族には払えず、タピオカミルクティーショップは利益率が薄く、「すべての対話で LLM を実行し、ルーティング / コンサルテーション / ハートビートのために複数のエージェントを生かし続ける」ことは旧モデル価格では**採算が取れませんでした** — だからこの種のセルフホスト、ガバナンス型 AI はデモ段階にとどまり、実際に本番化した人はいませんでした。
2. **DeepSeek の過去 2 ヶ月の新しい API が制約①を取り除きます**（LLM コスト）。**AipeHub はまさに制約②を満たします**（不足している製品）。2 つを組み合わせると、このセルは初めて「手頃な価格」と「使えるもの」の両方を持ちます。
3. **重要な非対称性 — 競合他社も安価な DeepSeek を使えるが、安価な LLM がこのセルに届くのを助けない**:
   - エンタープライズプラットフォーム: 安価な LLM は高タッチのエンタープライズ GTM を変えない; 彼らはトークンを節約するためだけに家族 / 小規模ショップに販売するために**降りてこない**。
   - OSS フレームワーク: 安価な LLM は組織間フェデレーション / ガバナンス / HITL を**追加できない** — 価格引き下げはアーキテクチャに欠けているものを補充しない。
   - 個人エージェント: 安価な LLM は監護 / 組織間 / 承認を**追加できない** — 設計上 1 人 / 1 マシンです。
   - → DeepSeek はすべてを持ち上げる上げ潮ですが、**AipeHub のセルを不均衡に解放します**: そのセルが「コスト ∧ 不足している製品」によって同時にブロックされていたため、DeepSeek はコストを取り除き、**AipeHub だけが不足している製品を供給します**。
4. **そして安価な LLM は他より AipeHub に多く恩恵をもたらします** — 見落とされがちな点: AipeHub の「フレームワークは LLM を実行せず、参加者が実行する」設計は自然に**多くの小さな LLM 呼び出しを生みます**（誰にディスパッチするかを決定するルーティングエージェント、マルチエージェントコンサルテーション、ハートビートの積極的ウェイクアップ、三本柱の成長エージェント、導師 + トピックスクリーン + コンテンツモデレーション…）。この「多くの安価な LLM 参加者」の形は旧モデル価格では**コスト負担**でした — まさに非エンタープライズユーザーを締め出していたもの; DeepSeek が単価を下げると、AipeHub の最も自然な設計が**価格 / 性能の甘いスポット**になります — そして以前に価格が絞り出していたユーザーにとってまさに甘くなります。

### 4.3 この選択がなぜ「フェデレーションのコールドスタート」の死のわなも便利に解決するか

§2 で述べました: フェデレーションクラスの製品の最大の死のわなは**ピア数 = 0**です。選んだ 2 つのビーチヘッドは**自分自身のピアを持ち込みます**:

- 「親 + 子供」は開始から**2 つの主権ハブ**です;
- 「ショップ + サプライヤー」「本部 + フランチャイズ」「師匠 + 弟子」は開始から**≥2 つの当事者**です。

言い換えれば、この種のユーザーの**使用シナリオ自体がペア / グループのフェデレーション**です — 2 番目のピアは BD で探しに行く必要がなく、使用ケースによって持ち込まれます。これは「企業が 1 つのシングルデプロイメントを購入する」とは根本的に異なります: 企業のシングルデプロイメントはフェデレーションネットワークをコールドスタートできませんが、一組の家族やサプライチェーンは**自然に 2 番目のノードを持ち込みます**。だからこのセルを選ぶことは「最も満たされていないニーズ」であると同時に、フェデレーションのネットワーク効果のコールドスタート問題を「死のわな」から「使用ケースによって持ち込まれる」に転換する方法でもあります。

### 4.4 正直な境界（これは「自動操縦での勝利」ではない）

- **壁 A の価格は押し倒されたが、壁 A の「成熟度」は押し倒されていない**: 家族 / 小規模ショップは**真に即使い**（1 行起動、デスクトップシェル、誰でも使えるオンボーディング）を望んでいます; AipeHub はまだ example-first + 設定が必要です。価格 / 性能が需要を解放しましたが、**使いやすさが次のゲートです**。
- **信頼 / コンプライアンスは家族や規制された小チームにとってまだ難しいゲートです**: 子供のデータを守る、法律事務所コンソーシアム — 監査 / コンプライアンスの裏付けがなければまだ使う勇気が持てません。
- **ディストリビューションはまだビジネスの問題であり、コードの問題ではありません**: このセルのユーザーは散在していて獲得が難しい; ノーコードのエントリ + テンプレートギャラリー + 実際のリファレンスカスタマーが必要で、いくつかの追加機能ではありません。

## 5. 意思決定のための一行

> **最もシャープなターゲットは「ガバナンス / 監護 / 組織間を必要とするが、エンタープライズプラットフォームは届かず OSS フレームワークは育てられない」小規模な組織 — 家族 / 教育が第一、中小企業の組織間協働が第二です。** 彼らは歴史的に「LLM が高すぎる」と「そのような製品がない」の両方でブロックされていました; DeepSeek の過去 2 ヶ月の価格引き下げが前者を取り除き、AipeHub はまさに後者の答えであり、この種のユーザーの使用ケースは**自分自身のフェデレーションピアを持ち込み**、コールドスタートを便利に解決します。競合他社も安価な DeepSeek を使えますが、価格引き下げはアーキテクチャに欠けている組織間ガバナンスを補充できません — **このセルの価格 / 性能の窓は AipeHub にとって構造的に開いています。** 残りの難しい戦いは使いやすさ、信頼の裏付け、ディストリビューションにあります — 技術ではありません。
