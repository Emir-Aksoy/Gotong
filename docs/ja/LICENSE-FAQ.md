## 1. Gotong をクローズドソース製品 / SaaS / 社内ツールに組み込むことはできますか?

**はい。** MIT は最も許可的な OSS ライセンスの 1 つです。以下を許可します:

- ✅ 商業利用、Gotong 全体を再パッケージして販売することを含む
- ✅ ソースの変更、名前の変更（ただし名前を変更する場合は「Gotong に基づく」と明記してください）
- ✅ クローズドソースの派生物 — あなたの変更はオープンソースにする**必要はありません**
- ✅ npm 依存関係として `@gotong/core` をクローズドソース SaaS に取り込む

**唯一の厳密な要件**: LICENSE ファイル + 著作権表示を保持する
（製品の NOTICE / サードパーティライセンスページに Gotong を記載すれば十分です）。

## 2. ソースを変更した場合 — 変更を貢献し直す必要がありますか?

**いいえ。** MIT はコピーレフトではありません。以下が可能です:

- 変更をプライベートに保持する
- 商業製品の一部として出荷する
- 上流に PR を送らない — それは完全に問題ありません

とはいえ、PR は歓迎します — プロジェクトが良くなればなるほど、次のアップグレードが
安くなります。プロセスについては [`CONTRIBUTING.md`](../../CONTRIBUTING.md) を参照してください。

## 3. `templates/community/` のサードパーティプロンプトテンプレートを商業的に使用する際に注意すべき点は何ですか?

`templates/community/` は 2 つの上流ソースを収集します:

| ソース | ライセンス | 商業利用 | 注意事項 |
|---|---|---|---|
| [`awesome-chatgpt-prompts`](https://github.com/f/awesome-chatgpt-prompts) | **CC0 1.0**（パブリックドメイン） | ✅ 任意の使用 | 帰属は法的に**不要**; 敬意として出典行を保持 |
| [`awesome-chatgpt-prompts-zh`](https://github.com/PlexPt/awesome-chatgpt-prompts-zh) | **MIT** | ✅ 任意の使用 | 著作権 + ライセンス表示を**保持する必要あり** |

表示はどのように保持されますか? `templates/community/` はすでに 3 つの層で保持しています:

1. すべての yaml ファイルの**4 行のヘッダーコメント**: `# Source` /
   `# Upstream` / `# License` / `# Adapted`
2. 集約ファイル
   [`templates/community/LICENSE-NOTICES.md`](../../templates/community/LICENSE-NOTICES.md)
   が完全な MIT テキスト + CC0 の概要 + 上流リポジトリ URL を保持
3. ディレクトリの [`README.md`](../../templates/community/README.md)
   が適応ルールとライセンスマトリックスを説明

**3 層をそのまま保持**して `templates/community/` を再配布する限り
（git フォーク / クラウド raw URL / 社内 CDN — すべて問題なし）、
完全にコンプライアンスを満たしています。

> 「テンプレートのコンテンツを管理 UI に貼り付けて
> `secrets.enc.json` / `agents.json` に格納しました — それは配布ですか?」—
> **いいえ。** あなたは単に自分のデプロイメント内で使用しているだけで、
> 第三者に転送しているわけではありません。帰属のアクションは不要です。

## 4. LICENSE を変更して「私たちの」製品として再リリースできますか?

**製品名を変更して自分のライセンス行を追加**することはできますが、
**元の MIT テキストを削除することはできません**:

- ✅ 派生物は `BobHub` と呼ばれ、Apache-2.0 /
  独自 / 自分で書いたものにできます
- ✅ 自分の LICENSE ファイルに自分の著作権を入れられます
- ⚠️ ただし Gotong の元の MIT テキスト + 上流の著作権行を
  **どこかに保持する必要があります**（例: NOTICE.md や THIRD-PARTY.md）
- ❌ 「Gotong は私たちのオリジナル作品だ」と主張することは**できません** —
  それは詐欺であり、ライセンスに関係なくそうです

## 5. 同僚が GPT を使って書いたプライベートプロンプトをエージェントとしてインポートしました — ライセンスリスクはありますか?

**Gotong 側にはありません。** あなたや会社が書いたプロンプトは会社自身の資産です;
Gotong はただのランタイムコンテナです。ただし確認する必要があります:

- 同僚の GPT 出力が OpenAI の利用規約に準拠しているか
  （モデル出力の「所有権」に関する OpenAI のポリシーは時間とともに変わっています
  — 法務に確認してください）
- プロンプトが他人のコード / 記事の抜粋を**引用**している場合、
  その引用が自身のライセンスで許可されているか

どちらも Gotong プロジェクトが管理するものではありません — MIT はソフトウェア
自体をライセンスするものであり、それを使って生成するコンテンツではありません。

## 6. 顧客のイントラネット内に Gotong をデプロイします — どのライセンスファイルを渡す必要がありますか?

最低限:

- Gotong リポジトリルートの `LICENSE` ファイル
- `templates/community/` を使用する場合: `LICENSE-NOTICES.md` も一緒に持参
- `@gotong/core` npm パッケージを組み込む場合: パッケージはインストール時に
  自身のライセンスを同梱します; 下流の再配布は
  `node_modules/@gotong/*/LICENSE` を削除しないだけでよい

一般的なパターンは、製品の「サードパーティライセンス」ページにすべての
上流 OSS ライセンステキストを一覧することです。そこに Gotong の MIT を追加すれば完了です。

## 7. Gotong のランタイム依存関係に GPL / AGPL スタイルのコピーレフトは含まれていますか?

現在はありません。主な依存関係:

| 依存関係 | ライセンス |
|---|---|
| `ws`（WebSocket） | MIT |
| `yaml` | ISC |
| `better-sqlite3`（オプション） | MIT |
| `@anthropic-ai/sdk`（オプションのピア依存） | MIT |
| `openai`（オプションのピア依存） | Apache-2.0 |
| `vitest`（開発のみ） | MIT |
| `tsx`（開発のみ） | MIT |

すべて許可的です。GPL / AGPL 依存関係が提案された場合は最初にイシューを開きます;
私たちの姿勢はダウンストリームの柔軟性を保つためにコピーレフトの依存関係を
**避ける**方向です。

## 8. Gotong のワイヤープロトコルはライセンスの一部ですか?

いいえ。`docs/PROTOCOL.md` に記述された JSON フレームフォーマットは
**事実上の仕様** — 誰でも**許可なく**自分のハブサーバーや SDK を実装できます。
言語エコシステムのポート（Go / Rust / ブラウザ SDK など）を推奨します;
それぞれが自分のライセンスを選択します。

## 9. 脆弱性を報告するにはどうすればよいですか?

プロジェクトリポジトリの **GitHub セキュリティアドバイザリ**（プライベート送信）を通じて
— それが唯一のセキュリティチャネルです; 意図的にセキュリティメールはありません
（[`SECURITY.md`](../../SECURITY.md) 参照）。脆弱性の詳細をパブリックイシューに
投稿することは**問題があります** — ライセンスが許可していても。

## 10. 会社が Gotong を社内でフォークして、フォークをオープンソース化せずに使用できますか?

**もちろんです。** MIT は伝播しません。以下が可能です:

- プライベート Git にフォーク → 自由に変更 → イントラネットにデプロイ
- フォークの名前を変更して顧客向けにプライベートにデプロイ
- フォークのビルド成果物をクローズドソースバイナリとして販売

**最終的な成果物がどこかに Gotong の元の MIT ライセンスを保持する限り**
（通常は「オープンソース表示」ページ）、すべて問題ありません。
