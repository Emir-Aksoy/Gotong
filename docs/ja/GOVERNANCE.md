# AipeHub ガバナンス

<!-- doc-version: 1.0 -->
> **ドキュメントバージョン 1.0** · 日本語訳 · 最終更新 2026-06-27 · 権威ある原典：[English](../../GOVERNANCE.md)。訳文と英語版に矛盾がある場合は、英語版が優先されます。

このドキュメントでは、AipeHub における**意思決定の方法**を説明します: 誰がプロジェクトをメンテナンスするか、変更がどのように取り込まれるか、コミュニティテンプレートがどのように公式ギャラリーに入るか、そして人々が意見の相違を持つときに何が起こるか。意図的にシンプルに保っています — プロジェクトは若く、小さなプロジェクトに重いガバナンス構造をかけるのは儀式に過ぎません。コミュニティが成長するにつれてこのドキュメントも成長させます。

このドキュメントはプロジェクトの憲章である [`CHARTER.md`](../../CHARTER.md) の下に位置します: 憲章は AipeHub が何であるか、そして何になることを拒否するかを述べています。このドキュメントはどのように決定するかを述べます。両者が出会う場合 — 例えば「フレームワークは LLM を実行しない」— 憲章が根拠であり、このドキュメントはその執行です。

一つだけ読むなら: **設計ラインは交渉できませんが、それ以外のほぼすべては交渉できます。** [唯一の譲れないもの](#唯一の譲れないもの) を参照してください。

## ロール

3 つのロールを保持します。秘密の 4 番目の階層はありません。

| ロール | 意味 | 獲得方法 |
|---|---|---|
| **コントリビューター** | イシューを開いたり、PR を送ったり、テンプレートを提出したり、Discussions で協力する人。 | 参加するだけ。申請不要。 |
| **メンテナー** | PR のレビューとマージ、イシューのトリアージ、リリースの公開が可能。サブシステムまたはプロジェクト全体を担当。 | 設計に沿った良いコントリビューションの実績を積み、その後公開で推薦される — [メンテナーになる](#メンテナーになる) を参照。 |
| **スチュワード** | 争われた決定の最終的な調停者であり、設計ラインの守護者。今日これは創設メンテナーです。 | プロジェクトがスチュワードを選出できるほど大きくなるまで創設者が保持します（[コンポーネント委員会への道](#コンポーネント委員会への道) を参照）。 |

現在のメンテナーは [`MAINTAINERS.md`](../../MAINTAINERS.md) に記載されています — 今日はスチュワード、レビュアー、リリースマネージャーも兼ねる創設メンテナーのみです。このドキュメントは、その体制が**一時的であり書き留められている**もので、習慣ではないことを明示するために存在します。次のセクションは 2 番目のメンテナーが辿る道です。

### メンテナーになる

ラダーは意図的に軽くしています — 若いプロジェクトであり、目標は設計ラインを保持する人材のベンチを育てることであり、門番をすることではありません。ゲームするためのチェックリストではなく、大まかなガイドライン:

- **数ではなく実績。** 数ヶ月にわたって、おおよそ ~5 件の非自明なマージされた PR — または同等のもの: 維持し続けているフラグシップテンプレート、実質的なアダプター、または継続的なレビュー / トリアージの貢献。この数字は「あなたの判断を信頼するのに十分な作業を見てきた」というための最低限であり、ドライブバイ PR で達成する目標ではありません。
- **設計ラインへの感覚。** あなたの PR とレビューが、ロジックの置き場所として Hub ではなく*参加者*を選んでいることを示す（[唯一の譲れないもの](#唯一の譲れないもの) を参照）。
- **公開で推薦される。** 既存のメンテナーが公開のイシューであなたを推薦します — 自己推薦も可、理由を述べてください。承認はメンテナー間のレイジーコンセンサスで、スチュワードが確認します。あなたの名前は同じ PR で [`MAINTAINERS.md`](../../MAINTAINERS.md) に追加されます。

引き受けること: 担当エリアの他者の PR のレビュー、設計ラインの維持、メンテナンスしているものへのイシューへの回答。いつでも降りられる責任です — いつでも退いてよく、名簿から外れるのではなく `MAINTAINERS.md` でエメリタスに移動されます。

今日メンテナーはちょうど一人 — 創設スチュワード — なので、このラダーは**書かれているが休眠中**です: まだ推薦する人がいません。これは*2 番目の*メンテナーがアドホックな肩叩きではなく既知の道で参加できるように存在します。

## 変更の取り込み方

ほとんどの変更は退屈であり、退屈は良いことです:

1. **非自明なものはイシューを先に開く** — 新しい依存関係、プロトコル形状の変更、新しいパッケージ、スケジューリングやフェデレーションの動作変更。ドライブバイのタイポ PR や小さなドキュメント修正はこれをスキップできます。
2. **小さな PR を送る。** 一つの変更、一つの PR。[`CONTRIBUTING.md`](../../CONTRIBUTING.md) を参照。
3. **メンテナーがレビューする。** レビューは順番に 3 つのことを確認します: 正確性、[設計ライン](#唯一の譲れないもの)、シンプルさ。
4. **マージ。** レイジーコンセンサス: 合理的な期間内にメンテナーが異議を唱えず、CI / ローカルチェックが通れば、マージされます。異議は議論で解決されます。本当のデッドロックはスチュワードに委ねられます。

CLA は不要です。コントリビューションすることで、プロジェクトの [MIT ライセンス](../../LICENSE) のもとで作業を提供します。

### PR 以上のものが必要な決定

いくつかのカテゴリは特別な配慮を要し、メンテナーが意図的にスローダウンさせます:

- **ワイヤープロトコルの変更** — [`docs/PROTOCOL.md`](../../docs/PROTOCOL.md) の形状を変えるもの。バージョンバンプと明示的な移行ノートが必要です。
- **不可逆なスキーマ変更**（列 / テーブルの削除）。プロジェクトは 1.0 以前の前方互換性を約束しませんが、永続化データを削除する前に影響範囲を議論します。
- **新しいランタイム依存関係**、特にネイティブのもの。イシューを開いてください。
- **パブリック API サーフェスの削除。** 誰も使っていないと思っていても、先に影響を説明してください。

## テンプレートが公式ギャラリーに入る方法

AipeHub は**テンプレート**（`aipehub.template/v1` — エージェントチーム + ワークフロー + ナレッジベースの*参照*を含む自己完結型 YAML で、シークレット、ナレッジコンテンツ、人員は含まない）を配布します。管理 UI の一クリックギャラリーと公開サイトに表示される*フレームワークに同梱される*基準は、*コミュニティテンプレートとして受け入れられる*基準より高いです。

二つの階層があり、それらは異なる約束です:

### コミュニティテンプレート — 「ライセンスを確認し、パースできる」

[`templates/community/`](../../templates/community/) に住みます。マージされるために、コミュニティテンプレートは以下を満たす必要があります:

1. **パースできる。** 実際の `parseTemplate` を通過し（すべての埋め込みワークフローが実際の `parseWorkflow` を通過する）。これは人間の目視ではなく自動バリデーションテストで確認されます — [`templates/community/templates/README.md`](../../templates/community/templates/README.md) を参照。
2. **正直なプロベナンスを持つ。** 別のテンプレートやアップストリームのプロンプトライブラリから派生した場合、`provenance` ブロック（`derivedFrom`、`author`、`notes`）でそれを宣言します。プロベナンスは引用クレジットがアップストリームに流れる方法です — 削除しないでください。
3. **シークレットを含まない。** すべての認証情報は `${ENV}` プレースホルダーです。リテラルなキーが含まれているテンプレートは完全に拒否されます。
4. **適応されたアップストリーム素材に対して明確な商業利用可能なライセンスを持つ**（CC0 / MIT / Apache-2.0 / BSD）。非商業または未ライセンスのソースは受け入れられません。[`templates/community/LICENSE-NOTICES.md`](../../templates/community/LICENSE-NOTICES.md) を参照。

以上です。基準を満たすコミュニティテンプレートはマージされます。この階層では趣味をキュレーションするのではなく、*安全さと誠実さ*をキュレーションしています。

### フラグシップテンプレート — 「私たちが保証する」

プロジェクトが非技術的なユーザーに積極的に推薦する小さなキュレーションセット（[`docs/zh/FLAGSHIP-TEMPLATES.md`](../../docs/zh/FLAGSHIP-TEMPLATES.md) を参照）。コミュニティの基準に加えて、フラグシップテンプレートは以下を満たす必要があります:

1. **決定論的なデモを同梱する**。API キーなしで実行でき、自身の動作を自己アサートするもの（`examples/*` の規約）。レビュアーが一つのコマンドで動作を証明できる。
2. **ガバナンスの姿勢を明確に述べる** — 何に触れられるか、何に触れられないか、どこで人間がループに入るか。ドアを施錠したり、お金を使ったり、子供のデータをフェデレーションリンクを通じて送ったりできるテンプレートは、それを埋めるのではなく人間確認ゲートを示す必要があります。
3. **メンテナンスされている。** フラグシップテンプレートには、そのイシューに答えるメンテナーがいます。ビット腐敗して誰も修正しなければ、コミュニティ階層に降格します。

コミュニティ → フラグシップへの昇格はメンテナーの決定で、公開のイシューで行われます。降格も同様です。

## 人々が意見の相違を持つとき

意見の相違は正常であり歓迎されます — それが設計を圧力テストする方法です。プロセス:

1. **イシュー / PR で話し合う。** 結論だけでなくトレードオフを述べる。「X を好む」は弱い。「Z の代償で Y のため X」は有用。
2. **メンテナーが決定する**、議論が止まった場合。メンテナーは*なぜ*かを記録に残して説明すべきです。
3. **スチュワードが本当に争われた決定の調停者であり**、変更が設計ラインを越えるかどうかの最終権限です。これはバックストップであり、日常的なステップではありません — 頻繁に調停しなければならないスチュワードは、メンテナーベンチを育てるのに失敗したスチュワードです。

行動の紛争は別途処理されます — [`CODE_OF_CONDUCT.md`](../../CODE_OF_CONDUCT.md) を参照。

## 唯一の譲れないもの

AipeHub には PR で投票により廃止できないアーキテクチャ上の約束が一つあります。それを変えることはプロジェクトがもはや AipeHub ではないことを意味するからです:

> **フレームワークは LLM を実行しない。** Hub はメッセージをルーティングし、タスクをディスパッチし、トランスクリプトを書き、イベントを発行します。すべての決定は参加者 — エージェント、人間、外部サービス — の手に残ります。状態はディスク上のファイルです。認証情報はローカルに留まります。フェデレーションは明示的なリンクごとの境界を持つピアツーピアです。

LLM 呼び出し、エージェントループ、またはビジネスロジックを*Hub に*入れるパッチはリダイレクトされます — アイデアが悪いからではなく、それは基盤ではなく参加者に属するからです。それ以外のすべて — スケジューラ、プロバイダ、アダプター、トランスポート、UI、テンプレート — は変更に開かれています。

## コンポーネント委員会への道

このプロジェクトの長期的な形は、**ガバナンスされた再利用可能なコンポーネントのマーケットプレイス** — テンプレート、アダプター、ナレッジベースコネクター — であり、人々が家、家族、またはお金に向けるのに十分信頼できるものです。そのマーケットプレイスをキュレーションするのは一人でできることではなく、一人で*すべきでは*ありません。

コントリビューターのベースがギャラリーキュレーションが本当の定期的な仕事になるほど大きくなったとき、**コンポーネント委員会**を設立します: フラグシップに昇格するもの、引用クレジットの表示方法、テンプレート作者間の紛争解決を担当する、選出された少数のメンテナーのグループです。委員会メンバーがどのように推薦・選出・交代するかを説明するために、そのときこのドキュメントを修正します。

このパラグラフを今、プロジェクトが小さいうちに書いています。委員会が**後のアドホックな権力掌握ではなく、書かれた使命を持つ計画されたマイルストーン**となるようにするためです。トリガーは日付ではなく、継続的なコントリビューション量です。

## 地域チャプター

最終的な姿は主権 hub の自由なグラフであり、一つの中央プラットフォームではありません — そのグラフの人間側は**地域チャプター**: 自分たちの hub を運営し、コミュニティ向けにテンプレートをキュレーションし、自分たちの言語で新参者を助ける、ローカル・言語・コミュニティに根ざしたグループです。憲章（[`CHARTER.md`](../../CHARTER.md) §11）はそれらを歓迎します。このセクションは実際の運用方法を述べます。

チャプターは**主権を持ち、フランチャイズではありません。** 自分の部屋を所有し、中央の大家に答えません。チャプターを*始める*のに誰かの許可は不要です — それはプロジェクトが基づいている「運営し続けるために許可が必要な単一の当事者がいない」前提に矛盾します。今日、hub を立ち上げ、ローカルコミュニティを集め、テンプレートをキュレーションして AipeHub チャプターと呼ぶことができます。

チャプターでは**ない**もの:

- **プロジェクトの公式な声ではありません。** チャプターは AipeHub ではなく自分のコミュニティのために語ります。設計ラインを設定したり、憲章を批准したり、フラグシップに何が昇格するかを決定したりしません — それらは正式なリポジトリ（[`MAINTAINERS.md`](../../MAINTAINERS.md)）のメンテナーとスチュワードのもとに留まります。
- **ラインを保持するフォークではありません。** チャプターは自分のコミュニティ向けに修正したビルドを実行できますが、正式な憲章、プロトコル、フラグシップセットは正式なリポジトリに存在します。変更を*本家の* AipeHub にしたいチャプターは、他の誰かと同様に上流にプルリクエストとして送ります。

### 認知はオプションで軽い

チャプターを運営するのに承認は不要です。**認知されたチャプターとして掲載される** — 新参者が見つけられるようにプロジェクトからリンクされる — ことは小さなオプトインのステップで、テンプレート昇格と同様に機能します:

1. **Discussions でアナウンスする** — 誰であるか、サービスする地域 / 言語 / コミュニティ、hub がどこにあるか。
2. **メンテナーが公開イシューでそれを承認する。** 基準は規模ではなく誠実さ: プロジェクトを誠実に表現し、[`CODE_OF_CONDUCT.md`](../../CODE_OF_CONDUCT.md) に従い、持っていない公式な地位を主張しない。
3. **認知は与えられたのと同じ方法で撤回できます** — 公開イシューで、理由とともに — チャプターがプロジェクトを誤って表現したり、行動基準を破った場合。プロジェクトはチャプターの hub をシャットダウンできません（そうしようとも思いません）— それはその主権の部屋です — しかしリスティングを止め、もはや持っていない承認を示唆するのをやめるよう求めることができます。

**コンポーネント委員会**が設立されると（上記参照）、チャプターリストのキュレーションとチャプター間の紛争解決がその使命の自然な一部になります。それまではメンテナーの軽い判断です。

## AipeHub の名前を使う

コードは [MIT](../../LICENSE) です — ライセンスと著作権の行を保持した上で、商業または独自ソースの製品に埋め込み、修正し、出荷できます。**ライセンスはコードをカバーします。プロジェクトの名前とアイデンティティを引き渡すものではありません。** ここには登録商標はなく、そのふりもしません — 以下は法的脅威ではなく、誠意を持って求められる**コミュニティの規範**です:

- **説明的な使用は歓迎されます。** 「AipeHub の上に構築された」「AipeHub hub」「AipeHub マレーシアチャプター」— これらは自由に言ってください。それらは真実であり、あなたが言っていることを喜んでいます。
- **持っていない承認を示唆しないでください。** 製品、フォーク、またはサービスを*AipeHub プロジェクトとして*あるいは公式に承認されたものとして提示するような名称をつけないでください — 「AipeHub Official」はなく、正式なダウンロードであるかのように見せかける紛らわしいものもありません。
- **正式プロジェクトのブランドを変更しないでください。** 修正した配布物はあなたが出荷できますが、正式な「AipeHub」— 憲章、設計ライン、フラグシップセット — は正式なリポジトリにあるものです。あなたのフォークが精神的に乖離するなら、独自の名前をつけてください。MIT ライセンスがそれを保証し、正直な名前はあなたのユーザーに借用した名前より良いサービスを提供します。

これは機能する名前保護の最も軽い版です: 「AipeHub」が憲章が説明するガバナンスされた、ファイルファースト、ループに人間がいるものを意味し続けるのに十分であり、それ以上ではありません。

## このドキュメントの修正

ガバナンスはコードと同じ方法で変更されます: イシューを開き、PR を送り、メンテナーのレビューを受けます。[唯一の譲れないもの](#唯一の譲れないもの) の変更にはスチュワードの承認と設計ラインが動くべき理由の明確な声明が必要です。そのセクションが変わることはないと期待しています。残りは成長することを意図しています。
