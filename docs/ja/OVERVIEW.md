## 一文で言うと

**AipeHub** は **TypeScript と Python のためのセルフホスト型コラボレーションワークスペース**です: 人間と AI エージェントが一つの「部屋」を共有し、意図的にシンプルな Hub がタスクをディスパッチし、結果を収集し、全実行を記録します。

**エージェントフレームワークではありません**（LLM を実行しない）— **マルチ参加者コラボレーションのための基盤**であり、組織は**キー、データ、または請求を手放すことなく**フェデレートできます。

---

## それが何であるか — そして何の*下に*あるか

ほとんどの「エージェント」プロジェクトはエージェントそのもの、または一つのエージェントのループを書くためのフレームワーク（LangGraph、CrewAI、AutoGen）です。AipeHub は**そのどちらでもなく**、それらがプラグインする*レイヤー*です。LangGraph グラフ、CrewAI クルー、CLI コーディングエージェント（Claude Code、Codex）、外部 A2A エージェント、そして人間がすべて同じ `Participant` として同じ部屋に参加します。Hub はメッセージをルーティングし、タスクをディスパッチし、トランスクリプトを記録し、境界を強制します — **LLM を実行しない**ため、すべての決定が参加者の手に残ります。

それをメッセージバス以上にする 3 つのこと:

- **平等な参加者** — 人間はエージェントとまったく同じ `Participant` です。「request-human-input ツール」はありません。人間とエージェントは同じタスク + トランスクリプトを通じてコラボレーションし、同じ非同期 / 長時間実行プリミティブを使います。
- **ガバナンス** — 機密性の高い、または組織をまたぐアクションはただ実行されません。インボックスから人間が承認することを要求できます（提案 → レビュー → 確認）、完全な監査証跡付きで。
- **主権** — すべてのワークスペースはあなたが所有するディスク上のディレクトリです。二つの組織がフェデレートするとき、認証情報、データ、請求はそれぞれのホームに留まります。境界を越えるものは**リンクごとの信頼契約**によって制約されます。

その組み合わせ — 単一の巧みなプロトコルではなく — が AipeHub とは何かです。人間-エージェントの平等、ガバナンスされたクロスオーグフェデレーション、セルフホスト型の主権を一つの実行可能なファイルファーストパッケージに組み合わせた最初の基盤です。

## 四つのエッジ — AipeHub が世界とつながる方法

AipeHub は 4 つのエッジを通じてエコシステムの他の部分に接触します。**オープンプロトコルが存在する場合はそれを話します** — 再発明しません:

| エッジ | プロトコル | 方向 | 運搬するもの |
|---|---|---|---|
| ツール & データ | **MCP** | 双方向 | エージェントが外部 MCP ツールを呼び出す。外部クライアント（Claude Desktop、Cursor）が Hub を駆動する。 |
| エージェント ↔ エージェント | **A2A** | 双方向 | 受信 `message/send` がディスパッチになる。送信呼び出しがリモート A2A エージェントを駆動する。 |
| コーディングエージェント | **ACP** | 送信 | Hub が Claude Code / Codex とのセッションを生成し保持し、ターン毎に駆動する。 |
| Hub ↔ Hub | **HubLink** | 双方向 | AipeHub 独自の 2 つの Hub 間のフェデレーションリンク — リンクごとの信頼契約、クロスオーグタスク転送、承認ゲートが住む場所。 |

最初の 3 つは AipeHub が実装するエコシステム標準です。HubLink は AipeHub が所有する一つの部分です — 巧みなワイヤーフォーマット（下はWebSocket + ベアラートークン + JSON-RPC）としてではなく、**2 つのガバナンスされた Hub が交換するものの契約**として: 能力マニフェスト、祖先を保持するタスク転送、そして以下のリンクごとの信頼契約。

→ さらに詳しく: [`MCP.md`](../MCP.md) · [`FEDERATION.md`](../FEDERATION.md) · [`PROTOCOL.md`](../PROTOCOL.md)

## 参加者のプラグイン方法

見出しのパスは**LLM エージェントを追加する 2 つの方法**です:

| パス A · ホスト管理 | パス B · 外部 SDK |
|---|---|
| 管理 UI でフォームに入力 / YAML をインポート / テンプレートを貼り付け → ホストが自プロセス内で `LlmAgent` を生成 | コード（Node / Python）で `AgentParticipant.handleTask` を実装し、Hub の WebSocket ポートに `connect(url, agents)` する |
| **コード 0 行** | コードを書く |
| LLM エージェントのみ（ラップされた Anthropic / OpenAI / Mock） | **任意の種類**（LLM、スクレーパー、ローカルツール、プライベートロジック、Python ML モデル） |
| プロバイダーキーは `secrets.enc.json`（エージェントごとまたはワークスペースデフォルト）にディスク上で暗号化、または env から読み取り | あなたが API キーを管理。エージェントはあなた自身のマシンで動く |
| ホスト再起動時に自動再生成 | あなたがライフサイクルを管理。SDK には組み込みの自動再接続がある |
| 最適: 一般ユーザー / 標準 LLM ロール / 60 秒で起動 | 最適: 開発者 / プライベートデータ / コードを公開しない |

→ パス A: [`HUMAN.md §1 エージェント`](../HUMAN.md#1-智能体v21) + [`TEMPLATES.md`](../TEMPLATES.md)
→ パス B: [`AGENT.md`](../AGENT.md)

…すべてが同じ `Participant` なので、同じ部屋は以下も受け入れます:

- **CLI / ACP コーディングエージェント** — Hub が保持された ACP セッション（実機検証済み）を通じて Claude Code / Codex を駆動し、破壊的コマンドを人間の承認のために一時停止できる危険アクションゲートがある。
- **外部 A2A エージェント** — 能力の下にリモートエージェントを登録する。ワークフローステップが他と同様にルーティングする。
- **フレームワークアダプター** — Python SDK 経由で LangGraph グラフまたは CrewAI クルーを `Participant` としてラップする。フレームワーク自体は Hub によってインポートされない。

これらはすべて**自由に混在**できます — 一つの部屋にホスト管理の `writer-zh`、独自の SDK 接続 `rag-agent`、Codex コーディングセッションを、スケジューラに対して完全に透明な形で保持できます。

## テンプレートの出所

```
                  templates/
                  ├── agents/           original official templates
                  ├── teams/            original official teams
                  └── community/        adapted from third parties (CC0 + MIT)
```

取得する 3 つの方法、好みで選ぶ:

1. **テンプレートギャラリー、ワンクリック** — 管理 UI には既製の hub のギャラリーが同梱されている（個人 / 組織 / クロスオーグ）。選ぶ → インストール → エージェント + ワークフロー + KB スロットが Space に着地する。
2. **コピーペースト** — GitHub で `.yaml` の **Raw** をクリック → コピー → 管理 UI 「エージェント → インポート」でペースト。
3. **ファイルをダウンロード** — `.yaml` をローカルに保存 → 管理 UI 「ファイルをアップロード」。

すべてのファイルに `# Source` / `# Upstream` / `# License` / `# Adapted` ヘッダーコメントがあるため、**アップストリームのプロベナンスは失われません**。サードパーティライセンスの全文は [`../templates/community/LICENSE-NOTICES.md`](../../templates/community/LICENSE-NOTICES.md) にあります。

> **テンプレートとフレームワークは設計上分離されています。** テンプレートは*構造と参照* — エージェント、ワークフロー、KB スロット — を運び、知識*コンテンツ*自体は含まず、あなたの人員やシークレットも含みません。インストールすると接続が配線されます。別の組織のデータを復元することはありません。

→ 完全なフロー: [`TEMPLATES.md`](../TEMPLATES.md)
→ インストールできる既製の hub: [`zh/HANDS-ON-HUBS.md`](../zh/HANDS-ON-HUBS.md) (中文)

## 一つの部屋にいる数人

AipeHub は「チーム」を**一つの部屋** = 一つの `.aipehub/` ディレクトリとしてモデル化します。3 つのロール階層:

| ロール | URL | この部屋でできること |
|---|---|---|
| **admin** | `/admin` | 部屋の設定、エージェントリクエストの承認/拒否、タスクのディスパッチ、作業の評価、他の admin の招待 |
| **worker** | `/`（`/me` ワークベンチ） | ニックネームとできる作業を選ぶ、自分向けメンバーワークフローを実行、インボックスを処理、タスクを完了または辞退 |
| **agent** | WS ポート | ディスパッチされたタスクを自動受信、結果を返す |

### 典型的な小チームワークフロー（スクリプト化）

```
0  Alice (admin) が hub を起動 → 起動時にブラウザが一回限りの
   管理 URL を表示。1Password に保存する。
1  Alice が管理 UI でプロバイダーキーを設定 → ワークスペースの
   デフォルトキーがディスクに暗号化される。
2  Alice がテンプレートをインストール（または storyteller.yaml をインポート）
   → ホストが即座に LLM エージェントを生成し、オンラインと表示される。
3  Alice が Bob と Carol に招待 URL を送る。彼らはニックネームを選び、
   できる能力（draft / review）を確認 → 部屋に参加。
4  Alice がタスクをディスパッチ:「忍耐についての子供の話を書く」、
   strategy = capability:[story] → ホスト管理の storyteller がそれを引き受ける
   → 30 秒後に 600 語の話が返ってくる。
5  ワークフローステップが承認を必要とする → Bob のインボックスに一時停止する。
   Bob が /me ワークベンチから承認し、実行が再開する — ツール呼び出しではなく
   ループに入る人間。
6  Alice が作業を評価する。コントリビューションリーダーボードが更新される。
   すべてのイベントが transcript.jsonl にあるので、クラッシュ + 再起動で
   完全に回復する。
```

**キーコンセプト**（HUMAN.md で詳細）:

- **3 つのディスパッチ戦略**: `direct`（名前による）、`capability`（スキルによる）、`broadcast`（最初に主張した者が勝つ）
- **ループに入る人間**: ワークフローステップが人のインボックスにディスパッチし、続行前に承認 / 選択 / 編集を待てる
- **`/me` ワークベンチ**: メンバーが自分向けメンバーワークフローを実行し、最近の実行を確認し、自分のエージェントを管理（BYO キー）、すべて自分自身にスコープされる
- **API キー、3 階層**: エージェントごとのプライベート → ワークスペースデフォルト → 環境変数

→ 完全な解説: [`HUMAN.md`](../HUMAN.md)

## 組織をまたいで — ガバナンスされたフェデレーション

**「マルチチーム」の 2 つの異なる意味 — 混同しないように:**

### 一つの部屋、多くのロール（= 上のセクション）

全員が同じ `.aipehub/` ディレクトリ、同じ hub プロセスにいます。これがデフォルトです。

### 多くの部屋、フェデレート（= 真のクロスオーグ）

各組織が独自の独立した hub（独自の `.aipehub/`、独自の人員とエージェント、**独自の API キーと独自の請求**）を実行します。2 つの hub が **HubLink** を通じて接続し、一方が他方に求めることができるものは**リンクごとの信頼契約**によって固定されます:

- **能力許可リスト** — ピアが呼び出せる能力の正確なリスト
- **データクラスゲート** — どのクラスのデータがリンクを越えることができるか（fail-closed）
- **クォータ** — リンクごとのレート / 予算上限、再接続後も保持
- **失効** — いつでもリンクを切断
- **ナレッジベース許可リスト** — ピアがアクセスできる共有 KB

最もシンプルなパターンは `TeamBridgeAgent`: サブ hub 全体が上流に対して**単一のエージェント**として表示され、内部メンバー / キー / サブタスクは親に対して不可視です。

```
   Company Hub (Bob is admin)
       │
       ├── agent · alice-team   ←─┐
       │                          │  TeamBridgeAgent  (over HubLink)
       │                  ┌───────┴────────┐
       │                  │ Alice's Hub    │ (Alice is admin)
       │                  │  · writer-bot  │   keys / people / billing
       │                  │  · reviewer-bot│   all stay on Alice's hub
       │                  └────────────────┘
       └── agent · david-team   ←── another team, same idea
```

ブリッジングを超えて、**一つの hub 上のワークフローが別の hub の能力でステップを実行できます**。そのピアが承認を必要とする場合、ステップは誰かが承認するまで人のインボックスに一時停止します — クロスオーグ呼び出しはガバナンスされ、2 ステップで、完全に監査可能で、ワークフロー YAML はピアを名指しすらしません（能力を名指しするだけで、リンクはランタイム設定です）。

**これが重要な理由 — 主権はそのまま:**

- 上流は*集計された結果*を見ます（「alice-team が N タスクを完了した」）、ピアのキーや生データは見ません
- 各 hub が**独自の認証情報ボルト**と**独自の使用量 / コスト台帳**を保持します — 請求は hub ごとです
- プライベートな内部 PoC が必要? ローカル hub を実行 — オンボーディングコストゼロ
- 会社全体でコラボレートしたい? ガバナンスされたリンクを上に掛ける — **既存のチーム構造に触れることなく**
